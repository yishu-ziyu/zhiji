# 一人公司效率提升 Agent 赛道命题约束

## 赛道原文理解

本赛道不是泛用效率工具。

它要求围绕一人公司在真实工作、学习、运营、交付中的痛点设计 Agent。

参赛队伍只需要选择其中一个方向切入。

也可以基于真实需求提出新的延展方向。

两天内的目标是做出最终作品，并准备路演展示材料。

## 官方给出的场景范围

1. 会议纪要和任务追踪 Agent。
2. 项目管理和交付提醒 Agent。
3. 客户沟通和报价辅助 Agent。
4. 资料检索和研究报告 Agent。
5. 公司知识库、日常学习和知识管理 Agent。
6. 小团队协作和自动化工作流。

## 交付物要求

每支队伍需要产出一个可演示、可解释、可复盘的 Agent Demo 或工作流。

可接受形态包括网页、工具链接、插件、脚本、自动化流程或录屏演示。

路演材料建议包含问题场景、解决方案、Demo 路径、工具或技术路线、商业化可能。

电商赛道账号需要自备。

效率赛道没有看到必须自备外部业务账号的要求。

现场鼓励使用各种模型。

比赛提供价值 500 USD 的等额算力支持。

## 对当前项目的命题收敛

当前项目选择的切口是：会议纪要和任务追踪 Agent + 项目管理和交付提醒 Agent。

不要再扩成完整的一人公司 OS。

不要平均覆盖六个官方场景。

黑客松主线应聚焦一个高频场景：一人公司开完会、接完客户需求、做完内部复盘后，如何把零散信息变成可执行任务，并在交付前持续提醒。

## 推荐产品定义

产品名称暂定为：一人公司交付管家。

一句话定义：把会议和沟通里的承诺，自动变成能追踪、能提醒、能复盘的交付看板。

用户不是大公司项目经理。

用户是一个人或小团队创业者。

他们最大的问题不是不会写纪要，而是承诺太散、任务太多、没人提醒、下次开会又从头问。

## 当前 Demo 必须证明的事

第一，Agent 能读懂一段真实会议或沟通转写。

第二，Agent 能提取决议、待办、负责人、截止日期和风险提醒。

第三，待办能自动进入看板，而不是停留在一段文字里。

第四，用户能看到哪些事快到期、哪些事没人负责、哪些事可能影响交付。

第五，评委能在 3 分钟内看懂这不是聊天框，而是把信息推进到行动的工作流。

## 不该做的事

不要做万能个人助理。

不要做六个场景的大杂烩。

不要把重点放在复杂模型炫技。

不要为了真实第三方集成牺牲现场稳定性。

不要只生成漂亮纪要却没有后续动作。

## 两天内的优先级

P0 是会议转写到看板的稳定闭环。

P0 是一键演示路径和备用录屏。

P0 是交付提醒视图，包括即将到期、无人负责、阻塞任务。

P1 是下次会议复盘清单。

P1 是路演材料中的问题场景、解决方案、Demo 路径、技术路线和商业化可能。

P2 才是真实日历、飞书、微信或邮件集成。

## 验收标准

打开 `/track/efficiency` 后，评委应能立刻看到这是参赛主线。

点击演示按钮后，页面应出现结构化纪要。

切换到项目看板后，待办应自动变成任务卡片。

任务卡片应包含任务、负责人、截止日期和优先级。

页面应能解释下一步该做什么，而不是只展示已生成内容。

## 路演叙事

开场不要说提升效率。

开场要说：一人公司最怕的不是开会，而是会后没人把承诺盯到交付。

Demo 不讲六个功能。

Demo 只讲一条链路：转写进入、任务出来、看板追踪、交付提醒、下次复盘。

商业化不讲大平台。

商业化先讲 99 元/月替代一部分助理、项目经理和运营跟进工作。
