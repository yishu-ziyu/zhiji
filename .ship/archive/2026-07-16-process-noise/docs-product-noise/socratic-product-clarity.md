## Q1 — 用户要提供什么？

**对照：** ChatGPT = 指令 +（可选）文件/上下文。

**本产品用户交的（调研代码/API/CONTEXT 后）：**

| 必交/核心 | 形态 | 不交则 |
|-----------|------|--------|
| 项目 | 侧栏项目 / 打开记录 | 无法落到「这一摊」 |
| 材料 | 卡片（带来源）或项目下文件（先 md） | 无依据可点回 |
| 工作项 | 状态、负责人、下一步 | 无法排「请先看」 |
| 拍板 | 确认状态、评论、改下一步 | 库不「算数」 |

| 可选但值钱 | 形态 |
|------------|------|
| 关系 | A→B + 类型 + 来源句 |
| 检索词 | 在项目内搜材料/任务/记录 |
| 助手写回 | 经 API/MCP 写入的结果（算进库才算） |

**不要求：** 向量库、全盘监听、每天填项目管理大表。

**验收句：** 用户最少交「项目 + 至少一条带出处材料或工作项」才能看到非空局面；只有打开空项目则几乎无价值。

**已定：** 人和外部 Agent 可以写入事实；产品 Agent 可以根据新内容提出状态、依据和下一步。只有可能改变当前重点或下一步时才请求人确认，确认后才写入正式状态。


## Q1b — 「写入」的操作形态（刁难澄清）

**调研事实（代码）：**
- 主路径「写入」= `新建 → 新增项目材料` 弹窗：标题 + **textarea 粘贴/键入** + 来源下拉 → `POST /api/knowledge/add` → JSON 卡片。
- **无** `input type=file` / `onDrop` / 拖拽上传。
- 「本项目材料」= 读 `data/knowledge/files/<projectId>/` 磁盘文件；产品内**只读列表+打开**，无浏览器拖入写盘。
- `CapturePanel`（粘贴纪要）组件在，**未挂**当前项目画布主页。

**因此：** 说「用户提供材料」若指今天真机能做的 = **在弹窗里粘贴/打字成卡**；不是 ChatGPT 式拖文件。


## 产品形态分歧（进料）— 进行中

**禁止用语：** 「真机」（改为：浏览器里现在点什么会发生什么）。

**现在浏览器里实际进料（唯一通的）：**

1. 点「新建」
2. 选「新增项目材料」
3. 在弹窗大框里粘贴或打字
4. 选来源类型
5. 点加入
6. 变成一条库里的卡片（JSON）

**没有：** 拖文件、选本地文件夹、自动发现本机项目。

**已定主次：**
- 本机目录是主要入口，负责持续发现项目里的代码、文档和 Agent 产物变化。
- 粘贴和拖文件是补充入口，负责目录之外的聊天、会议材料和临时文件。

**风险：** C 在纯网页里默认做不到扫全盘；要本机助手、目录授权或桌面壳。


## 已定：本机目录 ↔ 项目

- 用户选**一个大文件夹**（授权范围由用户定）。
- 关系：**一对多** — 大文件夹下再拆成多个产品项目。
- Agent 综合一级目录、`.git`、`package.json` 等项目痕迹和内容结构，先列出候选项目。
- 用户首次授权后确认或调整一次；未经确认的候选不直接成为正式项目。


## 已定（暂）：谁管数据

- **不要**账号体系 / 腾讯式托管（第一版）。
- 服务对象：个人工作者、小团队，**本机用**，求管理更省事。
- 持久化：跟**跑服务的这台机器**上的数据目录走。
- 产品形态仍未想清；慢慢聊，不急锁壳。

## 已定：网页是本地项目的可操作映射

真正的过程是：

> 本地项目文件 → Agent 理解和整理 → 网页呈现当前状态 → 用户查看、编辑、回应和确认 → Agent 作用回本地项目 → 网页更新

- 网页不是另一个存放项目的地方，而是人与 Agent 共同工作的界面。
- 主画布只判断“现在该看什么”，不堆全部文件和结果。
- 点开某个对象后，再生成适合它的工作面：文档审阅、截图指位、代码差异、方案修改或执行确认。
- 网页中每个操作必须带着原项目、原对象和同一次任务的精确引用回到 Agent。
- Lavish 提供的是这种“本地内容 ↔ 网页工作面 ↔ Agent”的交互参考；Excalidraw 只是其中一种工作面，不是产品本身。

## 已定：谁写项目状态

- 人和外部 Agent 可以直接写入材料、执行结果等事实。
- 产品 Agent 读取新增内容后，可以提出状态、依据和下一步。
- 只有判断可能改变当前重点或下一步时才请求人确认。
- 未经确认的推测不进入正式项目状态，也不能替换已确认的下一步。


## Q1 重述（用类比钉死）

类比：
- 游戏：账号 → 平台才服务你
- ChatGPT：文字/文件 → 它才理你
- **本产品（第一版）**：不要账号。你要给的是：

| 最少给什么 | 不给会怎样 |
|------------|------------|
| **至少一个项目**（新建，或以后从本机大文件夹里认出来） | 没有「这一摊活」可管 |
| **至少一点内容**（粘贴成材料卡，和/或一条工作项：标题+下一步） | 打开只有空壳，谈不上先看啥 |
| **打开本机上的这个产品** | 它不会自己找上门 |

可选加码：关系+来源句、选本机目录授权、改状态/确认。

**验收：** 像 ChatGPT 没输入就不说话——这里没项目+没内容，就不该装成有智能局面。


## Q2 — 丢进来之后产品做什么（攻击键类比）

用户：拖项目 / 粘贴材料 = 按攻击键。  
产品必须有**立刻可见的反应**，不能静默失踪。

**目标反应（待你认）：**
1. 收进对应项目（或新建项目）
2. 变成可点回的材料/文件记录（带来源）
3. **重算「请先看」**：若这条改变局面，注意力更新；否则安静进库
4. 画布/列表能打开、能搜到这条
5. （有工作项语境时）可挂到事项或只当材料

**现在代码实际：** 弹窗粘贴 → 存 JSON 卡 → 可聚焦；**没有**拖项目全套处理。

**已定：** 默认安静入库；仅当可能改变当前重点或下一步时才更新「请先看」并（按已定规则）请求确认。用户确认：默认安静入。


## Q2 底下的业务逻辑（游戏伤害系统类比）

现象 1–4 不是随便 UI，背后规则：

**世界：** 多项目。每条材料/事项必须属于一个项目。

**两类实体：**
- **依据（材料）**：可打开、可搜、可当证据；本身不推进状态。
- **工作项**：有负责人/状态/下一步；这才是「会受伤/要人处理」的对象。

**命中判定（像砍空气 vs 砍怪）：**
- 砍空气：只进库、不挂任何进行中事项、不造成阻塞/待确认 → **存着，不改「请先看」**
- 砍中：连到进行中/阻塞/待确认/计划可能失效 → **改注意力**

**伤害怎么显示：** 「请先看」一条主因 + 能点回来源；不是弹一堆红字。

**列表/画布：** 是战场小地图，不是逻辑本身；逻辑是归属、证据、状态、注意力优先级。


## 回答粒度（用户强制）

- 先讲**能力从哪来的整条逻辑**，再讲字段/算法。
- 错误：产品直接读 status/阻塞…（跳过 Agent 层，假完整）。
- 正确骨架：用户丢入 → **Agent** 去看/读/整理 → 提出「状态/证据/下一步」→ **人确认后**才算数写入 → 界面按已确认记录显示。
- 细节（Agent 怎么读、用哪些信号）**等用户往下追问再答**。


## 思考纪律（用户强制）

- **禁止附和。** 用户抛方案 ≠ 正确。先找漏洞、替代、代价。
- 同意必须附：比过哪些选项、为何这个更差/更好。
- 提问必须先给足判断所需上下文：为什么现在要定、不同答案如何改变产品、代价是什么、当前建议及依据。禁止让用户在信息不足时盲选。
- 角色：顶尖 thinker 对问，不是秘书复读。

## Q3 — 提效（已定）

**用户最先要干掉的浪费：** 少翻聊天找「上次干到哪」。

**主结果（开发与验收只认这个）：** 恢复项目并开始**正确的**下一步。  
即打开后迅速知道：上次做到哪、后来发生什么、现在该做什么，并开始行动。

**不作为主结果（防抢戏）：**
- 更快找到材料 → 易沦为搜索工具
- 更快录入 → 只是入口顺，不证明工作推进

**质量约束（非主 KPI，但违规算失败）：** 减少错误行动与返工。

**怎么量：** 从打开一个已经记不清进度的项目开始，到用户确认继续或修改下一步，并实际打开材料、开始工作项或调用 Agent。若没有新信息进入，却因产品漏掉授权范围内原本已有的信息而推翻决定，则本次失败。

来源：用户认可的 GPT 候选框架 + 本人「少翻聊天」优先级。

**具体例子：** 昨晚确认“继续修改画布”；离开后出现测试失败和新研究；今天打开产品后，Agent 说明原下一步、后来变化、建议和依据。用户确认继续或改为先处理测试，并实际开始对应工作。打开到开始行动的时间就是恢复时间。


## 已定：Agent 何时记录和分析

- 目录变化立即记录，不等用户打开项目。
- 连续编辑期间先收集，一段工作暂停后再合并分析，避免每次保存都重算。
- 测试结束、外部 Agent 回传等明确事件发生后，尽快分析。
- 用户打开项目前，补齐尚未处理的变化。
- 只有分析可能改变当前重点或下一步时，才请人确认。


## 已定：讨论与开发并行

## 已定：打开项目时画布先呈现什么

这个问题会决定首份开发文档的主界面行为。

- 项目保持在中心。
- Agent 在周围动态挑出少量当前值得关注的对象。
- 用户点击其中一个后，它成为新中心，再展开材料、关系、任务和执行记录。
- Agent 不在第一屏直接用它建议的单一下一步替换项目中心。


## 已定：Agent 怎么选第一屏对象

打开项目时，候选内容可能包括：上次确认的重点、新代码变化、测试失败、未读文档、阻塞项和临近截止时间的任务。

- 不建议只按时间最新排序：这会退化成活动记录。
- 建议：必须保留“上次由人确认的重点”作为恢复记忆的起点；其他对象按“是否可能改变当前计划”选入。
- 不强行凑数：只有一个重要对象就只展示一个；上限再另行确定。
- 每个被选中的对象必须能回答“为什么现在显示它”，并能点回依据。


## 已定：时间在产品中的作用

- 时间是 Agent 恢复项目状态的重要依据，但完整时间线不是第一屏的主界面。
- 底层完整保留先后记录；Agent 主动比较“上次确认”与“现在”，提取会改变当前计划的变化。
- 画布节点只显示理解当前局面所需的关键时间。
- 用户主动“查看变化”时再展开完整记录。
- 展开形式不做普通列表；希望用有趣的动态展示画布随时间变化。具体动效在当前状态定义清楚后再设计。


## 已定：“当前状态”到底是什么

当前建议：当前状态不是 `active / paused` 一个标签，也不是 Agent 即兴写的一段总结。它是“上次由人确认的项目记录 + 之后已发生且仍然有效的变化”，具体回答：

1. 当前在追求什么。
2. 哪些事实、决定和结果已确认。
3. 哪些工作正在进行。
4. 哪些工作阻塞、等待或尚未确认。

“下一步”不是状态本身，而是 Agent 根据当前状态和关系提出的行动建议。


## 已定：画布上的“关系”到底是什么

当前建议：关系不是共现、共享关键词或“两个东西有点相关”。第一屏只展示会改变用户对当前状态或下一步理解的关系，例如：

- 测试失败 **阻塞** Demo 验收。
- 已确认设计决定 **约束** 当前画布实现。
- 会议记录 **支持** 某条产品需求。
- 外部 Agent 结果 **改变** 某个工作项状态。

每条线都必须用一句普通话说清关系，并能点回依据。Agent 提出但未经确认的关系必须与已确认关系区分。

## 讨论中：产品在哪一层执行

### 官方材料确认的边界

- 作品需要让 Agent 进入真实场景、完成一段具体任务，并形成可体验、可见的输出。
- 工具和实现路径开放：代码、低代码、模型 API、工作流组合都可以。
- 公司/一人公司效率 Agent 的官方例子包括项目管理、风险识别、状态更新、跟进和行动建议。
- 官方没有要求产品连接所有外部工具。使用 MCP 时提交协议或链接；未使用 MCP 时可说明主要连接方式和技术路径。
- 90% 人工评分主要看场景价值、商业化、创新、体验和路演；10% 技术评分看联通、稳定和核心流程可运行。

### 已定：三层执行边界

1. **产品内动作（所有用户都有）：** 读取授权信息，更新项目对象，生成可执行的下一步，准备上下文和验收标准。这是产品操作自己管理的数据，不等于替用户完成外部工作。
2. **人工交接（没有工具连接时）：** 产品打开目标材料、准备完整任务，人在外部工具中完成实际动作，产品再记录用户手动回传或授权目录中的结果变化。这是备用路径，不算自动执行。
3. **外部工具直接执行（已连接时）：** 把任务、对象、依据、权限和验收标准发给 Codex、Claude Code 或其他已连接工具，接收过程与结果并写回项目。

比赛 Demo 建议只连一个真实执行器（优先 Codex 或 Claude Code）跑通完整过程。商业产品保留统一任务表达，再按真实需求增加其他连接，不在第一版声称支持所有工具。

因此，之前的“下一步”四项还不够，还需要明确：由谁/哪个工具执行、允许做什么、结果如何回到项目。

用户已接受这三层边界。后续提供的资源需根据四个事实评估：能否接收结构化任务、权限是否可控、能否返回过程和结果、结果能否写回项目。

### 已核实资源：OOMOL OpenConnector

用户提供的资源对应 [`oomol-lab/open-connector`](https://github.com/oomol-lab/open-connector)，不是同名的 `openconnector/open-connector`。

已核实：

- Apache-2.0，支持 Docker 自部署；官方仓库声明覆盖 1,000+ provider、10,000+ Action。
- 可通过 SDK、CLI、MCP、HTTP 或 OpenAPI 调用；HTTP 可发现 Action、读取输入/输出结构并执行具体 Action。
- Action 带输入结构、所需权限和连接身份；运行时支持允许/禁止策略。外部账号凭证留在网关内，不交给产品 Agent。
- 执行结果使用统一响应结构返回，并带执行标识；可作为现有 `WorkEvent(type="result")` 的外部执行结果写回项目。
- 自部署开源运行时的官方定位是“一台服务器、一个用户”，与本地个人版相符；若以后支持多租户，需要使用托管的 `ProjectConnector` 或另做严格隔离。

对本产品的直接影响：

- 它能把 Gmail、Notion、Slack、GitHub 等“有 API 且存在 Action”的工作，从人工交接变成外部工具直接执行。
- 它不能执行任意本机工作，不能自动操作没有 API/Action 的软件，也不能代替 Codex/Claude Code 完成代码修改。
- 因此它应是统一执行接口下的 **SaaS 执行适配器**，不是产品 Agent 本身，也不是唯一执行器。Codex/Claude Code 属于另一类本机开发执行器，人工交接继续作为最后路径。
- 产品不加载全部 Action。只暴露用户已连接、当前任务需要、权限允许的 Action；执行前展示目标对象、将发生的修改和权限，执行后把成功、失败、外部对象标识和执行标识写入项目事件。

用户已接受“统一执行接口 + OpenConnector 负责 SaaS + 本机 Agent 负责开发 + 无连接时交给人”的结构。

### 已定：Lavish 的位置

用户已通过 Lavish 审阅页确认：

1. **现在立即用于开发协作。** 复杂产品方案、关系图和界面截图优先用 Lavish 呈现与局部批注，减少“你指哪里”的解释成本；简单问题继续直接对话。
2. **最终产品暂不直接嵌入 Lavish。** 先把“人确认任务 → Codex 改代码并跑测试 → 结构化结果写回 → 人确认结果”稳定跑通。
3. **闭环稳定后再加可选详细审阅。** 只有复杂结果需要局部反馈时才打开；第一批优先验证界面截图和产品方案/关系图。
4. **Lavish 不负责授权或正式状态。** 批注只是给同一次执行的修订意见；只有产品内明确确认后，结果才改变项目状态。

当前 `lavish-axi@0.1.42` 是 MIT 本机 CLI/Express 工具，不是稳定 SDK。直接嵌入意味着额外维护认证、文件权限、会话持久化和升级，现阶段不做。

### 已定：Agent 生成白板的第一个用途

白板可以由 Agent 根据项目对象、关系和证据生成，但不做“边执行边连续重排”。建议行为是：

1. 编辑暂停、测试结束、Codex 回传或计划前提被推翻时，Agent 先分析。
2. 只有关系复杂到普通文字说不清时，才立即生成一个冻结版本。
3. 用户审阅期间不自动重画；新变化只提示可生成新版本。
4. 移动、缩放和手绘只改变表达；新关系、改名和下一步都只生成待确认建议。
5. 主画布仍是项目状态入口；白板只解释一次复杂变化或执行结果。

第一个用途已确定为 **Codex 执行证据图**：已确认任务 → 改动文件 → 测试命令与结果 → 剩余风险 → 建议下一步。用户已要求把它交给 Grok；但它必须等真实 Codex 执行与可信写回通过主 Codex 验收后才能进入产品主路径。

技术上，Lavish 当前是 Mermaid 源文件 → 可编辑 Excalidraw 场景；源文件变化会整图重建，不是节点级双向同步。近期实验可以把 Lavish 作为本机审阅工具；如果未来变成正式网页能力，应在产品内原生使用 Excalidraw，并用稳定的业务对象 ID 和证据绑定节点，不依赖 Lavish 的私有接口或图中位置。
