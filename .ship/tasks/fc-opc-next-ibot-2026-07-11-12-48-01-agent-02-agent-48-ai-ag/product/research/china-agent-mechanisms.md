# 中国消息生态与 Agent 机制专项研究

> 研究日期：2026-07-13<br>
> 研究范围：中国主流协作产品、智能体平台及其官方开放平台/API；重点核验微信、企业微信、钉钉、飞书的消息入口边界。<br>
> 判断链路：`原始信息 → 结构化业务对象 → 行动 → 双方确认/验收 → 可复用记忆`。<br>
> 四类问题：按当前产品假设中的四类业务对象界定为**报价/范围、承诺、项目、任务/等待与验收**。

## 一、结论

### 1. 中国市场并不缺“消息变任务”或“AI 工作流”

飞书已经提供消息事件、历史消息、交互卡片、多维表格、任务、审批和开放 API；钉钉提供机器人收发、待办、项目任务、审批、AI 表格、MCP 和大量 OpenAPI；企业微信提供客户联系、微信客服、会话存档和客户触达；飞书 aily、腾讯云智能体开发平台、腾讯元器、扣子均已把知识、工作流、工具调用、长期任务或记忆产品化。

因此，下列链路已经是常规能力组合，不构成独特 Agent 机制：

```text
机器人收到消息
→ 大模型摘要/抽字段
→ 写入表格或创建任务
→ 发卡片/提醒
→ 用户点确认
→ 数据库保存记录
```

这条链可以用飞书机器人 + 多维表格自动化 + 任务 API，也可以用钉钉机器人 + 待办/项目 API + 互动卡片实现。平台选择、节点数量和是否使用大模型，都不改变它作为普通工作流组合的本质。

### 2. 真正需要 Agent 的只有“模糊信息下的跨对象判断”

在输入已经结构化、对象和动作规则明确时，确定性代码优于 Agent。Agent 的有效职责应限制在：

- 从含省略、代词、条件、否定、反悔和多轮上下文的客户语言中识别候选变化；
- 把新信息与现有报价、承诺、项目和任务进行实体解析与版本对比；
- 发现跨对象冲突、缺失前提和下游影响；
- 在证据不足时明确弃权并请求补充，而不是补全事实；
- 为人生成可审查的行动方案，不替代价格、范围、承诺和验收的权威决定。

原始信息采集、批准动作、调用 API、双方确认、状态机、回执验证和事实入库都不要求 Agent；这些部分应尽可能确定化。

### 3. 最有区别的机制不是“会记忆”，而是“证据门控的事实晋级”

腾讯云智能体开发平台已经提供会自动从对话抓取个性化信息、保存 1–999 天并允许修改/删除的[长期记忆](https://cloud.tencent.com/document/product/1759/104206)；飞书 aily、腾讯元器和扣子也已提供知识、工作流、工具或长期任务。单纯宣称“越用越懂你”“有长期记忆”没有区分度，而且可能把误解永久化。

FC-OPC 可测试的差异是：

```text
候选理解
→ 服务方批准的内部事实
→ 经目标系统回执或客户本人动作验证的外部事实
→ 才可进入下一次行动的长期记忆
```

每次晋级都必须携带原始证据、对象版本、操作者、时间、权限范围和撤销关系。Agent 可以提出“什么可能变了”，但不能自己宣布“客户已同意”或“交付已验收”。

### 4. 最大入口约束是个人微信，不是模型能力

本次核验的微信官方能力没有提供第三方产品读取任意个人微信聊天历史的通用 API。可行入口都会改变用户动作或渠道：

- 用户主动复制文本或上传截图/文件；
- 用户把素材用小程序打开；
- 客户进入微信客服会话；
- 服务者把客户工作迁到企业微信，并开通有管理员范围和客户同意要求的会话存档。

所以“自动监听个人微信全部客户聊天”“无感读取历史消息”“零操作接入微信”都不是可辩护的产品前提。

## 二、逐环节判断：工作流、Agent 与人的边界

| 链路环节 | 常规产品/自动化已经能做 | Agent 只有在何时有增量 | 人必须保留什么决定 | 关键失败条件 |
|---|---|---|---|---|
| 原始信息进入 | Webhook、事件订阅、机器人会话、客服会话、会话存档、复制/截图上传 | 基本不需要 Agent；OCR/转写可用模型，但不解决渠道权限 | 是否提供该段证据、是否允许存档、客户身份关联 | 入口没有授权；把转发/引用当成客户本人新消息；漏掉附件或撤回 |
| 结构化业务对象 | 表单字段、规则解析、数据库 schema、任务/报价 API | 自然语言模糊、多轮指代、对象消歧、与旧版本比较、冲突发现 | 价格、范围、责任、截止日期、验收口径是否成为正式事实 | Agent 造出不存在对象；遗漏高风险影响；无法定位版本仍强行写入 |
| 行动 | 创建/更新任务、表格记录、项目字段、审批、消息、卡片、API/MCP 工具 | 动作集合随上下文变化，需要规划或异常处理时 | 高风险写入、对外发送、报价/承诺变更、删除和不可逆动作 | 重复执行、越权、部分成功、没有幂等和补偿 |
| 双方确认/验收 | 客户门户、链接、卡片回调、审批、状态机 | Agent 可解释差异、生成澄清问题；确认本身不需要 Agent | 服务方只能批准自己的行动，客户只能确认/拒绝自己的状态 | 服务方代点客户确认；点击者身份不可信；旧版本仍可确认 |
| 可复用记忆 | 业务数据库、事件日志、知识库、版本历史 | 从已验证事实中检索相关上下文、识别矛盾和过期事实 | 哪类证据足以晋级、保留多久、谁可撤销 | 把未验证摘要写成长久事实；权限变化后记忆继续泄露；撤销未传播 |

判断阈值很简单：**如果把大模型节点替换为字段映射和条件分支，结果仍然可靠，那个环节就不是 Agent 增量。**

## 三、中国主流产品的现有能力基线

### 3.1 飞书：消息、业务对象、动作与确认均已有官方拼装面

飞书是本次最完整的“常规工作流”反例：

- [接收消息事件](https://open.feishu.cn/document/server-docs/im-v1/message/events/receive)可按权限接收用户发给机器人的单聊消息、群内 @ 消息；读取群内全部用户消息需要额外敏感权限。事件可能重复，官方要求以 `message_id` 去重。
- [获取会话历史消息](https://open.feishu.cn/document/server-docs/im-v1/message/list)支持单聊和群聊，但机器人必须在被查询群组中，并受应用/用户令牌、权限和可用范围限制。
- 2024-11-14 起，飞书对相关事件和 API 执行[历史消息可见性管控](https://open.feishu.cn/document/platform-notices/breaking-change/visibility-control-of-historical-messages)：新成员不可见的历史消息也不会被机器人读取或操作。
- [发送消息 API](https://open.feishu.cn/document/server-docs/im-v1/message/create)可发文本、富文本、卡片、图片、文件等；发用户要求用户在机器人可用范围内，发群要求机器人在群内且有发言权限。
- [卡片回传交互](https://open.feishu.cn/document/feishu-cards/card-callback-communication)会返回点击者 Open ID 和交互数据，适合做“批准/拒绝/补充”；这是确定性的业务回调，不需要 Agent。
- [多维表格自动化插件](https://open.feishu.cn/document/base-extensions/base-automation-extension-development-guide)允许定义输入、执行动作和结构化输出，结果可传给后续节点；[多维表格 OpenAPI](https://open.feishu.cn/document/server-docs/docs/bitable-v1/bitable-overview)可创建、读取和更新业务记录。
- [飞书任务 v2](https://open.feishu.cn/document/task-v2/overview)已覆盖任务、清单、成员、子任务、评论、自定义字段和附件的增删改查，并为创建任务等操作提供幂等机制。
- [飞书审批状态事件](https://open.feishu.cn/document/server-docs/approval-v4/event/common-event/approval-task-event)已能返回待审批、通过、拒绝和完成等确定状态。

飞书 AI 也已经跨过“总结与知识问答”阶段。当前[飞书 AI 说明](https://www.feishu.cn/hc/zh-CN/articles/257240089464-%E5%BF%AB%E9%80%9F%E4%BA%86%E8%A7%A3%E9%A3%9E%E4%B9%A6-ai)把企业知识问答定义为基于成员可访问的消息、文档和知识库；智能会议纪要可提炼待办。2026 年的[飞书 aily 自定义智能体说明](https://www.feishu.cn/content/article/7631864469689240764)证明它可操作飞书文档、多维表格和任务，接入技能/MCP，并按定时任务主动工作；某些飞书内动作仍需要企业管理员审核权限。

**结论**：`飞书消息 → AI 抽取 → 多维表格/任务 → 卡片确认 → 状态回写` 已是普通平台组合。FC-OPC 只有在类型化商业影响、外部客户权威和证据门控记忆上才可能有增量。

### 3.2 钉钉：机器人入口与业务动作成熟，但入口不是全量聊天监控

钉钉 2026-06-30 更新的[机器人接收消息文档](https://open.dingtalk.com/document/dingstart/robot-receive-message)支持 Stream 和 HTTP 两种方式，包含单聊与群聊；官方示例要求先把机器人加入群，用户在群内 @ 机器人后才把消息推给服务端。它证明的是**面向机器人的会话输入**，不是读取任意员工历史私聊或所有群消息。

[机器人回复/发送消息](https://open.dingtalk.com/document/dingstart/robot-reply-and-send-messages)支持 Webhook 和服务端 API，可发群聊、人与机器人会话消息；部分服务端发送方式不支持 @，人与人会话中的机器人普通消息仅限单聊酷应用场景。

动作侧已经非常完整：

- [钉钉待办任务](https://open.dingtalk.com/document/orgapp/dingtalk-todo-task-overview)提供创建、更新、删除、执行者状态和查询列表。
- 钉钉项目 API 可[更新任务负责人、优先级、参与者、开始/截止时间和工作流状态](https://open.dingtalk.com/document/development/update-task-participants)。
- 当前[开放平台能力中心](https://open.dingtalk.com/)已经公开机器人消息、AI 表格、文档、日历和待办 MCP；待办 MCP 可创建待办、更新完成状态并查询列表。
- 官方[开放能力库](https://open.dingtalk.com/developer/list)把自然语言触发审批/报销/入职流程、企业知识检索和跨系统任务联动作为现有 AI 解决方案。

**结论**：钉钉内的自然语言理解、创建待办、更新项目、发消息和走审批都不能作为创新。有效边界是：机器人仅获得面向它的消息或授权数据；外部客户若仍在个人微信，钉钉无法替代该入口。

### 3.3 企业微信与微信客服：有真正的客户消息入口，但配置、同意和时限明确

企业微信提供两条不同的客户消息路径，不能混为一谈。

#### 路径 A：会话内容存档

[会话内容存档概述](https://developer.work.weixin.qq.com/document/path/91360)允许企业统一设置存档员工范围并通过 API 获取这些员工的工作沟通内容。[使用前帮助](https://developer.work.weixin.qq.com/document/path/91361)进一步规定：管理员要配置开启范围、调用 IP 和消息加密公钥；员工会被告知；企业只能获得涉及已开启员工的对话；员工与外部联系人的会话必须得到外部联系人同意。

这条路径输入最强，但它是企业级、管理员配置、员工范围受控且客户同意的能力，不适合被描述成个人微信连接器。

#### 路径 B：微信客服

[接收消息和事件](https://developer.work.weixin.qq.com/document/path/94670)覆盖微信客户在**微信客服会话**中发送的消息：企业微信先回调，再由企业调用 `sync_msg` 读取最近 3 天的内容和事件。它包含客户消息、接待人员消息、菜单回复、发送失败、会话状态和撤回等事件，并要求仅操作企业指定的客服账号和可见范围内的接待人员。

[发送消息](https://developer.work.weixin.qq.com/document/path/94677)只允许在客户主动发消息后的 48 小时内发送，最多 5 条；客户继续发送后才能再次下发。接口返回成功不代表最终送达，官方要求继续关注发送失败事件回调。这恰好说明“执行后验证”应由确定性回执完成，而不是让 Agent 自报成功。

微信客服的菜单消息还可让客户点击“满意/不满意”，点击会生成带菜单 ID 的客户回复。它可用于轻量确认，但若涉及报价、范围和验收，仍应使用带版本、身份和撤销规则的独立页面或小程序。

企业微信的[群发消息给客户](https://developer.work.weixin.qq.com/document/path/93555)允许有客户联系权限的成员把文本和附件传到群发助手，并受同一成员每天向一个客户最多一次等限制。它不是无限制、静默的 Agent 外发通道。

**结论**：如果目标客户能接受微信客服或企微，会出现可产品化的官方入口；如果他们坚持在已有个人微信私聊，仍必须主动选择证据或改变渠道。

### 3.4 个人微信与小程序：可以接收“用户主动交付的素材”，不能读取任意聊天历史

微信的[聊天素材打开](https://developers.weixin.qq.com/miniprogram/dev/framework/material/support_material.html)允许用户把聊天中的文件、图片、视频和网页/公众号文章交给审核通过且用户曾使用过的小程序处理；小程序必须声明素材类型并通过审核。普通聊天文本历史不在页面列出的开放对象中。

[小程序客服消息](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/customer-message/customer-message.html)要求用户点击进入小程序自己的客服会话；开发者收到的是该会话中的消息和事件，不是用户原有客户聊天。

因此个人微信 MVP 的官方可行顺序是：

1. 用户复制相关消息到 Web/小程序；
2. 用户上传或用小程序打开截图/文件；
3. 让客户进入微信客服会话；
4. 对已有企微工作方式的客户使用会话存档。

截图 OCR、转发和复制都可能有价值，但它们是**显式用户选择**，不是“自动微信接入”。

### 3.5 飞书 aily、腾讯智能体平台、腾讯元器与扣子：Agent 基础能力已经商品化

| 平台 | 官方已证明的能力 | 对本项目的反证 | 本次未证明的能力 |
|---|---|---|---|
| 飞书 aily | 企业知识、技能/MCP、文档/多维表格/任务动作、定时任务、对话中调整人设或学习技能 | “懂企业上下文、会调用工具、会主动工作”不是创新 | 没有证明以客户独立证据门控商业事实记忆 |
| 腾讯云智能体开发平台 | RAG、工作流、多 Agent、工具/API、API 接入、长期记忆、发布版本 | 工作流编排、工具调用、长期记忆均是平台能力 | 长期记忆抓取的是个性化信息，不等同于已验证业务事实 |
| 腾讯元器 | RAG、工作流、多 Agent，可发布到微信客服、公众号/服务号、企业微信等 | “把 Agent 接入微信生态”本身已有官方产品 | 发布渠道不等于读取任意个人微信旧会话 |
| 扣子 | Agent skills、自然语言生成工作流、长期任务、数据库/应用部署等职场生产能力 | 多节点智能工作流和长期后台任务已是大众产品 | 官方产品页未证明严格的逐动作审批和双边事实状态 |

腾讯云的[工作流编排](https://cloud.tencent.com/document/product/1759/122556)明确支持 Agent 自主调用工具、多个 Agent 节点和稳定可控的流程；[应用设置](https://cloud.tencent.com/document/product/1759/104206)提供变量和长期记忆。腾讯元器的[官方介绍](https://yuanqi.tencent.com/guide/yuanqi-introduction)列出 RAG、工作流、多 Agent 以及微信客服、公众号、服务号和企业微信发布渠道；其[发布到企业微信](https://yuanqi.tencent.com/guide/publish-agent-work-wechat)实际是在企微工作台打开应用对话，不代表读取员工已有聊天。

[扣子当前产品页](https://www.coze.cn/overview)已公开 Agent 技能、长期任务和自然语言生成工作流；这些能力可作为底层，但不能充当领域创新证明。

## 四、四类问题的最小产品与 Agent 分工

### 4.1 报价/范围变化

| 项目 | 最小要求 |
|---|---|
| 最小产品动作 | 保存当前有效报价/范围版本；从新消息生成逐字段 change draft；显示原文和 before/after；服务方批准；客户确认具体版本；批准后才更新报价状态和项目预算/范围 |
| Agent 职责 | 处理“这个先不要了”“按上次那个价格”“下周多加一版”等模糊语言；检索旧报价；识别数量、单价、范围、条件和版本冲突；证据不足时询问 |
| 人保留的决定 | 定价、折扣、是否收费、是否接受 scope change、最终对外文本与生效时间 |
| 可衡量结果 | 从消息到变更草稿时间；人工修订率；高风险影响召回率；确认时长；24/48 小时后报价与项目范围仍不一致的比例；未收费范围外工作的数量/金额 |
| 失败条件 | 无法定位有效报价仍生成差异；把讨论价当承诺价；客户未确认却覆盖旧版本；不同币种/税费/数量被静默猜测 |
| 是否只是备忘录/自动化 | 若只摘要“客户要加需求”并提醒报价，是备忘录；若字段已明确，仅写表和发审批，是自动化；只有含上下文的版本差异与冲突判断需要 Agent |

### 4.2 承诺变化

| 项目 | 最小要求 |
|---|---|
| 最小产品动作 | 形成结构化承诺候选：承诺方、对方、交付物、条件、截止时间、验收标准、来源；服务方审阅；客户独立确认/要求修改；旧承诺保留版本关系 |
| Agent 职责 | 识别代词、省略、条件句、否定、反悔、引用消息和多人角色；判断是意向、询问、条件同意还是正式承诺；找出与已有承诺的矛盾 |
| 人保留的决定 | 是否接受该承诺、是否需要改期/加价、谁有权代表服务方或客户确认 |
| 可衡量结果 | 承诺抽取 precision/recall；错误承诺晋级率；客户修改率；争议时定位原证据的时间；被旧承诺继续驱动的事件数 |
| 失败条件 | 服务方可代替客户确认；转发内容被当成发送者承诺；条件没有记录；旧版本仍可被点击确认；客户身份无法验证 |
| 是否只是备忘录/自动化 | 摘要和提醒不够；确认页面和状态机本身是普通工作流；Agent 增量只在承诺语义与版本冲突识别 |

### 4.3 项目/里程碑影响

| 项目 | 最小要求 |
|---|---|
| 最小产品动作 | 将已批准变化映射为范围、里程碑、依赖、资源和基线日期的候选 diff；项目负责人批准后，通过飞书/钉钉/现有 PM API 更新；保存写入回执和失败补偿 |
| Agent 职责 | 在多个项目、相似命名和隐含依赖中找到候选受影响对象；解释为什么受影响；发现跨项目资源/承诺冲突；给出不确定性和遗漏风险 |
| 人保留的决定 | 调整基线、重排资源、接受延期、对哪个客户升级沟通 |
| 可衡量结果 | 受影响对象 recall；误写率；变化发生到项目基线更新的时间；24/48 小时仍残留旧状态的对象数；回滚成功率 |
| 失败条件 | 项目基线本身过期；没有依赖图却声称找全；Agent 直接覆盖多个项目；API 部分成功但系统显示整体成功 |
| 是否只是备忘录/自动化 | 规则已知的字段级联是自动化；“延期三天→更新所有子任务”不是创新。跨项目语义链接和冲突检测才可能需要 Agent |

### 4.4 任务、等待与验收

| 项目 | 最小要求 |
|---|---|
| 最小产品动作 | 创建/更新/取消任务；明确当前责任方、等待的证据和升级时间；向客户发带版本的确认/验收入口；只有客户动作或目标系统回执才能关闭对应状态 |
| Agent 职责 | 从上下文判断下一步到底由谁行动、等待什么证据、何时异常；在标准提醒无效时提出澄清或升级；解释任务与承诺/里程碑的联系 |
| 人保留的决定 | 优先级、是否催客户、是否接受交付、是否关闭争议、是否升级商业关系 |
| 可衡量结果 | 重复任务率；错误责任人率；等待超时；客户确认/验收时长；服务方自报完成但客户未验收的比例；回执与内部状态不一致率 |
| 失败条件 | 把“消息已发送”当客户已读/同意；把“服务方完成”当客户验收；无幂等导致重复任务；提醒噪声增加但等待没有缩短 |
| 是否只是备忘录/自动化 | 创建任务、定时提醒、卡片按钮和状态流转都是普通自动化；只有证据责任判断、异常解释和跨承诺影响有 Agent 增量 |

## 五、入口的真实技术边界

| 入口 | 能获得什么 | 不能获得什么 | 必要条件 | 对 MVP 的建议 |
|---|---|---|---|---|
| 个人微信现有私聊 | 官方未发现通用读取 API；可由用户复制、截图或选择素材 | 任意历史文本、静默实时监听、无授权全量导入 | 用户主动选择；小程序素材能力需使用过、声明类型并过审 | 默认采用粘贴/截图，明确“用户选择证据”是隐私和准确性机制 |
| 微信小程序客服 | 用户在小程序客服会话里发来的消息/事件 | 用户原有个人聊天 | 用户主动进入客服会话；配置服务端 | 适合作为转发箱，不要宣传成微信历史接入 |
| 企业微信会话存档 | 已开启员工的内部及获同意的外部联系人会话 | 纯未开启员工会话；未同意的外部会话；个人微信任意历史 | 企业管理员、开启范围、IP、公钥、员工告知、外部联系人同意 | 只面向已有企微习惯的工作室/企业 |
| 微信客服 | 指定客服会话最近 3 天消息与事件；48 小时窗口内可回复 | 其他微信私聊；无限主动外发 | 指定客服账号、接口权限、客户先发消息 | 是微信生态中最现实的闭环实验入口，可用失败事件做执行验证 |
| 钉钉应用机器人 | 单聊与群内 @ 机器人的消息，及机器人会话输出 | 标准机器人文档未证明可读取任意历史私聊/所有群消息 | 创建应用、配置 Stream/HTTP、加群、用户 @ 或发起单聊 | 适合内部运营，不解决个人微信客户入口 |
| 飞书应用机器人 | 单聊、群 @；敏感权限可收群内全部用户消息；可读机器人所在且可见的会话历史 | 默认不获得所有消息；不可见历史仍不可读；可用范围外用户不可触达 | 机器人能力、事件订阅、权限、群成员身份和历史可见性 | 适合飞书原生团队，外部客户是否在飞书需先验证 |

### 对入口声明的硬约束

不能说：

- “自动读取个人微信全部聊天”；
- “无需用户操作导入微信历史”；
- “机器人加入钉钉/飞书后就能看到所有旧消息”；
- “企微会话存档无需客户知情或同意”；
- “接口返回成功就代表客户已收到、已读或同意”。

可以说：

- “支持用户主动选择相关聊天证据”；
- “支持微信客服、企微存档、飞书/钉钉机器人等受权限约束的官方入口”；
- “执行结果以官方失败事件、目标系统状态或客户本人动作验证”；
- “接入范围取决于渠道、租户权限、机器人所在会话和历史可见性”。

## 六、最小可验证产品机制

### 6.1 推荐的闭环

```text
用户主动选择一段客户证据
→ 系统关联客户/项目和当前有效对象版本
→ Agent 只生成四类候选影响与不确定性
→ 服务方逐项批准、编辑或弃权
→ 确定性执行器以幂等键写入目标系统并保存回执
→ 客户通过独立身份入口确认承诺/变更或验收交付
→ 只有经内部批准或外部验证的事实进入长期记忆
→ 下一次事件同时显示新证据与当前有效事实
```

### 6.2 最小数据结构

```yaml

  source_id: immutable-id

  raw_excerpt: original-text

  object_type: quote | commitment | project | task

  required_authority: provider | customer | both

  idempotency_key: stable-key

  level: candidate | internally_approved | externally_verified
```

### 6.3 评测门槛

| 机制 | 最低指标 | 立即停止条件 |
|---|---|---|
| 消息→四类影响 | 分对象 precision/recall、未解析率、人工修订率 | 高风险遗漏无法下降；Agent 总是给确定答案而不弃权 |
| 人工批准 | 审批耗时、拒绝/修改原因、批准后误写率 | 审批只是一键放行，无法看目标、参数和证据 |
| API 执行 | 幂等成功率、部分失败率、补偿/回滚成功率 | 重试制造重复任务或旧状态被覆盖 |
| 双方确认/验收 | 客户身份成功率、完成率、确认时长、旧版本误确认率 | 服务方能代点；旧链接仍改变当前状态 |
| 证据门控记忆 | 未验证事实晋级率、来源可追溯率、撤销传播率、错误记忆恢复率 | 候选摘要可直接驱动高风险动作；撤销后旧事实仍被检索执行 |

## 七、伪 Agent 风险

最大的伪 Agent 风险是：**用 LLM 生成一段看似聪明的摘要，后面仍是固定的建任务、发提醒和写表格，却把整个流程称为“自主经营 Agent”。**

识别标准：

- 没有读取当前有效报价/承诺/项目/任务版本，只看当前消息；
- 没有对象级 before/after，只输出摘要或待办；
- 没有不确定性、冲突和弃权；
- 没有逐项权限和人类批准；
- 没有目标系统回执或客户独立动作；
- 把日志当记忆，把对话历史当业务事实；
- 入口依赖复制粘贴，却宣传为“自动接管微信”；
- 成功指标只有生成速度，没有遗漏、误写、等待和状态一致性。

只要上述任一高风险缺口存在，产品都更接近“备忘录/提醒/自动化组合”，而不是能够承担业务判断的 Agent。

## 八、最终建议

1. **入口先诚实**：MVP 默认采用用户主动粘贴/截图；把选择证据设计成准确性和隐私边界，不承诺个人微信无感接入。若要做官方微信闭环，优先实验微信客服；企微存档只服务已有企微流程的客户。
2. **Agent 只做候选影响**：让 Agent 负责自然语言消歧、跨对象关联、冲突发现和弃权；不让它自行确定价格、承诺、验收或事实晋级。
3. **其他环节确定化**：批准、执行、幂等、回执、双方确认、状态机和版本历史全部由确定性程序保证。
4. **把“双边权威 + 证据门控记忆”做成核心协议**：服务方不能代表客户确认；客户动作或目标系统回执是事实升级的必要证据。
5. **不要重做飞书/钉钉**：把它们当写回目标和企业入口。若目标用户已有飞书/钉钉，FC-OPC 应做 overlay；若客户主要在个人微信，产品价值首先取决于用户是否愿意主动选择证据。

最窄且可 defend 的陈述是：

> FC-OPC 不把聊天摘要直接当业务事实。它让 Agent 生成报价、承诺、项目和任务的带证据候选差异，由有权的人批准并由目标系统或客户本人验证；只有验证后的变化才能成为下一次行动使用的记忆。

这项机制仍需以真实事件回放、对象级准确率、权限测试和执行回执证明。在此之前，它是待验证的产品设计，不是已经成立的技术创新。
