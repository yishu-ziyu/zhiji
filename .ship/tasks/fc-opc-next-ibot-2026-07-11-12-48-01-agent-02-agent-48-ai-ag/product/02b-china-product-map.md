# 中国场景产品决策与开发地图

日期：2026-07-13

状态：当前产品判断。它替代旧稿中把 `Waiting-for` 作为比赛第一刀的结论；
市场验证仍未完成，本文不是“需求已经验证”的声明。本文给出的是首轮招募条件、
比赛机制和开发边界，不把演示样例当作中国市场频率或付费证据。

交叉审查指出的客户身份、逐项权威、跨对象更新、版本 CAS、证据机器校验和单一 E2E
已经进入本文的开发要求；它们尚未实现，不能把这份设计稿当作运行证据。

## 一句话结论

我们先验证这样一类中国独立服务者：同时交付多个非标准项目，客户在微信等外部沟通中
改变需求时，Agent 找回当前有效的范围、价格、日期和验收约定，生成带原文证据的
影响差异；服务方决定如何调整，客户动作入口确认属于客户的事实，系统才把批准结果写成
新的当前有效版本。

它拟解决的不是“没记住任务”，而是：**客户已经说了新事实，公司仍按旧事实运行。**

## 1. 当前决定

### 做什么

做一个不替换微信、飞书、钉钉、表格或项目系统的“客户变更处理层”。首个可见对象
不是任务，也不是会议纪要，而是一张有版本、有出处、有影响范围的客户变更提案。

```text
客户变化证据
→ 找回当前有效的项目约定
→ Agent 生成类型化影响差异和未知项
→ 服务方审阅商业决定
→ 客户动作入口确认属于客户的事实
→ 确定性程序生成新的当前有效版本并保留旧版本
→ 任务、等待、验收与付款节点消费已确认的新事实
```

### 不做什么

- 不做通用项目管理、CRM、知识库、电子签或全能 OPC OS。
- 不把聊天摘要、建任务、定时提醒包装成 Agent 创新。
- 不让 Agent 决定价格、替客户确认、自动覆盖当前有效版本。
- 不声称能静默读取个人微信历史。
- 不把普通客户确认页称作合同或电子签。

### 为什么改变旧的 Waiting-for 决定

旧稿选择 Waiting-for，主要依据是创始人痛感、现有代码复用和现场演示可见性，
不是中国需求证据。中国专项研究得到的新排序是：

| 候选问题 | 中国证据 | 免费替代压力 | 决定 |
|---|---:|---:|---|
| 范围、验收、付款边界失效 | 中强 | 中 | 首要问题 |
| 客户变化后业务状态仍旧 | 中弱，核心因果仍待实物验证 | 中 | 最值得验证的 Agent 机制 |
| 多项目优先级冲突 | 弱至中，缺价值/容量输入 | 高 | 后置 |
| 等待事项被忘记 | 弱，且提醒器可替代 | 极高 | 降为下游状态，不做首要功能 |

Waiting-for 没被删除。它应由已经批准的客户变更、验收或付款节点产生，例如
“等待客户补素材”“等待客户确认加价”“等待客户验收”。这样它有业务上下文；
单独拿出来只是一只更复杂的闹钟。首个开发切片不解决“忘记等待”；它只保存由
已确认变化派生出的等待状态。后台时钟、外部回复和自动关闭留到真实入口成立以后。

## 2. 场景价值：问题、用户和发生时刻

### 首轮招募条件，不是已验证用户画像

首轮优先招募中国定制软件/AI 实施独立开发者和不超过 10 人的微型工作室，筛选条件是：

- 按项目或阶段收费，而不是纯自助 SaaS；
- 近 30 天同时推进至少 3 个客户项目；
- 需求会在交付期间变化；
- 客户主要在微信等组织外渠道沟通；
- 范围、日期、验收或付款变化会影响工时或现金流。

“同时推进 3 个项目”“主要在微信沟通”都是招募假设，不是中国总体事实。设计、内容
制作可以作为第二验证细分。咨询和代运营目前没有足够的中国日常工作流证据，不能与
软件开发一起宣称已经成立。

### 精确发生场景

服务方已经有一版有效约定：做什么、多少钱、什么时候交、如何验收。项目执行中，
客户在微信里说“再加一个版本”“先不做这个”“还是周五上线”“价格按之前”或发送
一份新文件。服务方理解了聊天，却没有同步修改报价、承诺、里程碑、任务或验收口径。

之后公司同时存在两套事实：

```text
客户脑中的新事实 ≠ 服务方报价/项目/任务里的旧事实
```

可能后果是免费增项、错误排期、返工、验收争议或尾款延迟。中国研究、平台规则和
法院案例证明需求动态变化以及范围—验收—付款边界是真问题；但还没有中国实物研究
直接量化“状态没有同步”发生多少次、平均损失多少、是否愿意付费。因此当前只能
有条件进入真实工件验证。

### 最强反证

- 很多服务者会在微信里直接说清楚，再顺手改表，新增产品反而增加摩擦。
- 问题可能是合同差、议价弱或客户没钱，不是状态未知。
- 平台内交易已有补充需求、限时验收和证据规则。
- 创意项目过度结构化可能伤害关系语境和创造性。

产品必须证明自己减少了跨对象重建，而不是增加一次录入仪式。

## 3. 产品形态

### 服务方：一个项目事实画布

不是把“会议纪要、任务、项目、客户、知识库”做成五个模块。一个画布只展示同一
项目的因果关系：

```text
原始证据 → 当前有效约定 → 候选变化 → 受影响状态 → 有权决定的人 → 下一动作
```

画布的最小布局：

1. 左侧是当前有效版本：范围、价格、交付日期、验收标准和付款节点。
2. 中间是用户本次选择的微信文本、截图或文件证据。
3. 右侧是 Agent 生成的逐项 before/after、冲突、未知和证据定位。
4. 底部是批准结果：服务方决定、客户决定、写回结果和版本历史。

用户不需要靠脑子回忆公司到了哪一步，因为系统展示的是“已确认事实”和“待判断
候选”的区别；Agent 在新事件到来时维护候选，公司事实仍由有权的人决定。

### 客户：手机轻链接

客户只看到与自己相关的具体变化、原约定、新约定和影响，不看到服务方内部项目、
利润或优先级。客户无需付费；客户可以确认或提出修改，不能只有“同意”。旧版本
链接不能改变当前版本。

比赛 MVP 的 bearer link 只能证明“客户动作专用入口”和版本安全，不能证明点击者
就是客户本人，也不能证明服务方技术上绝对无法代点。界面和路演必须标为“链接已
确认（身份未验证）”。若要把确认升级为客户本人证据，必须绑定客户控制的邮箱/手机
OTP、微信身份或已登录客户账号；这在商业产品中是必做项，不用文案掩盖。

每项变化同时声明最低身份保证级别。比赛的 `guest_link` 只能满足低保证级别的协作
记录，服务方可以选择让它更新本项目状态，但它不能晋级为“客户身份已验证”或法律
证据。涉及真实价格、付款和验收的商业部署，默认要求 OTP、微信身份或客户账号。

### 输入形态

个人微信没有面向第三方的通用聊天历史读取 API。比赛 MVP 采用用户主动选择证据：

1. 电脑微信复制相关文本到 Web；或
2. 手机/电脑上传相关截图或文件；或
3. 用户主动转发到以后提供的小程序/微信客服入口。

这里不是要求用户重填项目。用户只提供一个变化证据，系统负责检索已有当前版本、判断
影响并生成差异。未来官方入口按真实用户渠道选择：微信客服、满足管理员/员工/客户
同意要求的企微存档、飞书或钉钉机器人。不得把这些入口混称为“微信全量接入”。

## 4. 完整产品路径

1. 服务方打开一个已有项目；项目已有一版确认过的范围、价格、日期和验收标准。
2. 客户在微信提出变化；服务方复制一段相关文本或上传截图。
3. 系统保存 `SourceEvent`，记录渠道、时间、声称的发送者和原始证据。
4. Agent 检索该项目当前有效版本，判断这是不是询问、意向、条件同意或明确变化。
5. Agent 生成 `ChangeProposal`：每个影响都有 before、after、证据、冲突、置信度、
   未知项和所需权威；证据不足时弃权或提问。
6. 服务方逐项编辑或拒绝；价格、免费做、延期与资源取舍始终由服务方决定。
7. 涉及双方约定的变化通过客户动作专用链接确认；确认绑定提案 revision 和每项
   `impactHash`。任何发送后的编辑都生成新 revision，并使旧确认失效。
8. 每项变化分别记录 `requiredAuthority`、`minimumActorAssurance` 和决定；只有被
   选中且满足自身权威与身份保证要求的变化可以进入应用集合。客户提出修改会关闭旧
   revision，再由服务方重新审阅。
9. 唯一领域命令以 `baseVersion` compare-and-set 应用批准结果。比赛内存实现只承诺
   单进程、单聚合的 all-or-nothing；系统生成新版本，旧版本只读保留。
10. 首个切片确定性更新两个既有对象：交付里程碑和付款里程碑。任务、Waiting 和
    外部系统写回不得在未实现时宣称已经完成。
11. 只有内部批准、达到所需身份保证级别的客户动作或目标系统回执支持的变化进入事实
    记忆，并保留 assurance level，供下一次 Agent 检索。候选摘要不能直接驱动高风险动作。

## 5. Agent、人和程序的分工

| 环节 | Agent | 人 | 确定性程序 |
|---|---|---|---|
| 理解变化 | 消歧、关联项目与旧版本、发现矛盾、允许弃权 | 纠正语境和发送者判断 | 保存不可变原始证据；机器校验证据位置 |
| 判断影响 | 提议范围/日期/价格/验收/任务差异 | 决定价格、是否接受、是否延期 | 校验 schema 和当前版本号 |
| 对外确认 | 起草简洁说明 | 服务方发送；客户动作入口确认或修改 | 权限、revision-bound 令牌、身份保证级别、版本失效、幂等 |
| 写回状态 | 解释受影响对象和遗漏风险 | 批准高风险更新 | 单聚合 CAS 应用、回执、失败补偿、历史 |
| 长期记忆 | 检索已验证事实，提示冲突 | 撤销或更正错误事实 | 事实等级、来源、有效期、替代关系 |

创新不在“用了模型、工作流、MCP 或多 Agent”。可辩护的机制是：

> 分类型说明哪些状态会变化 + 服务方工作台不能直接写客户状态 + 只有达到所需身份/权威级别或目标系统验证，新事实才会生效。

“带原文证据”也不能只靠模型返回一段 `sourceExcerpt`。文本证据使用
`EvidenceSpan { sourceEventId, start, end, sha256 }`，展示文本由程序从不可变原文切出；
截图证据使用原图哈希和 OCR block/bbox。模型只返回位置和候选解释，服务端校验引用，
引用越界、模型超时、坏 JSON 或空结果都只能形成失败的 analysis attempt，不能形成
可批准变化。fixture 只能由显式演示按钮触发并在界面标明，禁止静默回退。

## 6. 需求到产品动作的决策图

| 用户损失 | 现有替代 | 替代为什么仍可能失败 | 本产品动作 | 衡量 | 失败即停止 |
|---|---|---|---|---|---|
| 新需求没有进入旧报价/范围 | 微信说明 + 手改表 | 要找旧版本并多处重录 | 带证据的范围/报价 diff | 24/48h 旧状态残留数；未收费增项工时/金额 | 实物里只有变化，没有旧状态继续生效 |
| 日期改变但里程碑/任务仍旧 | PM 工具批量改期 | 先要人判断影响对象 | 从已批准变化提出受影响对象 | 对象召回率、误写率、更新耗时 | 漏报高或审批比手改更慢 |
| 双方对验收版本理解不同 | 微信“好”、表单、电子签 | 确认可能脱离原文和项目版本 | 客户动作绑定具体 revision，旧版本失效；身份保证级别单独显示 | 确认完成率、旧版误确认率、争议定位时间 | 客户不愿点且微信回复同样清楚 |
| 等待客户素材/确认被漏掉 | 闹钟、待办 | 没有说明在等什么、影响什么 | 首切片只从已批准状态派生 Waiting-for，不宣称已解决遗忘 | 先记录责任方、期待证据和阻塞对象 | 若要主动发现仍只有人工输入，则后置 |
| 多项目优先级打架 | 看板、GTD、Kanban | 仍缺价值、容量、关系输入 | 本期不自动排序；只展示已确认影响 | 用户决策耗时 | 用户不愿提供决策输入 |

## 7. 如何衡量“提高效率”

### 产品结果指标

主指标：**客户变化发生后，所有受影响状态形成一致、已批准新版本的中位时间。**

同时记录：

- 变化后 24/48 小时仍保留旧事实的对象数；
- 每个真实变化需要人工查找和重录的分钟数；
- 未收费增项的可核对工时或金额；
- 客户确认完成率与确认时长；
- 验收/付款因版本不清而延迟的天数。

### Agent 质量指标

- 分对象影响 precision/recall；
- 高风险遗漏率；
- 人工修改/拒绝率；
- 正确弃权率；
- 未验证事实被错误晋级率；
- 旧版本或重复请求造成的误写率。

不能用“生成了多少任务”“提醒了多少次”“节省 80% 时间”作为没有基线的数据。

## 8. 商业化判断

- 付款人：承担返工、延期、未收费增项和尾款风险的服务方主理人。
- 客户：免费 guest；要求客户付费会直接损害核心闭环。
- 形态：先做不要求全量迁移的上层处理页；写回用户已有表格、飞书、钉钉或 PM。
- 当前价格：未知。
- 第一笔钱实验：`¥299 / 10 个真实客户变化闭环` 的人工增强服务，固定输入、输出
  和数量；不是市场价格结论。
- SaaS 假说：只有出现无补贴预付和连续续购后，才测试 `¥99/月，客户免费`。

最低商业门槛：完成至少 20 个真实变化案例；首批 10 名合格买方至少 3 名预付；
至少连续两期有续购；多数案例复用同一数据结构，并能测出时间或损失差异。

最大竞争者不是飞书，而是“我在微信说清楚，再顺手改一下表”。产品只有减少多处
状态重建，并连接可核对的经营损失，才有独立收费资格。

## 9. 180 秒比赛 Golden Path

演示样例：客户在原定“周五上线、¥8,000、单版本落地页，已付 ¥4,000，验收后付
¥4,000”的项目中说：
“再加一组 A/B 测试，还是周五上，价格先按之前的。”

| 时间 | 现场动作 | 评委必须看见的价值 |
|---:|---|---|
| 0–25 秒 | 从显式 `project-01` fixture 创建一份独立演示项目，展示 v1、交付里程碑和付款里程碑 | 系统不是从空白文本开始；fixture 在界面标明，且不是市场证据 |
| 25–50 秒 | 粘贴客户变化证据 | 入口发生在真实客户沟通之后，只选相关证据 |
| 50–85 秒 | Agent 在 8–12 秒内生成影响差异，随后展示证据定位 | 增加范围；仍要求周五但工期影响未知；“按之前”不自动解释成同价；交付与尾款对象受影响 |
| 85–120 秒 | 服务方提出反方案：`+¥2,000、周一交付、尾款变为 ¥6,000`，逐项批准并发出 | 商业决定属于人，客户动作入口确认的是服务方明确后的新版本 |
| 120–145 秒 | 第二个浏览器/手机入口确认同一 revision | 比赛只证明客户动作入口和版本绑定；未接身份验证时明确标为 guest confirmation |
| 145–180 秒 | `baseVersion` CAS 应用 v2；交付里程碑和付款里程碑各更新一次；v1 只读；旧链接/replay 返回 409 | 同一事件修正两个已存在对象，`stale objects 2 → 0`，不是只发提醒 |

这条路径证明的是产品机制，不证明中国需求频率或支付意愿。它同时服务赛事五项：
场景连接钱和交付；商业上付款人明确且不要求全量迁移；创新机制可见；服务方桌面和
客户手机体验完整；路演只讲一条因果链。结构化影响推理、冲突识别和正确弃权属于
Agent 能力；权限、版本 CAS、幂等和 E2E 是可信执行护栏，不能混称为 AI 创新。

现场验收条件：提前部署并验证 HTTPS 地址和二维码；手机不访问临时 localhost；模型在
路演前预热并设置 12 秒硬超时；超时后只显示明确的失败或由用户主动点选的
“确定性演示 fixture”，不得静默把 fixture 演成实时 Agent。

## 10. 对现有代码的改造地图

当前代码已经跑通：粘贴文本 → 提取承诺 → 服务方审阅 → 客户动作链接确认 → 服务方
交付 → 客户动作链接验收。它证明了 provider API 与 client API 的角色路由分离，
**没有证明点击者身份**。最该保留的是显式状态规则、追加历史、两端交互和确定性测试，
而不是当前“承诺单”文案或 bearer link 安全性。

### 保留

- `shared/delivery/state-machine.ts`：借用显式守卫的写法；新变更流程使用独立状态模块。
- `app/c/[token]/*`：复用客户动作页面的布局与移动端交互，不复用旧 token 语义。
- `shared/delivery/repository.ts`：保留追加历史的思想；当前进程内 `Map` 只用于演示。
- `app/api/efficiency/commitments/route.ts` 与 LLM adapter：复用结构化模型调用边界。
- `shared/delivery/extract-mock.ts`、fixtures、unit/E2E：复用确定性演示和权限测试。
- `shared/delivery/metrics.ts`：保留 cohort 纪律，替换指标。

### 现在不引入的基础设施

- 不引入 LangGraph/Mastra：当前显式状态机与一次结构化调用更小、更可审计。
- 不引入完整 CRM/PM/ERP：它们替换产品，不解决入口事件解释。
- 不为四个本地函数引入 n8n/Activepieces。
- 不为确定性比赛演示引入 Trigger.dev。

真实产品需要可靠后台等待时，优先评估 Apache-2.0 的 Trigger.dev；确认具体输入/写回
系统后再评估 Activepieces 连接器；需要法律签署时调用 e签宝、法大大或 Documenso，
不自建电子签。开源项目先作为领域模型、权限、审计和连接器先例，不盲目搬入大栈。

### 首个切片不解决

- 不解决 forgotten waiting；只保存由已确认变化产生的本地 waiting projection。
- 不写回飞书/钉钉；演示只称“本项目内派生更新”，不显示外部系统成功标识。
- 不证明客户本人身份；guest link 明示身份未验证。若路演要声称客户本人确认，OTP/
  微信身份/客户账号必须进入本切片和 E2E。
- 不证明生产级持久记忆、跨系统事务或自动读取微信。

### 生产化但不属于首个演示切片

- 持久化关系数据库、租户边界、正式 provider session 和客户身份验证；
- 渠道 adapter、写回回执、失败补偿与重试；
- 审计、撤销和数据保留策略；
- 企微/微信客服/飞书/钉钉的官方授权入口；
- 真实后台调度和异常 Agent。

当前进程内 `Map` 重启即丢失，不能对外声称“持续公司记忆”。

原始客户证据也不能进入普通日志或完整暴露在客户页。比赛实现至少记录创建者、用途、
内容哈希和删除入口；客户页只展示该 proposal 必需的证据片段。正式保存周期、脱敏和
撤销规则在真实用户验证后进入生产规格。

## 11. 开发前的唯一证据门槛

在大范围开发前，先对 8–12 名目标用户做真实工件回放。每人必须展示最近 30 天的：

1. 一条真实客户变化；
2. 变化前和 24/48 小时后的报价/范围/里程碑/任务/验收/账单；
3. 可核对的返工、未收费金额、延期或收款影响；
4. 只读影响 diff 与“微信里直接说清楚”的对照。

继续条件：前 8 人至少 5 人提供真实工件；至少 3 个事件出现
`客户事实变化 → 旧状态继续生效 → 可观察后果`；至少 2 个有可核对损失；目标用户
平均每月至少 2 次；影响 diff 比备忘录/提醒多发现了一个真实后果。

如果只有需求变化、没有旧状态继续生效，产品应收缩为变更单工具；如果主要原因是
议价或客户付款能力，停止“状态同步 Agent”；如果客户不愿选择证据，改做企微/飞书/
钉钉原生插件或停止独立产品路线。

## 12. 当前可向组委会陈述的产品假设

| 问题 | 当前答案 |
|---|---|
| 以什么形态 | 服务方 Web 项目事实画布 + 客户手机轻确认页；作为现有系统的上层变更处理页，不替换 PM/CRM |
| 在什么场景 | 我们正在验证：中国项目制独立开发者中，是否有一批同时推进多个项目、主要在系统外接收客户变化，并因旧状态继续生效而受损的人 |
| 通过什么路径 | 选择变化证据 → Agent 对照当前有效约定 → 人审变化前后差异 → 客户动作入口确认同一 revision → 新版本生效并更新本地对象 |
| 拟发挥什么作用 | 把散落的新消息变成有证据、可批准、可传播的一致公司状态；该效果仍待真实工件验证 |
| 解决什么问题 | 待验证问题：客户事实已变但报价/承诺/项目/任务仍旧，并进一步造成免费增项、返工、延期或验收/收款争议 |
| 如何衡量 | 变化到一致状态时间、24/48h 旧状态残留、人工重录时间、未收费工时/金额、Agent 漏报误写 |
| 商业是否可行 | provider-paid、client-free、不要求全量迁移有结构合理性；中国频率和付费未验证，先用 `¥299/10 次闭环` 预付实验 |
