# 项目画布独立调查稿

**日期：** 2026-07-15
**调查基线：** `a34bc9a352cbf86a78af9c598f764e3f47c3a730`
**说明：** 原独立调查会话未在时限内返回。本稿按设计流程的降级规则，由主 Agent 依据三个互相独立的源码调查结果重新起草；因此独立性弱于完整的第二会话。

## 1. 用户真正需要完成的事

用户重新进入项目时，需要在一个界面内完成五件事：

1. 恢复离开时的目标、完成情况和原下一步。
2. 看见离开后新增的决定、代码、材料、阻塞和 Agent 结果。
3. 判断原下一步是继续、调整，还是现有信息不足。
4. 点开任何判断，回到原始事件、材料或执行结果。
5. 自己决定下一步，或把明确工作交给 Agent，并继续看到执行结果。

项目画布不是关系图本身。它是上述过程的共同界面：中央负责选择关注对象，右侧负责解释和行动，底部负责时间变化。

## 2. 四个连续状态

| 状态 | 必须回答 | 界面行为 |
|---|---|---|
| 进入项目 | 上次做什么；离开后发生什么；原计划是否还有效 | 项目居中；最多三个有依据的重点；右侧先显示恢复说明 |
| 聚焦对象 | 这个对象与什么直接相关；为什么现在重要 | 对象重新居中；只展开一层；右侧和时间线同步切换 |
| 作出决定 | 继续什么、调整什么、由谁执行 | 动作前展示下一步、负责人、依据和验收标准；动作后写事件 |
| 执行写回 | Agent 是否开始、阻塞、产出和验证 | 当前执行与历史结果分开；关键事件始终可见；重点重新计算 |

## 3. 必须补充的业务对象

### 3.1 项目归属

当前 `KnowledgeCard` 和 `ActionItem` 没有项目归属，无法支持用户在多个项目之间恢复工作。新增 `Project`，并给卡片和工作项增加 `projectId`。旧数据在读取时进入默认项目，不能因迁移丢失。

### 3.2 用户确认的工作状态

仅靠“最新事件”无法知道用户离开时认定的目标和下一步。需要在 `WorkEvent.meta` 中保存用户确认的状态记录，至少包含：

```ts
type WorkCheckpoint = {
  goal: string;
  completed: string[];
  unresolved: string[];
  nextStep: string;
  confirmedBy: string;
  confirmedAt: string;
};
```

没有用户确认记录时，恢复说明必须标为“根据现有事件整理”，不能表现成确定事实。

### 3.3 原计划判断

原计划判断只允许：

- `continue`：没有已知变化改变原下一步的前提，或用户重新确认继续。
- `adjust`：存在明确阻塞、决定变更、冲突证据或失败结果。
- `insufficient`：出现新材料或结果，但现有信息无法判断影响。

每个判断必须引用事件或卡片。无依据的模型文字不能改变状态。

### 3.4 Agent 当前状态与历史结果

当前执行和历史结果不能用同一个颜色或标签表达。第一阶段只交付一个受控 Agent：读取当前工作项、直接依据和最近关键事件，给出判断、缺口、建议下一步和引用依据。开始、阻塞、结果、验证全部写入 `WorkEvent`。

不创建完整 Agent runtime。未来出现第二个真实 Agent 后，再把负责人拆成结构化 Agent 身份和容量模型。

## 4. 数据投影规则

画布只显示现有事实的投影，不建立第二套事实：

- 项目连接所属卡片和工作项。
- 卡片连接现有确认或建议关系。
- 工作项连接 `evidenceIds` 指向的卡片。
- 工作项连接自己的关键事件。
- Agent 结果是 `actor=agent:*` 的事件。

重新居中只返回一层。默认隐藏普通评论和低价值活动；决定、阻塞、下一步变化、Agent 失败、结果和验证必须单独显示，不能被活动数量汇总吞掉。

## 5. Agent 当前重点

当前重点必须用确定规则计算，模型只能改写理由：

1. 阻塞。
2. 等待确认。
3. 逾期。
4. 离开后改变原计划的决定、下一步变化或 Agent 结果。
5. 长时间没有推进的进行中工作。

同一工作项只显示最重要的一条，总数不超过三条。每条包含目标对象、原因、触发事件或依据。没有异常时显示最接近执行的工作项，并明确“未发现新的阻塞或冲突”。

## 6. API 与代码位置

### 类型与存储

- 修改 `shared/types/knowledge.ts`：Project、projectId、画布引用、当前重点、恢复说明和原计划判断。
- 修改 `shared/knowledge/repository.ts`：项目文件、默认项目迁移、按项目过滤、用户确认的状态记录。
- 新建 `shared/knowledge/project-canvas.ts`：纯函数生成当前重点、恢复说明、一层节点、右侧内容和时间线。

### 接口

- `GET/POST /api/knowledge/projects`
- `GET /api/knowledge/projects/[id]/canvas?focus=<kind>:<id>`
- `POST /api/knowledge/work-items/[id]/agent-run`

同一个 canvas 响应必须同时驱动中央、右侧和底部，避免三个区域读到不同时间的状态。

### 界面

- `app/track/knowledge/page.tsx` 只协调项目、focus 查询参数和数据刷新。
- 左侧是项目与入口；四个开源项目只能标为参考来源。
- 中央使用确定性 SVG 连线和普通按钮节点，不增加图数据库或复杂画布依赖。
- 右侧默认显示恢复说明；聚焦后显示对象说明、依据、影响和动作。
- 底部显示当前 focus 的时间线；当前执行与历史结果分区。
- 当前 focus 写入 URL；返回、前进和刷新后可以恢复。

## 7. 测试

### 单元测试

- 旧数据归入默认项目且不丢失。
- 重点排序符合固定优先级、去重并限制三条。
- 三种原计划判断都有明确证据；无依据时只能是 `insufficient`。
- 每类 focus 只生成一层关系。
- 关键事件不会被普通活动汇总隐藏。

### 浏览器测试

1. 打开项目，看到恢复说明和不超过三个重点。
2. 点击工作项后 URL、中央、右侧、时间线同时改变。
3. 点击依据后能看到来源句或原始事件。
4. 交给 Agent 后看到开始和结果；刷新仍存在。
5. Agent 结果改变当前重点。
6. 项目切换不串数据。
7. 旧检索、来源、工作项状态和卡片关系仍可使用。

## 8. 不做

- 不做自由拖放、无限缩放或完整白板。
- 不把整个项目全部加载到浏览器。
- 不做多个 Agent 的队列、容量、在线状态和权限系统。
- 不复制参考项目的界面代码或品牌资产。
- 不把模型生成文字当成用户已确认事实。
- 不为匹配参考图伪造项目、Agent、活动数量或执行结果。

## 9. 验收

- 用户进入后能直接回答：离开时做什么、后来发生什么、原计划是否有效。
- 每个重点和原计划判断一次点击可到依据。
- 点击对象后只展开一层，中央、右侧、底部保持一致。
- 用户决定和 Agent 执行写回真实事件，刷新不丢。
- 无足够依据时显示信息不足，不替用户批准方向。
- 1440×900 与 1280×800 可完成四个连续状态；键盘可选择节点；减少动态效果生效。
- 单测、Playwright、lint 和生产构建通过，或把仓库原有失败单独列出。

## 10. 与主调查稿可能不同的重点

1. 不能只做“当前重点排序”，还要有用户确认的离开状态和三态原计划判断，否则没有回答最初的问题。
2. 右侧进入项目时先显示恢复说明，不先展示通用项目介绍。
3. 当前 Agent 状态和历史结果必须分开；但第一阶段不需要完整 Agent 身份和容量模型。
4. 参考图的多 Agent 活动不能直接照抄；只有真实事件才显示。
5. 完整小屏画布不是第一阶段目标；小屏按列表、说明、时间线顺序展示。
