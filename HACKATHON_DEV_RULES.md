# 黑客松项目开发规范

> 适用范围：本仓库及同一黑客松工作区内的项目
> 版本：v1.0 | 2026-06-16 确立

---

## 1. 前后端必须对齐

**改完任何一层（后端 / 前端 / 接口）之后，立即做 Feature Inventory 审计。**

不做审计 = 不知道有多少功能是假的。

### Feature Inventory 三步法（每次改完必做）

```
第 1 步 — 真集：列出后端实际提供的
  扫 routes.py / main.py / controllers，提取每个端点的：
  - 路径 + 方法
  - 请求体字段
  - 响应体字段
  - SSE 事件类型（如有）

第 2 步 — 需集：列出前端期望的
  扫 App.tsx / 所有组件，提取每个 fetch/API 调用：
  - URL + 方法
  - 请求体字段
  - 期望的响应字段（实际怎么用的）

第 3 步 — 对比找缺口
  前端调了但后端没有  → 必须修（或加 TODO）
  后端有但前端没调用  → 确认是遗漏还是未完成
  字段名/类型不匹配    → 必须统一
```

### 僵尸 UI 检测（集成前必做）

| 检测项 | 方法 | 通过标准 |
|--------|------|----------|
| 按钮点了有反应 | 手动点一遍所有按钮 | 每个按钮都有实际行为 |
| 页面有数据 | 每个页面都能加载 | 没有空页面/404 |
| API 调用有后端 | grep fetch + 对照路由 | 每个 URL 都对应一个端点 |
| SSE 事件有渲染器 | 对照后端 event types | 每个 type 都有 switch case |
| 模型/配置选项有效 | 检查下拉框选项 | 每个 option 都被后端识别 |

### 接口契约先行

- 任何新端点，先写契约（OpenAPI / 甚至就是一份 Markdown 表格）
- 契约放在项目的 `docs/contract.md` 或 `docs/api.md`
- 改契约 = 改两端代码，不允许只改一端

## 2. 产品问题先于实现

**在做任何功能之前，先回答"为什么"，再回答"怎么做"。**

### 写代码前的 4 个问题

1. **这个功能解决什么具体问题？** （用户说的，不是你猜的）
2. **用户现在怎么解决这个问题？** （竞品 / 手工 / Excel）
3. **如果没有这个功能会发生什么？** （区分 Must vs Could）
4. **Demo 时评委会看到这个功能吗？** （看不到 = 不做）

### MoSCoW 分类（每个项目必须写）

| 类别 | 数量上限 | 说明 |
|------|----------|------|
| Must | ≤ 3 | 没有它产品完全不能工作 |
| Should | ≤ 2 | 显著提升体验 |
| Could | ≤ 2 | 锦上添花 |
| Won't | 不限 | 明确放弃，写在纸上 |

### 红灯停

以下情况**立即停止**，先讨论再决定：

- "技术上很酷但用户价值不明确"
- "反正挺简单的，顺便加上"
- 没有对应具体用户需求的功能
- 需要超过 4 小时且不在 Must/Should 列表里

## 3. 并行开发要有交接

**当后端和前端由不同 Agent 或不同轮次开发时：**

1. 后端 Agent 完成后，输出一份 **API 文档**（包含所有端点、字段、事件类型）
2. 前端 Agent 启动前，**先读这份文档**，对照自己的 fetch 调用做审计
3. 任何一方改了接口，**更新文档并通知另一方**
4. 不允许"口头约定"的接口变更

### Agent 交接检查清单

```
后端 Agent 交付时：
□ 所有端点已实现
□ 每个端点有请求/响应示例
□ SSE 事件类型清单（如有）
□ 错误状态码和错误格式
□ 已更新 docs/api.md

前端 Agent 启动时：
□ 已读取 docs/api.md
□ 已对照后端文档检查所有 fetch 调用
□ 已确认字段名/类型匹配
□ 已删除所有指向不存在端点的调用
□ 已测试核心流程跑通
```

## 4. 先场景与测试，再实现

**参见 `docs/architecture.md` §10 或项目级 `CLAUDE.md` 中的 SDD+TDD mandate。**

简要版：
1. 先写 Given/When/Then 场景
2. 写测试，确认失败（RED）
3. 写最小实现，确认通过（GREEN）
4. 跑完整测试套件（Closed-Loop）

## 附录：快速检查清单

### 每次提交前

```
□ 后端改了什么？对应的前端改动是什么？
□ 新增的端点前端用到了吗？
□ 删除的端点前端还调吗？
□ 新增的 SSE 事件类型前端有渲染器吗？
□ 改了字段名？两端都改了吗？
□ 新增了下拉框选项？后端认吗？
```

### Demo 前

```
□ 每个页面能打开
□ 每个按钮点了有反应
□ 核心流程从头到尾跑通
□ 没有 console 报错
□ 没有僵尸 UI（点了没反应的按钮/链接）
□ 备用视频已录制
```
