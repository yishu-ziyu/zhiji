# Q-04 研究输入：知识范围先隔离，再显式复用

- 输入：G4 当前代码 fail gates、G6 Owner 可见范围场景。
- 状态：供 Owner 决定范围规则；不构成权限/ACL 实现规格。

## 研究结论

当前产品的 `projectId` 已能支撑项目画布、材料目录、当前项目搜索和部分同项目写入约束；但多个 list/search/state API 在没有 `projectId` 时会全局读取，且没有跨项目复用、个人、团队或敏感分类模型。

所以 Q-04 不应问“现在是否已经隔离”，而应先确定：**默认项目隔离，跨范围使用必须显式。**

## 建议带给 Owner 的四条范围规则

| 范围 | 建议规则 | 当前事实 |
|---|---|---|
| 当前项目 | 默认唯一可见范围；所有产品读取、搜索、工作项与 Agent 判断都带当前项目。 | UI 画布/材料/当前项目搜索大多符合；全局 API 仍会泄漏。 |
| 跨项目复用 | 不是默认搜索结果；通过“引用其他项目依据/提升为可复用资料”的显式动作进入，显示来源项目、时间、引用者与权限。 | 没有产品模型；部分写入会禁止跨项目，但全局读取可混入。 |
| 个人长期 | 独立于项目的个人层；项目只能显式引用，不能默认把所有项目资料变成个人记忆。 | 不存在。 |
| 团队共享 | 独立共享空间与成员/角色；未共享项目默认不可见。 | 不存在，当前是本地单操作者存储。 |
| 敏感资料 | 最低限度先有 `normal/sensitive` 分类；敏感项不出现在全局面/跨项目复用/Agent 扩范围中，需拒绝或确认并留痕。 | 不存在；项目文件夹隔开不等于敏感隔离。 |

## Owner 要决定的产品选择

1. 是否确认“项目默认只看自己”作为硬规则？研究建议：**确认**。
2. 跨项目资料是先只允许“手动引用原项目依据”，还是先允许“提升为个人可复用资料”？研究建议：先前者，避免把全局 library 误当共享。
3. 个人长期与团队共享是否进入首版知识范围？研究建议：作为后续显式 scope，不从现有 `projectId`/全局 map 推断出来。
4. 哪些材料在首版必须标为 sensitive，以及对搜索、Agent、外部检索/导出的默认拒绝或确认规则？

## 当前实现不可宣称的事

- “切换项目就一定看不到其他项目”：全局 library、无 projectId 搜索、state/work-item 等路径不满足。
- “已有跨项目复用”：目前只有全局混读或同项目写入拒绝，没有 Owner 可见的 reuse 记录。
- “已有个人/团队/敏感权限”：领域对象没有对应 scope、member、role、sensitivity 或 visibility 字段。

## 对后续产品验收的最小语言

```text
在 scion 内搜索，不会出现 G6体验空项目独有内容。
要使用别的项目资料时，我主动选择来源项目，并看见来源、范围和留痕。
敏感材料不会因全局搜索、Agent 或项目切换而静默露出。
```

Q-02 的不可变来源与“Agent 不得自确认”仍是共同前置：范围正确也不能让无版本、可自我确认的知识成为可信真相。
