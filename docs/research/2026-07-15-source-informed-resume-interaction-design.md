# 基于源码的项目恢复交互研究

**日期：** 2026-07-15
**状态：** 设计研究，尚未批准为 PRD，尚未进入产品代码实现
**目的：** 研究 SignalGraph、mindwalk、Multica、Canvasight 的真实源码，判断哪些方法能服务本项目，避免只看截图拼界面。
**前置文档：** `CONTEXT.md`、`docs/product/001-resume-project-state.md`、`docs/research/2026-07-14-content-structure-task-audit-alignment.md`

> **2026-07-15 设计反馈：** 四个项目的源码分析继续保留；但把“恢复上次工作状态”做成整个产品的首屏，并用固定区域承载说明、依据和执行记录，这个界面方向已被用户否定，不能作为后续实现依据。恢复项目只是一个重要场景，不是产品的全部形态。下一版应回到产品的核心过程，研究以项目为中心、由 Agent 主动提出当前关注点、用户逐层展开材料关系与行动的画布；先出图片确认，再写界面代码。

## 结论

四个项目不能决定本产品长什么样。交互必须先服务这段业务过程：

1. 用户选择要重新进入的项目。
2. Agent 说明用户离开时在做什么。
3. Agent 只列出离开后真正改变项目状态的事情。
4. Agent 判断原来的下一步是仍然成立、需要调整，还是信息不足。
5. 用户需要时查看判断依据，并能回到原始对话、文档、代码或 Agent 结果。
6. 用户确认或修改下一步后，才可以交给人或 Agent 执行。
7. 执行结果、阻塞、验证和新的下一步写回同一条工作记录，供下次恢复。

来源项目分别提供这段过程中的一种方法：

- SignalGraph：关系有方向、有类型、能回到原句。
- Canvasight：一个时刻只保留一个活动对象，画布和右侧检查区保持稳定。
- Multica：任务、负责人、当前状态和历史结果分开表达。
- mindwalk：事件是事实，回放只是派生视图，异常和关键决定不能被普通活动淹没。

它们提供方法，不提供本产品的业务判断。

## 四个项目教会了什么

### SignalGraph：先写清“什么影响了什么”

源码把文本依次处理为解析结果、实体、关系和图，而不是直接把一段总结当成事实：

- `backend/pipeline.py:28-43`
- `backend/graph/relationship.py:9-18,27-34,42-52`
- `backend/graph/graph.py:53-91,93-211`
- `backend/exporters/markdown_exporter.py:19-38`

对本产品的要求：

- “新变化影响了旧计划”必须表达方向和关系，例如“新产品定义反驳旧计划前提”“测试结果支持继续执行”。
- 每条判断贴着显示原句、来源、时间、执行者和确认状态。
- 默认只展示当前事项的一层直接关系；用户继续追查时再展开路径。
- Agent 建议的关系和用户确认的关系必须分开。
- 文字说明优先，关系图只用于解释复杂关系。

不能照搬：

- SignalGraph 没有可复用的前端设计，产品表面是 CLI。
- 英文大写实体和固定关系词不适合中文项目材料。
- 规则匹配后的 `confidence=1.0` 只说明规则命中，不说明事实可信。
- 内存图、弱时间模型和最短边数路径不能作为项目状态来源。

### mindwalk：事件是事实，视觉只是观察方式

源码将 Agent 事件归一化后再投影到时间线和 3D 场景：

- `web/src/playback/reducer.ts:3,66`
- `web/src/ui/Timeline.tsx:13,69,157,179,228,291`
- `web/src/ui/Hud.tsx:83,99,142,170,203,319`
- `web/src/scene/TreeScene.tsx:28,344,444,592`
- `web/src/scene/trail.ts:4,42`
- `.impeccable.md:18,32`

对本产品的要求：

- `WorkEvent` 是事实；恢复说明、变化列表和活动概览都是从事件生成的视图。
- 同时保留“用户离开时的确认点”和“现在的状态”，不能只给一段最终总结。
- 阻塞、失败、决定、下一步变化和验证结果单独显示，不能被活动数量压缩掉。
- 点击事件能回到完整记录和原始材料。
- 颜色只表达一种业务含义；选中状态用轮廓或形状，不覆盖原状态颜色。
- 执行记录超过一定数量后才需要压缩时间带，第一阶段先做可读事件列表。

不能照搬：

- 不做 Three.js 城市、树、自动旋转和视频导出。
- `search / read / edit` 色谱适合代码触达，不适合表达计划是否有效。
- “运行过 verify”不等于“验证通过”。
- 最近若干步的轨迹只能辅助方向感，不能替代完整记录。

### Multica：当前状态和历史结果必须分开

源码先定义稳定对象和状态，再由界面回答“谁在做什么”：

- `docs/design.md:23-41,88-155,159-290`
- `packages/ui/styles/tokens.css:62-99,135-194`
- `packages/views/issues/components/board-card.tsx:68-93,100-287,329-350`
- `packages/views/issues/components/board-column.tsx:35-36,88-176`
- `packages/views/agents/components/tabs/activity-tab.tsx:58-69,87-185,278-426,578-690`
- `packages/core/agents/types.ts:6-24,38-70`
- `packages/core/agents/derive-presence.ts:26-121`

对本产品的要求：

- 默认先显示现在正在发生的事；已完成和已取消内容进入历史。
- 当前任务只需标题、状态、负责人、下一步、最后变化时间和阻塞原因。
- 任务详情替换列表主体或进入稳定检查区，不再堆在列表下方。
- 新建工作项按需展开，不在默认页常驻完整表单。
- Agent 的“能否接活”“当前负载”“过去结果”是三个不同维度。
- “交给 Agent”之前检查下一步、负责人、验收标准和依据；不能只修改一个字符串负责人。
- 界面分组优先使用间距和分隔线，再考虑背景和卡片。

视觉规则：

- 90% 使用中性色，语义色只表达进行中、风险、阻塞或完成。
- 同屏控制在两到三种语义色。
- 工作界面正文以 14px 为主、元数据 12px、区块标题 16px。
- Hover 只轻微改变背景或文字，约 150ms；不缩放、不加装饰阴影。
- 选中状态必须比 hover 多一个可识别维度。

不能照搬：

- 不复制完整 Runtime、Squad、并发 Agent 和拖拽看板。
- 不套用 Multica 的状态枚举，本项目还有“待确认”和计划有效性判断。
- 修改版 Apache 许可对第三方托管或嵌入式商业服务有限制；未经单独确认，不复制其代码、Logo 或资产。

### Canvasight：对象能继续操作，关系不是装饰线

设计规范和真实组件都强调稳定工作区、一个活动对象和上下文操作：

- `design.md:13-32,34-61,156-199,201-221`
- `plugins/canvasight/src/components/TaskNode.tsx:521-725`
- `plugins/canvasight/src/components/ScatterEdge.tsx:34-75`
- `plugins/canvasight/src/App.tsx:940-975,1700-1735,2076-2187,2514-2810`
- `plugins/canvasight/src/styles/app.css:1-113,1518-1587,2044-2069,2080-2301,2303-2725`

对本产品的要求：

- 画布、项目导航和右侧检查区尺寸稳定，展开详情不让用户失去当前位置。
- 一个时刻只有一个活动工作项或一条活动关系。
- 节点是可编辑、可连接、可执行的工作对象，不是只供观看的泡泡。
- 选择节点时再显示相关操作；普通状态保持安静。
- 关系被选中或与当前对象有关时才提高视觉强度。
- Agent 生成的对象使用和人工对象相同的数据、交互和编辑方式。
- 内容顺序不是天然依赖；只有真实的支持、矛盾、依赖、来源或执行关系才画边。

本次源码审计纠正了一个错误：Canvasight 的真实设计规范使用中性表面、有限强调色和克制边框，并明确反对装饰背景。此前把“暖纸、胶带和手绘痕迹”说成 Canvasight 的源码方法是不准确的。暖纸可以来自本项目自己的视觉系统，但不能借 Canvasight 为其背书。

不能照搬：

- 不复制完整画布编辑器、Graph Writer、Skill 选择和并发页面协议。
- 不把所有项目内容都画成图。
- 不让自由画布先于项目恢复说明出现；用户首先需要答案，不是先学习地图。

## 尚未批准的交互推演

以下内容保留为一次推演记录，只能说明“恢复项目”这个场景需要哪些信息，不能决定整个产品的界面结构。

### 1. 状态说明

用户选择项目后，首屏固定回答：

1. 离开时的目标和原下一步是什么。
2. 离开后有哪些重要变化。
3. 原下一步是仍然成立、需要调整，还是信息不足。
4. Agent 为什么这样判断。
5. 现在需要用户做什么决定。

只列会改变目标、范围、阻塞、结论或下一步的事件。普通读写活动不占首屏。

### 2. 查看依据

用户点击某条变化或判断后：

- 中央显示当前工作项的一层关系。
- 右侧显示选中关系的原句、来源、时间、执行者和确认状态。
- 建议关系可以确认或否决。
- 用户可以继续展开两层或三层解释路径，但全项目图不是默认入口。

### 3. 确认下一步

状态说明只提供与当前判断直接相关的动作：

- 按原下一步继续。
- 修改下一步。
- 暂时无法判断，并记录缺失信息。
- 下一步和验收标准确认后，交给人或 Agent。

若 Agent 判断“需要调整”或“信息不足”，“交给 Agent”默认不可用，并说明缺少什么。

### 4. 执行记录

执行后展示：

- 当前负责人或 Agent。
- 当前状态和阻塞原因。
- checkpoint 之后的事件列表。
- Agent 的开始、阻塞、结果、验证和下一步变化。
- 活动概览只辅助浏览，完整事件始终保留。

执行结果写回后，下一次重新进入项目继续使用同一条工作记录生成新的状态说明。

## 页面结构

```text
固定项目列表
  └─ 选择要恢复的项目

稳定工作区
  ├─ 状态说明（默认）
  │   ├─ 离开时
  │   ├─ 离开后
  │   ├─ 原计划判断
  │   └─ 用户决定
  ├─ 查看依据（按需）
  │   ├─ 当前事项的一层关系
  │   └─ 右侧原文与来源
  └─ 执行记录（按需）
      ├─ checkpoint 后事件
      └─ 当前工作对象与 Agent 状态
```

项目列表在这里不是普通导航。用户的问题就是同时推进多个项目后不知道各自进展到哪里，因此项目范围是 Agent 恢复记忆的必要输入。

## 现有实现能复用什么

当前代码已经有：

- `KnowledgeCard`：内容、来源、时间和链接。
- `ActionItem / WorkItem`：状态、负责人、下一步、验收标准和依据。
- `WorkEvent`：决定、状态变化、阻塞、结果、依据关联等事件。
- `KnowledgeRelation`：方向、类型、原句、确认状态、来源和创建者。
- 邻居、路径、工作项时间线、关系确认和否决。

主要位置：

- `shared/types/knowledge.ts:1-71,159-289`
- `shared/knowledge/repository.ts:49-121,149-208,476-615`
- `app/track/knowledge/components/CardRelationsPanel.tsx:177-250`
- `app/track/knowledge/components/WorkItemsPanel.tsx:141-505`

因此不需要再建一套 SignalGraph 图模型或 Multica 看板。需要的是重组现有对象，让它们服务“恢复、判断、追证、决定、执行”。

## 实现前必须补的能力

### 1. 项目范围

当前 `KnowledgeCard`、`ActionItem`、`WorkEvent` 和 `KnowledgeRelation` 都没有 `projectId`；存储和查询也是全局文件、全局列表。现有代码无法可靠回答“这个项目离开后发生了什么”。

至少需要：

- 一个明确的 `Project` 或 `Workspace` 对象。
- 卡片、工作项、事件和关系都归属一个项目。
- 所有列表、搜索、关系、路径和恢复查询都按项目过滤。
- Agent 写回时必须带项目身份，不能依赖当前页面猜测。

### 2. 用户确认点

目前没有“用户离开时已确认状态”的对象。仅按最后更新时间推断会把 Agent 尚未核对的结果当成用户状态。

建议设计 `ProjectCheckpoint`，至少包含：

- `projectId`
- 当时目标
- 已完成和未解决事项
- 原下一步
- 相关工作项和事件
- 确认者、确认时间
- 原始依据

没有确认点时，恢复说明必须写“Agent 根据现有记录推断”。

### 3. 计划判断

建议只允许三种结果：

- `still_valid`：仍然成立。
- `needs_change`：需要调整，并有明确变化或冲突依据。
- `insufficient_evidence`：信息不足，禁止默认批准继续执行。

每次判断必须引用事件或依据；用户确认后写入 `decision` 事件。

### 4. 结构化负责人和 Agent 运行

当前 `assignee` 和 `actor` 都是字符串，无法稳定区分人、Codex、Grok 或其他 Agent，也无法表达可用性、当前负载和运行结果。

实现前需要决定：

- `assigneeType + assigneeId` 是否进入工作项。
- Agent 运行是否作为独立对象，还是通过带 `runId` 的 `WorkEvent.meta` 表达。
- Agent 完成后必须写回产物、验证结果、状态和新的下一步。

## 视觉规则

视觉服从业务顺序：

- 状态说明可以使用本项目的暖纸表面，使“Agent 给出的当前说明”与普通工具区分开；只用一块，不加胶带等装饰。
- 其余区域使用中性暗色表面，依靠间距和分隔线组织，不把每行都做成卡片。
- 红色表示计划失效、阻塞或当前需要决定；蓝色表示可检查依据；绿色表示已完成或已验证；琥珀表示信息不足或等待。
- 同屏不超过三种主要语义色。
- 选择用边框和背景共同表示，不用大面积品牌色。
- 只有新事件到达、判断改变和状态推进需要动效；150–280ms，支持 reduced motion。
- 搜索是持续可用的辅助入口，不占据首屏主要位置。

## 不做

- 不做 3D 代码城市。
- 不做完整 Agent 管理平台和 Squad。
- 不做全项目默认知识图。
- 不让用户先整理一份完整项目管理表格，才得到恢复说明。
- 不把 Agent 生成的总结当成用户已确认事实。
- 不把颜色、亮度或动画当成证据。
- 不在新的下一步和验收标准未确认时自动交给 Agent。

## 验收标准

1. 用户进入项目后的首屏，在十秒内能回答：离开时在做什么、后来发生什么、原计划是否还成立。
2. 每条计划判断能一次点击到至少一条原始依据。
3. 没有足够依据时明确显示“信息不足”，不能默认判为“仍然成立”。
4. 用户能在同一工作项中确认或修改下一步，并看到将交给谁、如何验收。
5. Agent 的开始、阻塞、结果和验证状态都写回事件并可回看。
6. 切换项目后，任何材料、关系、工作项或事件都不会串到另一个项目。
7. 状态说明、依据和执行记录共享同一组事实对象，不产生三套互相矛盾的数据。
