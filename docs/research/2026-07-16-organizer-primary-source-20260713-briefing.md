## 1. 主办方要求（摘录 · 效率赛道相关）

### 1.1 定位

- 不是只看代码量/模型调用次数。
- 要把 **Agent 放进具体业务场景**，帮 **真实角色** 完成 **具体任务**。
- 工具/路径/细分场景 **开放**；方向下 **把问题做小做深**。
- 工具是 **手段**；核心让用户 **少走一步、少花时间、少犯错误**，提升业务效率与产出。
- 用模型 API / 开源 / 第三方模板插件时提交须 **披露**。

### 1.2 效率赛道（公司 / 一人公司）示例切入

面向：创业者、小团队、项目负责人、商务/运营、知识工作者等。

示例（主办方原意）：

| 方向 | 期望行为 |
|------|----------|
| 项目管理 | 拆阶段/任务、识别延期风险、每日提醒、**更新项目状态**；协作 **不只停在聊天记录** |
| 客户跟进 | 基于客户背景、历史沟通、**现有产品资料** 生成跟进话术/邮件草稿/需求摘要，并给出 **下一步行动** |

重点：说明 **帮谁**、省什么时间/沟通成本、提升什么交付质量；不必颠覆全公司流程。

### 1.3 交付与评分（评委相关）

- Demo / 可访问链接或录屏；**≤5 页** 路演 PPT；MCP 或技术路径说明；**AI 使用披露**。
- 不要大而全：真实、具体、业务流跑通。
- 现场评委约 90%（场景价值 25 · 商业化潜力 25 · 创新 15 · 体验完成度 15 · 路演协作 等）；AI 技术评分约 10%。
- 优先：**真实场景/清晰目标用户** · **商业潜力** · **创新** · **可跑/易懂 demo**。

## 2. Lead 综合（开放 · 非 FINAL）

> 下列为 Lead 综合，供 Owner 拍板（**OA-19**）。**不是** D-* 已定结论。

### 2.1 一致性 · 连续性（路演叙事候选）

| 词 | 工作定义（Lead · 开放） |
|----|-------------------------|
| **一致性** | 不同工具 / 人 / 行动，与 **同一份当前项目状态** 对齐，不各说各话 |
| **连续性** | 今天的工作能 **因果接上** 昨天；历史完整可核对，不靠聊天记忆硬扛 |

### 2.2 内部构件 vs 对外 pitch

- **检索 / 知识 / 工作流** = 产品内部构件（实现层）。
- **对外 pitch** 不卖构件名词；卖场景结果：状态一致、工作可续、决策有依据。

### 2.3 主用户故事（**D-44 FINAL · OA-19 已确认**）

**角色:** 项目负责人 / 一人公司操盘手  
**场景:** 数日/数周后回到项目；变化发生在 **已授权** 文件/代码/工具上  
**Agent 产出（有精确来源）:**

1. **重建当前状态**  
2. **解释为何变化**  
3. **标出过期理解 / 受影响行动**  
4. **协助下一步决定并连回精确依据**

对齐 D-39/D-40/D-41/D-42（实现侧）；pitch 不堆内部构件名词。

### 2.4 痛点 / 方案措辞（推荐 · **OA-20 开放**）

| 项 | 推荐文案（未批准） |
|----|-------------------|
| 痛点 | 项目每变一次，文件变了，但人的理解、任务和沟通没有一起变；用户只能靠记忆重新拼出现在怎样、为什么、下一步是什么。 |
| 方案 | Agent 持续观察授权项目来源，保留版本与变化历史，重建当前状态，指出过期理解和受影响行动，并把下一步判断连回精确依据。 |

### 2.5 效率指标候选

- **return-to-decision time**：从重新进入项目到做出有依据的下一步决定所需时间。  
- **测量前不宣称数字提升。**
