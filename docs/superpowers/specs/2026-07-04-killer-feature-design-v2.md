| 项 | 数据 |
|---|---|
| 黑客松时间 | 2026-07-11 ~ 07-12 杭州 · 48h |
| 今日 | 2026-07-04 · 剩 7 天准备 |
| 项目已有 | Next.js 16 + shadcn/ui + step-3.7-flash(Anthropic 兼容代理) + 5 API + 3 prompt + 11/11 E2E |
| 现有 4 个 Must | 电商(选品分析/短视频脚本/私域话术) + 效率(会议纪要/项目看板) |

| 评分结构 | **90% 评委人工评分**(场景价值、商业化潜力、创新、体验、路演与协作)+ **10% AI 技术评分**(连通性、稳定性、代码相关表现) |

### 1.2 现状分析(v2 改写 · 修复 v1 自爆式定性)

**v1 自爆式定性是错的**。我们重新表述:

> 现有 4 个功能(选品分析 / 短视频脚本 / 私域话术 / 会议纪要)+ 1 个看板,**每个都是独立的 Workflow**(各自有 Intent Router + Prompt Template + 输出 schema),**但它们之间没有共享状态**——选品分析的输出不能直接喂给脚本生成,会议纪要的待办需要手动拖到 Kanban。
>
> **杀手锏要做的是:加一个统一的 Agent 编排层**,让 4 个 Workflow 的状态可以**跨任务流转**——选品分析识别出 SKU-12 是潜力款,小掌柜自动把它从"观察库"挪到"主推库" + 预生成 3 条脚本草稿 + 在 Kanban 自动建"今晚发布"任务。**这才从独立 Workflow 升维成 Agent 编排层**。

> **金句**:"**我们之前是 4 个独立的 Workflow,现在有了 1 个 Agent 把它们串成 Agent。**"

### 1.3 用户原始诉求

> "加一个杀手锏功能,需要充分调研和讨论。"

**核心需求解读**:在 7 天窗口内,通过 1 个新能力让 demo **从独立 Workflow 升维到统一 Agent 编排**,不破坏已有投入,且**必须与现有 DEMO_SCRIPT.md 合并成单一 180s 走查**。

## 二、调研方法

| 阶段 | 投入 | 输出 |
|---|---|---|
| 阶段 1 | 派 3 个 Agent 并行调研不同立场 | 3 份候选杀手锏方案 |
| 阶段 2 | 派 3 个 Agent 互相质询 | 暴露每个方案的真实弱点 |
| 阶段 3 | 综合诊断 → 组合方案 | 选定最终方向 |
| 阶段 4 | 派 3 个 PM Reviewer 对 spec v1 跑审查 | spec-review.md · 3 P1 / 7 P2 / 8 P3 |

### 阶段 1:三个候选方案(各 Agent 立场)

| | 实战派黑客松老兵 | OPC 叙事派 PM | Agent 形态派架构师 |
|---|---|---|---|
| 方案 | Agent Loop Live | AI 团队作战室 | 小掌柜日报 |
| 核心画面 | Thought→Tool→Obs 卡片流 | 大楼 + 3 Agent 工位 | 早 9 点推送 + 自动决策 |
| 主攻评分 | 创新性 20% | 创新性 + 商业潜力 | 商业潜力 + 应用价值 |

### 阶段 2:三个 Agent 互相质询(关键攻击)

**对方案 A(Agent Loop Live)的攻击**:
1. 评审看不出"过程透明"的价值,只 tech 评审 buy in
2. 金句空白,无一句话能复述
3. LLM tool_call 调试是真正无底洞,2.5 天估算是低估

**对方案 B(AI 团队作战室)的攻击**:
1. 复杂度失控(估 6 人天,实际 12+),3-Agent 状态机风险叠加
2. 是 Flowith Matrix(2026-07-02)的简化版,易被认成复刻
3. **砍掉项目看板是砍错了**(三个 Agent 都同意)

**对方案 C(小掌柜日报)的攻击**:
1. 没有"哇时刻",日报 + 自动回复 + 跳转听 30 秒完
2. **"AI 运营总监"是过度承诺**(三个 Agent 都同意),没真决策权
3. 复用已有功能在 2026 黑客松是减分项

### 阶段 3:综合诊断

**三个 Agent 的共识信号**:
1. 真的 Agent 必须能做决策(不只给建议)
2. 简单功能复用在 2026 黑客松是减分项,但**跨功能状态编排**(把 4 个独立 workflow 升级为 1 个 Agent 编排层)是加分项 ← v2 修复
3. 48h 内不能花 > 6 人天

**为什么选组合方案 C + A**:
- C 的商业叙事最完整 + 风险最低(完成度 25% 稳)
- 但 C 的核心弱点是"无真决策" → 补 A 的"可追溯 trace"+ **加 1 个真决策动作**
- 不选 B:虽然画面震撼,但复杂度失控 + 砍看板不对 + 形式>内涵

## 三、最终方案设计

### 3.1 方案名(v2 修订)

**「小掌柜」AI 运营编排层 + Agent 实时追溯**(v1 的"AI 运营总监"措辞已降级,避免《广告法》风险)

副标题:侧边栏常驻 + 跨 Workflow 状态流转 + 真决策动作 + 完整可审计 trace

### 3.2 核心金句(v2 重构 · 修复撞车)

> **主金句**:「**一个 AI 团队,你的运营助理叫小掌柜,99 元/月**」
> (把 v1 的"AI 运营总监"嵌入 narrative-framework 已有的"虚拟创始团队"叙事,既不撞车又保留拟人化)

> **备选金句**:"**你的 AI 不是回答问题,你的 AI 是开会——它在会上做决策,你在会上签字**"
> (回应 v1 的"金句借方案 C"的论证漏洞,这是杀手锏自己的金句)

### 3.3 三个核心组件

#### 组件 1:侧边栏常驻「小掌柜」AI 头像

**形态**:任何页面右下角都常驻一个 👨‍💼 emoji 头像 + 三色状态灯(绿:空闲 / 黄:工作中 / 红:需决策)
**v2 改进**:**用 2 次后可折叠成右下角极小图标**,不打扰工作流(Notion AI 现行做法,挡 P2-7 媒体评委追问)

**进入页面时第一眼看到**(不是 push notification,避免 Day 30 留存 < 5% 风险):
> "老板,昨晚 23:40 用户 @林小姐 私信问 XX 蓝牙耳机续航,已用话术库 #EP-07 自动回复;今早淘宝热搜新词『骨传导耳机 2026 款』匹配你的选品库 SKU-12,建议今日主推;昨天上线 3 个 SKU 中,SKU-09 加购率 2.1% 远高于平均 0.8%,建议今晚追加投放 50 元。"

#### 组件 2:Agent Trace Panel(实时追溯)

**触发**:小掌柜执行任何决策时,自动展开
**形态**:右侧浮层,纵向时间轴

**v2 重大改进 · 折叠式设计(挡 P2-6 攻击)**:
- **默认折叠**:每张卡片只显示 3-5 句关键决策点 + 类型 badge
- **点击展开**:每张 Thought 卡片可展开全文 CoT
- **悬停问号**:每个 Thought 卡片右侧「为什么这样想」问号,鼠标悬停弹 1 句话理由
- (类似 GitHub Copilot Workspace + Cursor 的现行做法,评审熟悉)

**卡片类型**:
- 💭 **Thought**:小掌柜在思考什么(默认折叠为 1 句话 + 类型 badge)
- 🔧 **Tool Call**:调用了什么工具(JSON 参数 + 工具名)
- 📊 **Observation**:工具返回了什么(数据/状态)
- ✅ **Decision**:做出了什么决策 + 为什么(可审计)

**核心金句**:
> "Agent 不是黑盒,我每一步怎么想、怎么干、为什么这个结论,你都看得到。"

#### 组件 3:真决策动作(v2 重大修订 · 挡 P2-5 攻击)

**v1 的"自动加入看板"被 Reviewer 一致反对**,理由:它是 UI 自动化,跟 Notion AI 没差别,**不是商业决策**。

**v2 推荐:自动调整选品库权重 + 一键确认**(对应 narrative-framework "空军地面引导员"隐喻——"该打的时候打、该撤的时候撤")

**完整动作链**(用户点"确认"一次完成 4 个动作):
1. 选品分析出 SKU-12 是潜力款 → 小掌柜**自动**把它从「观察库」挪到「主推库」
2. 同时**预生成 3 条**短视频脚本草稿(基于 SKU-12 卖点)
3. **同时**在 Kanban 自动建「今晚 23:59 发布 SKU-12 视频」任务
4. **同时**给运营同事发一条 mock 通知(电商 message 通道)

**商业后果**(为什么这是真决策而不是 UI 自动化):
- 主推库影响明日算法权重(模拟的算法,数据可见)
- 4 个动作是一次"空军打击"——选了目标 → 准备了弹药 → 排了任务 → 通知了队友
- 评审追问"这跟 Notion AI 自动建任务有什么区别"——答:**Notion 不会动选品库权重,这是有商业后果的 Agent 决策**

**确认按钮 + 撤回机制**:
- 小掌柜**自动执行**(不需要用户先点确认) ← 这才是真 Agent
- 但提供 5 秒「撤回」窗口
- 5 秒后不可撤回(因为已影响主推库权重)

**v2 合并方案**:保留 DEMO_SCRIPT.md 第一幕开场 30s + 第四幕收尾 15s,中间 90s 替换为新叙事,效率 Track 从 45s 压到 15s。

### 3.5 与现有架构的兼容性

**完全兼容(不需要重写)**:
