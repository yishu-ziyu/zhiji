## 评分项拆解

### 评委人工评分(90%)

人工评分看:

1. **场景价值**
   - 是否解决真实用户/真实业务场景
   - 是否能在 3 分钟 Demo 中让评委一眼看懂"谁在什么场景下为什么需要它"

2. **商业化潜力**
   - 是否有清晰用户、付费理由、价格锚点
   - 是否能讲清 99 元/月相比人工成本的 ROI
   - 是否有后续生态合作或规模化可能

3. **创新**
   - 是否从普通聊天框升级为 Agent 行为
   - 是否有可被记住的差异化瞬间
   - 是否不是简单 ChatGPT wrapper

4. **体验**
   - Demo 是否流畅
   - UI 是否清楚、稳定、易理解
   - 评委能否在 3 分钟内看懂主流程

5. **路演与协作**
   - 讲述节奏、金句、时间控制
   - 团队协作/分工展示
   - 是否有清晰的现场兜底方案

### AI 技术评分(10%)

AI 技术评分看:

1. **连通性**
   - API/模型/页面能否连通
   - Demo 时核心链路能否调用成功

2. **稳定性**
   - LLM 慢/失败时是否有 fallback
   - 页面是否无 crash、无 console error
   - 核心流程是否能重复跑通

3. **代码相关表现**
   - 代码结构是否清楚
   - API 契约是否前后端一致
   - 是否有测试/构建/回归验证

### 创客共鸣奖

基于选手互评总得分形成。

这意味着除了评委视角,还要准备一个面向其他参赛者的"一句话传播点":

> **一个 AI 团队,你的运营助理叫小掌柜,99 元/月。**

## 对当前项目优先级的影响

### 旧假设需要下调

旧假设是:

```text
完成度 25% / 应用价值 25% / 商业潜力 20% / 创新性 20% / 安全性 10%
```

新评分实际是:

```text
人工主观评分 90% + AI 技术评分 10%
```

因此,后续优先级必须调整:

| 原先优先级 | 新优先级 |
|---|---|
| 技术深度 / 真 tool_use / adapter 复杂度 | **降级为 10% 技术评分的一部分** |
| 3 分钟 Demo 叙事、视觉、体验、商业故事 | **升为 90% 主战场** |
| P1-B step-3.7-flash tool_use spike | 仍要做,但不再决定生死 |
| P1-A DEMO_SCRIPT 180s 合并 | **绝对最高优先级** |
| 小掌柜 / Trace Panel 是否好懂、是否像 Agent | **最高优先级** |

## 新的工程判断

### 1. 不要为了真 tool_use 牺牲 Demo 稳定性

如果 step-3.7-flash 不支持 tool_use,不应硬切 DeepSeek 或重写 adapter 导致工程风险上升。

**推荐策略**:
- Demo 走 deterministic mock + LLM 文本生成
- Trace Panel 命名为"思考过程可视化"
- 把技术解释留到 Q&A,不在主路演里讲

### 2. D2-D7 的主线从"技术真 Agent"调整为"评委看懂的 Agent"

关键不是让技术评审在 DevTools 里看到 `tool_use` block,而是让 90% 人工评分看见:

```text
小掌柜主动早报 → 采纳建议 → 4 个动作一次完成 → 看板落地 → 99 元/月 ROI
```

### 3. 创客共鸣奖需要准备传播点

同场选手互评不会细看代码。

他们会记住的是:

- 小掌柜
- 99 元/月
- 一人公司
- 4 个动作一次完成
- 不是聊天框,是运营助理

## 结论

**本项目现在应该按"90% 人工评分产品 Demo"来打,不是按"10% AI 技术评分工程深度"来打。**

这不代表代码可以烂,而是说技术必须服务 Demo 稳定和评委理解。

最优策略:

1. 主线:小掌柜 Agent 编排层的可见体验
2. 兜底:mock / replay / 录像保证现场稳定
3. 技术:最小必要实现 + 可解释 + 有测试
4. 路演:3 分钟金句、ROI、创客共鸣
