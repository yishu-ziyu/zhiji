# 开发协作：全流程步骤与术语

**用途：** 你和 Agent 共用同一套说法与顺序。  
**范围：** 本仓库 `fc-opc-ibot` 开发（黑客松效率 Agent）。  
**原则：** 平常中文；不要废话；不要黑话；步骤内化，不必每次念给对方听。

---

## 1. 角色

| 谁 | 负责 | 不负责 |
|----|------|--------|
| **你（用户）** | 定目标、定取舍、拍板、真机手感、对外提交 | 被要求填无限表格才开工 |
| **Agent** | 澄清验收、实现、对照、简短说明、跑测试 | 替你发明产品目标；偷偷改已定边界 |
| **工具链** | 编译、测试、Git、浏览器 | 替双方做产品决策 |

一句话：  
**你定「做什么、怎样算完」；Agent 做「做完并证明」。**

---

## 2. 五步协作（默认全流程骨架）

任何功能、修 bug、写文档，默认都走这五步。  
不必每次报步骤名；但缺步时要说清楚卡在哪一步。

| 步 | 名称 | 谁主做 | 产出 | 怎样算这一步完 | 禁止 |
|----|------|--------|------|----------------|------|
| 1 | **意图** | 你 | 目标、约束、成功样子（可粗） | Agent 能复述且你点头或默许 | Agent 替你编目标 |
| 2 | **规格** | Agent | 要做什么、怎样算做完、明确不做；关键歧义最多问一轮 | 有可勾选的验收条目 | 无限追问；未对齐就大改架构 |
| 3 | **实现** | Agent | 最小可运行改动 | 代码/文档按规格落地 | 半途把已知项再问一遍 |
| 4 | **对照** | Agent | 逐条：做到 / 未做 / 偏差 | 对照表或等价清单 | 只说「做完了」 |
| 5 | **说明** | Agent | 做了什么、路径、风险、下一步 | 短；无自我辩护长文 | 废话、套话、工序腔 |

**和英文规则文件的对应：**  
`AGENTS.md` 里的 intent → clarify → implement → compare → explain  
= 上表 1→5。同一件事，不要发明第三套名字。

## 3. 术语表（只准用这些说法）

### 3.1 协作与文档

| 我们说 | 意思 | 不要说成 |
|--------|------|----------|
| **意图** | 你想要的结果与约束 | 愿景诗、空口号 |
| **规格 / Spec** | 可验收的约定：做/不做/怎样算完 | 金脚本、场景钉死 |
| **验收 / 验收条目** | 可勾 true/false 的检查 | 感觉更好 |
| **不做 / 非目标** | 明确排除 | 「以后再说」含糊带过当已定 |
| **对照** | 拿结果逐条比规格 | 「基本完成」 |
| **演示步骤** | 固定几步怎么点、给评委看 | 金路径、黄金路径、金脚本 |
| **交件** | 交给主办方/评委的材料与链接 | 「闭环交付」当黑话 |
| **手测** | 人在浏览器里点 | 只靠单测绿 |
| **e2e** | 浏览器/API 自动走用户可见路径 | 与单测混称 |

### 3.2 产品对象（与当前 Spec 对齐）

| 我们说 | 意思 | 不要说成 |
|--------|------|----------|
| **工作项** | 一件要推进完的事（状态+负责人+下一步+时间线） | 随便一个 todo 气氛 |
| **依据 / 卡** | 带来源的可引用结论 | 没出处的 AI 总结当事实 |
| **来源** | 会议/邮件/聊天/文档/手记 | 「好像在哪看过」 |
| **事件 / 时间线** | 写回工作项的历史 | 只在聊天里说过 |
| **下一步** | 当前一句要做的动作 | 长篇描述冒充下一步 |
| **负责人** | 人的名字或 agent id | 没有归属的「我们」 |
| **状态** | 字典里的枚举 | 每人自定义一套词 |
| **知识工作台** | 本产品页面总称 `/track/knowledge` | 知识经、知识闭环（产品名） |
| **知识足迹** | 依据卡在库图上的触达点亮（检索/挂接等） | mindwalk 代码地图；无校验的特效 |

完整字段与验收见：  
`docs/SPEC-work-item-shared-situation.md`（工作项）  
`docs/SPEC-knowledge-footprint.md`（知识足迹；先规格后实现）

### 3.3 工程产物位置

| 我们说 | 路径或位置 | 干什么用 |
|--------|------------|----------|
| **CONTEXT** | `CONTEXT.md` | 本轮产品说法、禁词、演示怎么说 |
| **产品说明 / PRD** | `docs/PRD-*.md`、`.ship/.../product/08-prd.md` | 给人或评委的产品说明 |
| **Spec** | `docs/SPEC-*.md`、`.ship/.../plan/spec.md` | 实现与验收的技术/产品规格 |
| **交件包** | `docs/demo/SUBMIT_PACK.md` | 使用说明、演示、路演页纲、链接位 |
| **任务目录** | `.ship/tasks/<id>/` | 单次任务的 plan/e2e/control（若用 yishuship） |
| **数据** | `data/knowledge/` | 本地持久化（一般不进 git） |
| **单测** | `shared/**/*.test.ts`、`npm run test:unit` | 领域逻辑 |
| **e2e** | `tests/e2e/`、`npm run test:e2e` | 用户可见路径 |

### 3.4 可选：yishuship 阶段名（仅在真跑该工具链时用）

与五步对应，**不强制念出来**：

| yishuship 说法 | 我们五步里 | 白话 |
|----------------|------------|------|
| pm-intake / 对齐 | 意图 + 部分规格 | 做不做、做多大 |
| design | 规格 | 写成 spec/plan |
| dev | 实现 | 写代码 |
| e2e | 对照的一部分 | 自动验收用户路径 |
| review | 对照 | 静态看 diff |
| qa | 对照 | 手测/探查 |
| handoff | 说明 + 交件 | 提交、PR、推送 |

未启动 yishuship 时，只说五步即可。

## 4. 端到端开发流程（从想法到可交件）

```text
意图（你）
  → 规格（Agent：验收 + 不做）
    → 实现（Agent：小步代码/文档）
      → 对照（单测 / e2e / 手测 / 对照表）
        → 说明（路径、风险、下一步）
          → 需要时：提交 Git、推送、更新交件包
```

### 4.1 每一步的「入口话」与「出口物」

| 阶段 | 你可能说 | Agent 应交出 |
|------|----------|--------------|
| 意图 | 「我要…」「先别做…」 | 复述目标与边界（短） |
| 规格 | 「写好 spec」「对齐验收」 | Spec/验收列表；未决问题最多一轮 |
| 实现 | 「做」「实现 P0」 | diff；可运行 |
| 对照 | 「测一下」「对一下」 | 条目勾选 + 命令结果 |
| 说明 | （默认交付附带） | 路径、风险、你可点什么 |
| 发布 | 「提交并推送」 | commit / push；给链接 |
| 交件 | 「交件包」 | URL/录屏位、演示步骤、说明 |

### 4.2 什么时候必须停下来问你

- 改 **产品边界**（做/不做变了）  
- **删除** 用户已有重要文件/数据  
- **推送/强制推送/改远程** 若你未在本轮授权（你明确说过则可）  
- **密钥、登录态、第三方账号**  
- 规格里 **两个实现方向成本差一个数量级** 且不可逆  

其余默认 Agent 做完再对照说明。

### 4.3 什么叫「这一项开发完了」

同时满足：

1. 规格里的验收条目已对照  
2. 约定的测试已跑（或明示跳过原因）  
3. 你知道：改了什么、怎么打开看、还有什么风险

## 5. 和「效率 / 共享情境」工作的关系

讨论产品时用同一套词，避免各说各话：

| 层 | 术语 | 本仓库现状（2026-07-14） |
|----|------|---------------------------|
| 依据 | 卡、来源、检索 | 已有 |
| 行动 / 局面 | 工作项、状态、负责人、下一步、时间线 | Spec 已写，实现按 P0 推进 |
| 执行 | agent 写回事件 | Spec 预留 API；真 CLI 非 P0 |

开发功能时：  
先说清动的是 **哪一层**，再改代码。

## 6. 沟通纪律（开发对话专用）

1. **先结论**：做成了没有 / 卡在哪一步。  
2. **要路径**：文件、URL、命令。  
3. **对照规格**：不要只报「重构了」「优化了」。  
4. **禁止黑话：** 金路径、知识经、收一刀、压成、场景钉死、闭环当产品名等（见 `CONTEXT.md` / `AGENTS.md`）。  
5. **禁止废话：** 见 `~/.grok/AGENTS.md` → No fluff。  
6. **纠错优先记规则：** 你说「不对」→ 改行为并写入约定，不只道歉。
