## 一、五分钟 Pitch 结构（5 张幻灯片）

### Slide 1: 问题 - 一人公司正在爆发，但工具不给力

**标题**: 一个人就是一支团队 - 但你的工具箱还停留在 2019

**核心话术**:
- 中国一人公司（OPC）从 2023 年开始爆发式增长
- 2026-05-11 人民日报报道杭州火光会 OPC 社区，一人公司已成为被官方认证的社会现象
- 这些创业者一个人要干完：选品、拍视频、写文案、开周会、管进度 - 没有一个人能用的完整工具链
- 现有工具都是给团队设计的：Shopify 给电商团队、Notion 给产品团队、剪映给内容团队 - 一个一人创业者要拼凑 5-6 个工具

**一句话钩子**:
> "当一个人要干五个人的活，你需要的是五个助手，而不是五个软件。"

**视觉建议**: 左侧是一个人在电脑前被工具淹没的剪影，右侧是这个人旁边站着两个 AI Agent

---

### Slide 2: 方案 - FC-OPC iBot，一人公司的双轨 AI 助手

**标题**: 一个平台，两条赛道，覆盖电商全链路和日常运营

**核心话术**:
- **电商轨道**：从选品分析到短视频脚本，一键生成可执行的电商内容
- **效率轨道**：会议纪要自动沉淀为待办，看板实时追踪执行进度
- 两个轨道共享一个底座，同一个账号、同一个 LLM、同一套工作流

**差异化点**:
- 不是给"需要 AI 的公司"做，是给"只有一个人但要做完整业务"的人做
- 不做"万能助手"的泛化叙事，聚焦电商 + 效率两条已验证的高频场景
- 首次实现了"分析 -> 创作 -> 管理 -> 追踪"的完整流程

**视觉建议**: 左右分栏，电商轨道展示"选品 -> 脚本"流程，效率轨道展示"会议 -> 看板"流程，中间用同一底座连接

---

### Slide 3: Demo - 真实场景跑通

**电商轨道 Demo 脚本**（约 90 秒）:
```
1. 用户在输入框粘贴一个商品标题（如 "无线蓝牙耳机 2026新款"）
2. 点击"分析"
3. iBot 返回：卖点拆解（3个核心卖点）、目标人群画像、差异化建议、价格建议区间
4. 用户点击"生成脚本"
5. iBot 返回：15秒短视频分镜脚本（含画面描述、口播文案、字幕、BGM建议）
6. 整个过程用户只做了一次输入
```

**效率轨道 Demo 脚本**（约 90 秒）:
```
1. 用户粘贴一段会议录音转写的文字
2. 点击"纪要"
3. iBot 返回：结构化纪要（参会人/时间/议题/决策/待办）+ 每个待办自动分配负责人和截止日期
4. 切换到看板视图，待办以卡片形式出现在 Kanban 上
5. 用户拖拽卡片更新状态（待办 -> 进行中 -> 已完成）
6. 整个过程自动完成，无需手动录入
```

**关键**: Demo 要用真实可操作的界面，不要动画模拟。评委看的是"这个产品能不能真的工作"，不是"这个 PPT 好不好看"。

### Slide 4: 技术 - 不是调个 API 就完事了

**标题**: 从意图识别到多步工作流，Agent 真正在工作

**核心话术**:

**架构层面**:
- Intent Router：用户输入自动判断走电商轨道还是效率轨道
- Workflow Registry：每条轨道有独立的工作流编排，不是单次 Prompt
- Streaming：生成过程实时展示，不是等 30 秒然后丢一段结果

**技术选型**:
- Next.js 16 + TypeScript：现代 Web 框架，RSC 性能优化
- shadcn/ui：生产级组件库，不是手搓 UI
- step-3.7-flash：国产模型驱动，支持工具调用和推理
- Tailwind CSS：设计系统一致

**强调**:
> "我们没有把 AI 当聊天框装饰品。每一步 Agent 调用都有明确的工具链、状态流转和用户可见的产出。"

### Slide 5: 愿景 - 一人公司的 AI OS

**标题**: 从双轨工具到一人公司的 AI 操作系统

**核心话术**:
- 短期（3个月）：电商轨道扩展到选品数据对接、效率轨道集成日历/邮件
- 中期（6个月）：加入财务 Agent（自动记账+报表）、客服 Agent（自动回复+售后）
- 长期：一人公司只需要一个 FC-OPC iBot，覆盖从获客到交付到财务的全链路
- 一人公司不是过渡态，是一种可持续的工作方式 - FC-OPC iBot 是这个群体的基础设施

**收尾金句**:
> "未来的工作方式不是一个团队 + 一个 AI，而是一个人的能力 × 10 个 AI 助手。FC-OPC iBot 让这个未来今天就发生。"

## 二、Before/After 转化故事

### 电商轨道 - 产品分析 + 视频脚本

**Before - 传统流程**:
```
卖家小陈早上 9 点打开电脑：
1. 打开淘宝/抖音搜索竞品（30分钟）
2. 打开 Excel 记录 5 个竞品的价格、卖点、评价关键词（45分钟）
3. 打开浏览器查小红书/知乎看用户真实评价（30分钟）
4. 打开 Word 写差异化分析（20分钟）
5. 打开剪映，对着空白的脚本框发呆（15分钟）
6. 终于开始写脚本 - "家人们今天给大家推荐..."（40分钟）

总计：3小时，产出1条短视频脚本
小陈的第 3 条视频拍完，已经下午 2 点了
```

**After - 用 FC-OPC iBot**:
```
卖家小陈早上 9 点打开电脑：
1. 在 iBot 输入 "无线蓝牙耳机 2026新款"
2. iBot 30 秒返回：3 个核心卖点 + 差异化建议 + 目标人群画像
3. 点击"生成脚本"，15 秒返回：分镜脚本 + 口播文案 + 字幕 + BGM 建议
4. 复制脚本到剪映，开始拍摄

总计：1分钟分析 + 15秒脚本 = 2分钟，产出1条短视频脚本
小陈 9:05 就完成了分析，当天可以批量产出 10 条
```

**对比维度**:

| 维度 | Before | After |
|------|--------|-------|
| 时间 | 3小时 | 2分钟 |
| 工具数 | 4个 | 1个 |
| 思维切换 | 5次 | 0次 |
| 数据来源 | 凭感觉记 | 系统分析 |
| 可复制性 | 不能 | 同输入同质量 |
| 精力消耗 | 高强度 | 轻量决策 |

总计：1分钟整理，待办执行率预估 80%+

## 三、"一人军团"主题的正确打开方式

### 为什么这个主题有效

**时机正确**:
- 2026 年人民日报报道杭州 OPC 社区 = 一人公司从边缘现象变成社会趋势
- AI 工具的成熟让"一个人干一支团队"从修辞变成真实可能性
- 评委中大概率有人关注过"AI + 个体经济"这个话题

**情感正确**:
- 一人创业者是最孤独的打工人 - 没有团队可以商量，没有上下班界限，什么事都要自己扛
- "AI 助手"的叙事很容易变成"AI 替代人"的威胁感，但对一人创业者来说，AI 是"终于有人搭把手了"的解放感
- 这个主题天然带有"赋能小人物"的叙事张力

### 讲故事的正确姿势

**不要说**:
- "AI 将彻底改变工作方式"（太大、太模糊、评委听腻了）
- "一人公司是未来趋势"（这只是描述，不是洞察）
- "我们的产品能帮你提高效率"（每个 AI 产品都这么说）

**要说**:
- "一个人做电商，从选品到拍视频要花 3 小时，现在 2 分钟"（具体、可量化、有冲击力）
- "中国有超过 X 万一人公司，但他们用的工具是给团队设计的"（把问题锚定在真实人群）
- "我们不给你一个聊天框，我们给你一个能帮你把事做完的搭档"（区分于"聊天机器人"的定位）

### 必须锚定的三个真实感来源

1. **人民日报 2026-05-11 报道**: "杭州火光会 OPC 社区" - 一人公司不是虚构概念，有官方背书
2. **具体的用户画像**: "27 岁的张薇，在义乌做跨境电商品，一个人管店铺、拍视频、回客服" - 让评委能想象出具体的使用者
3. **可验证的数据**: "抖音电商个体户占比 XX%"，"一人公司年均工作时长 XX 小时" - 数据支撑痛点存在

## 五、AI  Hackathon  Pitch 常见失败模式（避坑清单）

### 1. "ChatGPT Wrapper" 陷阱

**症状**: 产品就是调了 ChatGPT API，前面套了个 UI

**为什么失败**: 评委一眼就能看出来。shadcn/ui + fetch(api.openai.com) = 不是 Agent

**FC-OPC iBot 的防御策略**:
- 强调 Intent Router：不是用户直接跟 LLM 对话，是系统先判断意图再路由
- 强调 Workflow Registry：不是一次 Prompt，是多步有状态的编排
- 强调工具调用：电商分析用了搜索+结构化输出，脚本生成用了模板引擎
- 如果被问"和 ChatGPT 有什么区别"，直接说："ChatGPT 给你答案，iBot 给你执行计划"

### 2. "功能堆砌" 陷阱

**症状**: PPT 列了 20 个功能，但每个都是半成品

**为什么失败**: "Must/Should/Could" 没分清，评委看到满屏的功能但每个都点不动

**FC-OPC iBot 的防御策略**:
- 只讲两个 Must 功能：电商分析 + 视频脚本、会议纪要 + 看板
- Should 功能留到愿景 Slide，一句话带过
- Could 功能直接不说，主动砍 = 专业度
- Demo 只跑一条完整链路，不要试图展示所有功能

### 3. "技术自嗨" 陷阱

**症状**: 花 2 分钟讲 LangGraph StateGraph 的节点设计、向量数据库的 chunk 策略

**为什么失败**: 多数评委（尤其是投资人/媒体背景）听不懂，而且不在乎

**FC-OPC iBot 的防御策略**:
- 技术 Slide 只讲 3 个信息：用了什么框架、为什么选它、带来了什么用户价值
- "step-3.7-flash" 一句话带过，"流式输出" 演示给看，不必解释底层原理
- 如果有人追问技术细节，准备好 30 秒的简洁回答

### 4. "没有痛点" 陷阱

**症状**: "这是一个 AI 助手，可以做很多事情" - 但没有讲清楚解决什么具体问题

**为什么失败**: 评委评分表第一个维度就是"痛点评估"

**FC-OPC iBot 的防御策略**:
- 开场直接给数字：一人创业者平均花 3 小时做电商内容，FC-OPC iBot 压缩到 2 分钟
- 用具体人物画像锚定痛点：不要说"很多用户"，说"义乌做跨境电品的 27 岁个体户"
- Before/After 对比是最强的痛点表达方式

### 5. "空愿景" 陷阱

**症状**: "我们要做中国的 OpenAI / 我们要改变世界 / 我们要赋能十亿用户"

**为什么失败**: 黑客松评委听了 20 个项目，这种开场白已经免疫了

**FC-OPC iBot 的正确愿景**:
- "我们要做一人电商创业者的标配工具"
- "让一个人在电商赛道上的启动时间从 3 个月压缩到 3 天"
- "从双轨到四轨到全轨 - 一人公司不需要换工具就能扩展业务"
- 数字具体、时间具体、用户具体 = 有可信度的愿景

### 6. "Demo 翻车" 陷阱

**症状**: Demo 时 API 超时、页面报错、数据加载不出来

**为什么失败**: "交付能力评估" 直接打骨折

**FC-OPC iBot 的防御策略**:
- 准备两套数据：在线真实调用 + 本地 mock 数据
- 如果在线调用失败，立刻切换到 mock："为了演示流畅，我先用预设数据展示完整流程，实际效果在线下也可以验证"
- 提前录制一个 90 秒的 Demo 视频作为备份
- Demo 前 30 分钟跑一遍完整流程，确认没有 console error

## 六、FC-OPC iBot 叙事自检清单

在正式 Pitch 前，逐项确认：

**内容层面**:
- [ ] 开场 30 秒内说清楚"解决了什么具体问题"
- [ ] Before/After 对比有具体数字（时间、工具数、效果）
- [ ] 每个功能都有对应的用户场景，不是"因为 LLM 能做"
- [ ] 提到了"一人公司"这个真实社会趋势，不是自说自话

**Demo 层面**:
- [ ] 每条轨道至少有一条完整链路跑通（从输入到产出）
- [ ] 数据是真实的，不是硬编码的假数据
- [ ] 页面没有 console 报错
- [ ] 准备好备用方案（视频录屏 / mock 数据）

**技术层面**:
- [ ] 能说清楚 Intent Router 做了什么（不只是"判断意图"）
- [ ] 能说清楚 Workflow Registry 和单次 API 调用的区别
- [ ] 知道 step-3.7-flash 在项目中具体做了什么（意图分类 + 内容生成？还是只有内容生成？）

**差异化层面**:
- [ ] 能一句话回答"和 ChatGPT 的区别是什么"
- [ ] 能一句话回答"为什么一人创业者需要这个，而不是直接用剪映/Notion"
- [ ] 能说出至少 3 个现有产品没有覆盖到的点

**评委适配层面**:
- [ ] 有传播点（张彩琴类评委能写进报道的金句）
- [ ] 有技术深度（蒙美铮类评委能看到工程思维）
- [ ] 有社会意义（所有人能看懂这个产品为什么值得存在）
