# 非线性工作台：Owner 可见的三种交互状态（骨架）

- 状态：供 Owner 讨论的行为选项；已补入 G3 FRONTEND-R4 与 G3B FRONTEND-R2 的固定源码证据，以及 T-16 的当前产品事实。
- 不是：画布架构、Agent 宣传页、自由白板、或实现规格。
- 已吸收：2026-07-14「内容→结构→任务→可审」、当前项目画布合同、FRONTEND-R1 的证据/候选/复查界面研究。

## 这不是一条线性流程

用户不应被迫按“先聊天、再建卡、再建任务、再看历史”的顺序工作。一个项目可以从对话、材料、关系、任务或结果任一处进入；每次只把**当前对象与一层直接关系**放到中心。用户随时能点另一个对象换中心，也随时能回到项目。

共同底线：画布不是全库总览。它只回答“我现在看哪个对象、它为何重要、它和什么直接相关、下一步可做什么”。

---

## 状态一：对话或材料发散成候选节点

### Owner 做什么

在聊天中提出想法、上传/打开一份材料，或在项目里选中一段内容；例如：“这个方向和昨天的结论冲突吗？”

### Owner 看见什么

1. 项目仍是背景，中央出现一张**候选节点**，不是已经写死的知识卡。
2. 节点用一句普通话说明它是什么：`想法`、`材料摘录`、`待核对判断` 或 `待处理问题`。
3. 节点旁只给两类轻操作：
   - `查看依据`：打开原材料位置、聊天片段或外部 locator；
   - `接到项目`：选择“作为依据”“提出关系”“生成工作项草稿”。
4. 如果 Agent 参与整理，节点明确显示“Agent 候选”，附依据锚点；Owner 没有确认前不显示为决定、已完成或项目当前事实。
5. 不重要的发散内容不抢占画布中心；它们留在候选抽屉/会话中，只有被选用时才进入项目结构。

### 取舍

| 选择 | 得到 | 放弃/约束 |
|---|---|---|
| 对话自动沉淀为**候选**，不是自动入库 | 减少聊天丢失与手工抄写 | 不能承诺“所有聊天都已成为知识” |
| 一个候选先显示一条来源锚点 | 用户立即知道它从哪来 | 不在小节点塞完整正文/全部引用 |
| “接到项目”需要明确动作 | 内容→结构→任务的转变可审 | 多一步点击，但避免垃圾卡与 Agent 自确认 |

### 不应出现

- 一句聊天总结自动生成“已确认结论”。
- 材料一打开就把所有段落铺成节点网。
- Agent 候选与 Owner 已确认状态只靠颜色猜测。

---

## 状态二：点对象换中心，只看一层

### Owner 做什么

点击候选、材料、关系、工作项或结果中的任意一个对象。

### Owner 看见什么

1. 被点对象平滑成为中央焦点；项目名和返回项目入口保持可见。
2. 画布仅显示它的一层直接关系：
   - 材料支持/反驳的判断；
   - 判断关联的工作项；
   - 工作项引用的依据与最近结果；
   - 结果触发的待复查对象。
3. 每条边用普通话说清方向和影响，例如“此材料支持该判断”“此结果改变此工作项的下一步”；边上或右侧能打开证据句。
4. 右侧稳定检查区显示三组信息而不混在画布上：来源/版本、当前状态、可做动作。
5. 点击相邻对象即可继续换中心；返回时恢复上一个焦点，而不是重新渲染全项目。

### 取舍

| 选择 | 得到 | 放弃/约束 |
|---|---|---|
| 只展示一层关系 | 用户能理解“为什么相关”，不必读图 | 深层影响要按需继续点开，不一次画完 |
| 中心对象 + 稳定右侧检查区 | 保留位置感，证据不淹没画布 | 不追求无限自由拖拽/白板式编辑 |
| 边必须有类型、方向、证据句 | 关系可审、可确认/否决 | 不能用共享关键词或时间相近自动画线 |

### 不应出现

- 一打开项目就绘制所有文件、事件、任务与关系。
- “相关”边没有方向、证据或状态。
- 点击对象后只换详情面板、不改变当前理解中心。

## 状态三：行动结果回流并留下足迹

### Owner 做什么

从一个工作项、已确认决定或候选判断发起/接收行动结果：Agent 复核、人工完成、或以后受控外部工具返回结果。

### Owner 看见什么

1. 原工作项保留其依据、责任和行动前状态；结果作为一张**结果记录**回到同一项目。
2. 结果卡清楚回答：做了什么、得到什么、使用了哪些依据、是否需要复查；外部结果另显示对象链接/执行标识。
3. 结果不会自动把项目结论改成“已确认”。若它支持、推翻或改变旧判断，中央画布显示一个“待复查/候选更新”连接，等待 Owner 处理。
4. 项目“现在怎样”只提取最影响当前计划的结果；完整经过留在时间线/足迹中，可从结果点回任务、依据和相关变化。
5. 再次进入项目时，Owner 能看见“上次确认后发生了什么”，并选择采纳、否决、继续行动或回到依据。

### 取舍

| 选择 | 得到 | 放弃/约束 |
|---|---|---|
| 结果先回流为证据/候选更新 | 保留人类终判与历史 | 不追求“一跑完就自动结案” |
| 当前局面与完整足迹分层 | 第一屏安静，仍可审计全过程 | 用户需点开历史看细节 |
| 结果影响旧判断时显式待复查 | 版本变化不被静默覆盖 | 需要管理少量复查项，不能只显示绿勾 |

### 不应出现

- Agent 或外部执行成功后自动确认知识、关闭工作项或覆盖旧结论。
- 结果只在内部日志/聊天里，工作台没有可点回的记录。
- 来源变化后把旧判断静默换成新文本，用户失去当时依据。

## 三态如何连在一起

```text
对话/材料的候选
  └─ 点“接到项目” → 对象成为当前中心的一层关系或工作项草稿

对象换中心
  └─ 点“行动” → 工作项携带依据进入执行

行动结果
  └─ 作为结果记录回流 → 候选更新/待复查 → Owner 决定是否改变当前项目判断
```

用户可以从任何一格重新进入，不要求按箭头走完；箭头只说明信息何时变得可审、可行动、可回看。

## 与已有界面研究的对齐

- FRONTEND-R1：判断旁应有可点来源锚点；Agent 候选、Owner 确认、工作状态必须分开；来源变化显示需复查。
- 2026-07-14 三层：状态一承接“内容→结构”，状态二让结构成为可理解对象，状态三实现“结构→任务→可审”。
- 当前画布合同：项目先居中；点击对象后才展开一层直接关系；完整时间线不占据第一屏；关系有方向、普通话说明与依据。

## 用这些事实约束三态

1. **状态一不是“聊天自动入库”。** 现有 Agent result 已能生成项目内候选卡，但聊天消息与材料段落仍没有同等的候选入口；任何扩展都应先让 Owner 看见来源锚点与候选身份。
2. **状态二不需要自由白板。** 当前一层焦点已经是可复用的产品合同；借 Canvasight 的焦点强调即可，不能把任务树父子边误作证据关系。
3. **状态三已经有“结果不自确认”的领域基础，仍缺可见的 Owner 决策面。** 结果候选、工作项状态、Owner 确认的知识应继续分开；时间线只展示足迹，不能成为唯一结果入口。

以上外部源码均已固定 revision 和许可证；在单独获得实施授权、完成依赖/许可证与退出审查前，所有参照只可称为模式，不可称为已采用。
