## 2. 北极星与本轮范围

### 2.1 北极星（产品一句话）

重新进入项目时，本地 Agent 在明确授权范围内恢复项目态势，用可核对的依据说明现在发生了什么、为什么重要，并帮助用户完成下一步决定。

形态参考：左多项目 · 中关系画布 · 右摘要/Agent · 底时间线。唯一入口：`/track/knowledge`。

### 2.2 本轮做什么

| 优先级 | 做什么 | 落点（路径须真实） |
|---|---|---|
| **第一刀** | 右栏「问 Agent」结构输出 + 纸感 UI + 后端默认结构化回复 | `AgentChatPanel.tsx` · `agent-chat-format.ts` · `agent-dialogue-reply.ts` · `agent-runtime.ts` · `agent-tools.ts` |
| **能力** | Owner 提问走 `owner_question` 工具环；快捷检索意图；follow-read 预算；fail-closed | 同上 + `page.tsx` 发送链路 |
| **守住** | L0 五条红线；双记忆边界；无密钥入包 | 见 §4 · §7 · §8 |

2026-07-17 对话中：**右栏 UI + 后端结构化/能力加强已落地代码**，产品清单 §1/§1b/§2b 等仍待 Owner 亲手点验打勾。

### 2.3 明确不做什么

1. **中栏画布布局暂缓**：不照搬参考图重排中栏；参考图无底部时间轴，不得冲掉时间轴。
2. **不冲掉 timeline**：`WorkbenchShell` 的 `timeline` 槽位是硬槽位；删槽 / 藏底栏当默认 = 越界。
3. **无第二入口 / 无第二产品面**：禁止新建 `app/track/knowledge/mvp/` 或平行 URL；旧 `/mvp` 只重定向。
4. **不把通用 bench 当分发门槛**：SWE-bench / 通用 GAIA 总分不作为知几上线门槛。
5. **对话不得改写已接受项目事实**：候选判断默认「未自动写入项目事实」。
6. **不做编辑器、不做全机监控、不接微信私聊**（CONTEXT 已定）。

## 3. 开发范式（日常怎么开工）

### 3.1 开工前必读（按顺序，短读）

1. `docs/product/产品清单.md` — 未完成 / 北极星 / Agent 红线（唯一产品问题台账）
2. 本文 — 范围、门禁、验收句式
3. 若改 Agent 真读/过程/确认：`docs/product/AGENT_PRESENCE_ACCEPTANCE.md`
4. 词表与双记忆：`CONTEXT.md`（前百行足够开日常任务）
5. 写代码前再读目标文件 + 直接调用方（见 §6 落点表）

产品问题只记产品清单，不另起平行 issue 文件。

### 3.2 任务切片规则

| 刀序 | 区域 | 何时开干 |
|---|---|---|
| **第一刀** | **右栏 Agent** | 默认优先；结构卡、过程可见、入口常驻、结构化回复 |
| **第二刀** | **左栏导航** | 须产品/Owner 点名或清单立项；可借鉴参考图密度与 IA |
| **第三刀** | **中栏画布** | **必须 Owner 明示**；且须保留底时间轴共存方案，禁止整页照搬参考图 |

切片原则：

- 一次只降一类理解成本；加噪声不发（`AGENTS.md` 产品规则 5）。
- 小 diff；改产品逻辑就补行为单测。
- 可拆并行时：**互斥文件所有权**，主会话只做集成与验收。

### 3.3 并行窗 / 平级协作

- 全平级，无 Leader 链；需要时由人点名一人去派活一层。
- **勿抢文件清单**：同一文件同一时间只有一个写入方。
- 可写并行 → 分文件或分 worktree；禁止两窗同改 `AgentChatPanel.tsx` / `agent-runtime.ts` 等热文件。
- Roster 只用项目约定窗口身份；未经 Captain 链核验的外部前缀身份丢弃（见 `AGENTS.md` Lessons）。
- 状态只能在观察到文件 / 哈希 / 命令证据后跃迁；禁止把「已授权要做」写成「已完成」。

### 3.4 分支与代码树

- 分支：**只用 `main`**。
- 主树：仓库根下 `app/` · `shared/` · `tests/` · `docs/`。
- `data/` 慎写，勿把 e2e/冒烟测试标题塞进用户默认中心视图。

## 4. Agent 产品契约（硬）

### 4.1 产品类型

**领域内、多步工具 + 强制 grounding + HITL 的项目情报 Agent。**  
不是通用助手，不是编程 Agent，不是开放 Web Agent。

### 4.2 输出结构（默认每条 Agent 回复）

后端 `formatAgentDialogueReply` 写出、前端 `parseAgentMessage` 解析，字段对齐：

```text
（可选 lead 一句）

当前判断
<一句有证据倾向的判断；无证据则诚实 gap>

依据
<path> @ <rev>          # 可多行；芯片展示 path@rev
…

你现在只要决定
<一个拍板问题>

候选判断 · 未自动写入项目事实
```

契约细则：

| 字段 | 规则 |
|---|---|
| **当前判断** | 优先用 supported 的 why；否则 now；无材料不编路径 |
| **依据** | `path @ rev`（rev 可短写 `r…`）；无 pin 时写限制/gap，不伪造文件 |
| **你现在只要决定** | **一个**决策问题；不是任务列表 |
| **候选注脚** | 固定语义：未自动写入项目事实；UI 可显示页脚 |
| **模型侧** | now 直接答 Owner 问题；why 最多约 3 条 supported 且须 pin；无证据诚实 gap |

别名兼容：`你现在只需决定` 解析侧等同「你现在只要决定」。

### 4.3 L0 五条红线（任一 FAIL 不可当上线/演示过线）

与产品清单 §1、`AGENT_PRESENCE_ACCEPTANCE.md` 对齐：

| # | 红线 | 可见失败 |
|---|---|---|
| 1 | **没读真实文件就下结论** | 无 map/search/read 真实动作就出「读懂」 |
| 2 | **有结论但点不开出处** | 依据空、路径假、rev 对不上授权夹 |
| 3 | **界面无真实工具过程** | 只见最终一段话，过程不随收据动 |
| 4 | **模型挂了仍显示读懂** | 静默降级 / 假候选；正确：`status=failed`，不写假理解 |
| 5 | **确认后重进丢失** | 接受/修改/拒绝后刷新或换项目再进丢状态 |

L1/L2 评测阈值与自建 Bench 属于后续能力门槛（见 §10），**不替代** L0。

### 4.4 fail-closed · HITL · 双记忆

| 主题 | 规则 |
|---|---|
| **fail-closed** | 无可用出处或模型失败 → 失败态；**不** `saveCandidate` 假读懂。演示/验收禁止开 `AGENT_ALLOW_DETERMINISTIC_FALLBACK=1`（仅本地 deterministic 自测可） |
| **HITL** | 候选须 Owner 确认/改/拒绝才进入项目真源；建议不自动建 `todo` |
| **项目记忆** | `shared/project-memory`：事件 + 不可变理解 + CAS；长期真源 |
| **对话记忆** | `shared/agent-memory`：会话回合 + 未完成工具意图；**不能**改写已接受理解 |
| **语义检索** | 可作附属，不作记忆本体 |

### 4.5 Owner 提问主链路（勿另造）

```text
Owner 输入（右栏）
  → page onAgentChatSend
  → runAnalysis(ownerUtterance, trigger=owner_question)
  → agent-runtime 工具环 + model loop
  → UnderstandingBody
  → formatAgentDialogueReply → dialogue append agent 消息
  → 右栏 parseAgentMessage → 结构卡
```

## 5. UI 范式（右栏）

### 5.1 结构区块（验收视觉）

1. **头**：`问 Agent` · 状态（在线 / 待授权等）· 副文案「用自然语言问项目态势，不闲聊」
2. **你**：右对齐浅米气泡
3. **知几**：左对齐 + **结构卡**（当前判断 / 依据芯片 / 你现在只要决定高亮条）
4. **页脚注**：候选判断 · 未自动写入项目事实
5. **快捷 pill**：只看决策 / 冲突在哪 / 重进后变化
6. **输入区**：大框 + 深墨「发送」+ Enter 提示

气质：纸感白、细线、低阴影、深墨发送；反大泡泡 / 电蓝玻璃。样式类前缀 `.agentChat*`，在 `project-canvas.module.css`。

### 5.2 产品行为（与清单 §2b 对齐）

- 任意已打开项目：**问 Agent 常驻**（不依赖有 session 才挂载整块）。
- 无本地夹：输入仍在 + 待授权提示；发送引导先授权。
- 有夹：发送即 `ownerUtterance` → 工具环。
- 顶栏 AI Copilot → **聚焦右栏输入**，不是跳随机材料。
- 底栏文案区分：**Agent 动态**（时间线筛选）≠ 对话。

## 6. 后端 / Runtime 范式

### 6.1 关键落点（已核实路径）

| 职责 | 路径 |
|---|---|
| 右栏 UI | `app/track/knowledge/components/AgentChatPanel.tsx` |
| 结构解析 | `app/track/knowledge/lib/agent-chat-format.ts` |
| 对话格式化 | `shared/project-memory/agent-dialogue-reply.ts` → `formatAgentDialogueReply` |
| Runtime | `shared/project-memory/agent-runtime.ts` |
| 工具 / 检索意图 | `shared/project-memory/agent-tools.ts` → `extractOwnerSearchQueries` |
| 类型 trigger | `shared/project-memory/types.ts` → `owner_question` \| `source_change` \| `retry` |
| 壳与时间轴槽 | `app/track/knowledge/workbench/WorkbenchShell.tsx` |
| 页面编排发送 | `app/track/knowledge/page.tsx` |
| 样式 | `app/track/knowledge/project-canvas.module.css`（`.agentChat*`） |
| 单测 | `tests/unit/agent-chat-format.test.ts` · `tests/unit/agent-dialogue-reply.test.ts` · `shared/project-memory/agent-*.test.ts` |

### 6.2 硬规则

1. **对话写入必须走 `formatAgentDialogueReply(body, { ownerUtterance? })`**，禁止手写仅「now + 下一步」的旧字符串绕过结构。
2. **Owner 提问**：`trigger: "owner_question"`，context 带 `ownerUtterance`；prompt 要求先答该问题。
3. **检索意图**：快捷语与自由问句经 `extractOwnerSearchQueries` 驱动 bootstrap search（只看决策 / 冲突 / 重进后变化等）。
4. **follow-read 预算**：有 Owner 问句时 follow-read 上限 **8**；否则 **4**（实现：`agent-runtime` 内 `ownerFollowBudget`）。仍受总 `maxFilesRead` / `maxToolCalls` / wall 约束。
5. **证据门**：无 pin 不装「读懂」；gap 明文。
6. **canvas 意图**：Owner 话语可触发 `set_canvas_view`（菜单词表见 `docs/product/CANVAS_MENU_V1.md`），不另开第二对话面。
7. **round-trip**：改格式时同时跑 `formatAgentDialogueReply` ↔ `parseAgentMessage` 单测。

### 6.3 预算与默认选项（format）

| 参数 | 默认（实现时） | 说明 |
|---|---|---|
| `maxEvidence` | 4 | 依据芯片上限 |
| `maxChars` | 1600 | 整段对话字符上限 |
| follow-read | 8 / 4 | Owner 问 / 非 Owner 问 |

调参须有单测或验收句式支撑，禁止为「更像 ChatGPT」抬长废话。

## 9. 验收句式（给主会话验收用）

每条格式：**当我做 X，我应看到 Y**。勾选时只认可见行为。

### 9.1 UI

1. 当我打开任意已有项目，我应在右侧一眼看到「问 Agent」输入区（不靠猜 tab）。
2. 当项目尚无授权夹，我应仍看到输入框与待授权提示；发送时应提示先授权，而不是整块消失。
3. 当我发送一条问题且 Agent 返回结构化正文，我应看到「当前判断 / 依据芯片 / 你现在只要决定」分区，而不是一整块无结构气泡。
4. 当依据含 `path @ rev`，我应看到可辨识的依据芯片（path 与 rev）。
5. 当回复含候选语义，我应看到「候选判断 · 未自动写入项目事实」类注脚。
6. 当我点快捷 pill「只看决策 / 冲突在哪 / 重进后变化」，我应看到对应问题进入输入并触发发送链路。
7. 当我点顶栏 AI Copilot，我应看到右栏对话输入被聚焦，而不是跳到无关材料页。
8. 当我查看工作台，我应仍看到底部时间线槽位内容可用（本轮未因参考图被删掉）。

### 9.2 Agent 能力

1. 当我授权含真实文档的夹并完成分析，我应先看到 map/search/read 类过程在动，再看到理解/对话结论。
2. 当 Agent 给出当前判断，我应能从依据落到授权夹内真实路径；点不开或对不上 = 不合格。
3. 当材料不足以支持结论，我应看到明确限制/未知，而不是编造路径。
4. 当模型网关失败或中断，我应看到失败态，且**不应**出现「已读懂」的假候选写入。
5. 当我确认/修改/拒绝候选后刷新或换项目再进，我应仍看到已记住的裁决结果。
6. 当我用自然语言提问（`owner_question`），回复正文应符合 §4.2 结构，且 now/判断应直接服务该问题。
7. 当工具环跑完写入对话，消息体应由 `formatAgentDialogueReply` 生成，并能被 `parseAgentMessage` 解析为 `kind: "structured"`（在有完整段落时）。

### 9.3 安全与发布
