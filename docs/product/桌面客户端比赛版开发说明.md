# FC-OPC iBot 桌面客户端比赛版开发说明

> 执行者：Grok 团队
> 验收者：Codex
> 目标平台：当前演示机，macOS 15.5 / arm64
> 产品入口：`/track/knowledge`，不得增加第二套产品面

## 1. 一句话目标

把当前本地 Next + Node 产品打成一个可以从 Finder 启动的 macOS `.app`：应用自己启动并监督本机能力层，打开既有 `/track/knowledge`，真实授权、真读、证据、确认和重启恢复都保持成立。

这不是把网页换成原生 UI，也不是把网站放进一个没有本地生命周期的 WebView。

## 2. 本轮冻结范围

### 必须完成

1. 当前 arm64 Mac 上可以产出并启动 `.app`。
2. 用户不需要先运行 `npm run dev`。
3. Electron 启动一个只监听 `127.0.0.1` 的 Next standalone 服务。
4. Electron 只加载 `http://127.0.0.1:<port>/track/knowledge`。
5. 数据写到 Electron `userData` 下的独立 `knowledge` 子目录，退出重开仍存在。
6. Renderer 保持沙箱：无 Node、启用 context isolation、禁止任意导航和新窗口。
7. 继续使用现有授权策略、本机会话、CSRF、Agent Runtime、证据链和失败语义。
8. 现有 Web 开发模式、单测和唯一入口不回归。
9. 提供真实 `.app` 路径、启动日志、测试结果和一次黄金路径录屏。

### 本轮不做

1. 不做 Apple 签名、notarization、Universal、自动更新和公开下载。
2. 不重写 React/Next 工作台，不增加 SwiftUI/Tauri 产品面。
3. 不把文件选择器改成 Electron IPC；比赛版继续复用现有 macOS 原生选择流程。
4. 不顺便把 JSON 默认存储迁到 SQLite。
5. 不顺便完成独立 Worker、全量 Claim 替换或 Prompt 评测。
6. 不把 LLM Key、`.env.local`、测试材料或用户数据打进应用包。
7. 不创建过程分支，不覆盖其他窗口的未提交成果。

默认交付策略：**自己的 Mac 真实运行 + 录屏兜底**。如果 Owner 后续明确要求评委在另一台 Mac 安装，再单独增加签名、芯片和 Gatekeeper 计划。

## 3. 已核对的当前事实

- `package.json`：Next `16.2.10`、React `19.2.4`、Node `>=22`。
- `docs/architecture.md`：页面是呈现层，`shared/*` 是本机能力层，二者通过 `/api/*` 连接。
- `shared/project-memory/native-folder-picker.ts`：现有 macOS 文件夹选择是真实本机动作。
- `shared/project-memory/observer.ts`：使用 `@parcel/watcher` 真实监听授权目录。
- `shared/security/local-session.ts`：已有 loopback、同源、session cookie 和 CSRF header 边界。
- `app/api/local-session/route.ts`：可以作为启动健康检查；它同时验证数据目录可写并建立本机会话。
- `KNOWLEDGE_DATA_DIR` 已被知识、项目记忆、对话记忆、Claim 和本机会话共同识别。

Context7 核对结果：

- Next standalone 会生成最小 `server.js`，但不会自动复制 `public` 和 `.next/static`。
- Electron `utilityProcess.fork()` 可以启动独立 Node 子进程，支持 `env`、`cwd`、`stdio` 和 `exit` 事件。
- Electron 推荐 Renderer 保持 `nodeIntegration=false`、`contextIsolation=true`、sandbox 开启。
- Electron `app.getPath('userData')` 适合存应用配置和状态；本项目统一放入其 `knowledge` 子目录，避免和 Chromium 缓存混放。

## 4. 目标架构

```text
用户打开 FC-OPC iBot.app
        │
        ▼
Electron Main
  - 单实例
  - 选择 loopback 空闲端口
  - 设置 KNOWLEDGE_DATA_DIR
  - utilityProcess 启动 Next standalone
  - 轮询 /api/local-session，成功后才显示窗口
  - 监督退出并写本地日志
        │
        ├──────────────► Next standalone
        │                 127.0.0.1:<private-port>
        │                       │
        ▼                       ▼
Sandbox BrowserWindow ──HTTP──► 现有 app/api/*
/track/knowledge                 │
                                ▼
                          shared/* 本机能力层
                           │             │
                           ▼             ▼
                    userData/knowledge  用户授权项目夹
```

关键边界：

- Electron Main 只管理桌面生命周期，不承载产品业务真相。
- BrowserWindow 只是现有网页呈现层，不获得读盘、Shell 或任意 IPC 权限。
- Next standalone 继续调用 `shared/*`，因此现有可信边界和测试仍是同一套。
- 后端未就绪时窗口不能伪装成已启动；模型失败时不能伪装成已读懂。

优先新增文件，降低与并行窗口冲突：

## 6. 接口冻结

`desktop/runtime.cjs` 至少导出：

```js
module.exports = {
  chooseLoopbackPort,
  createWindowOptions,
  isAllowedAppUrl,
  parseDesktopEnv,
  resolveDesktopPaths,
  waitForHttpOk,
};
```

约定：

```js
resolveDesktopPaths({ userDataDir, appPath }) => ({
  knowledgeDir, // <userDataDir>/knowledge
  envFile,      // <userDataDir>/.env.local
  logFile,      // <userDataDir>/logs/desktop.log
  serverEntry,  // <appPath>/runtime/server.js
})

createWindowOptions(preloadPath) => ({
  show: false,
  webPreferences: {
    preload: preloadPath,
    nodeIntegration: false,
    contextIsolation: true,
    sandbox: true,
  },
})

isAllowedAppUrl(candidate, port) => boolean
// 只接受 http://127.0.0.1:<port>/track/knowledge 及其同源资源/API。
// 拒绝 localhost 别名、其他端口、file:、https:、外部域名。
```

`.env.local` 处理：

- 只从 `<userData>/.env.local` 或当前进程环境读取。
- 只允许 `LLM_BASE_URL`、`LLM_API_KEY`、`LLM_MODEL`、`AGENT_RUN_MODE`、`AGENT_ALLOW_DETERMINISTIC_FALLBACK`。
- 进程环境优先于文件。
- 日志不得输出值，只能记录某项“已配置/未配置”。
- 打包脚本必须断言 staging 和 `.app` 中不存在 `.env.local`。

### Task 2：建立可审计的 standalone staging

**文件**

- Modify: `next.config.ts`
- Create: `scripts/prepare-desktop-bundle.mjs`
- Create: `tests/unit/desktop-packaging.test.ts`

构建流程：

1. `npm run build` 生成 `.next/standalone/server.js`。
2. 清空并重建 `.desktop-stage/`。
3. 将 standalone 根内容复制到 `.desktop-stage/runtime/`。
4. 将 `.next/static` 复制到 `.desktop-stage/runtime/.next/static`。
5. 将 `public` 复制到 `.desktop-stage/runtime/public`。
6. 将 `desktop/*.cjs` 复制到 `.desktop-stage/desktop/`。
7. 写入最小 staging `package.json`，`main` 指向 `desktop/main.cjs`。
8. 扫描 staging，发现 `.env*`、`data/knowledge`、`.ship/evidence`、测试夹具即失败。
9. 断言 `@parcel/watcher` 的当前平台产物被 Next tracing 带入；缺失则构建失败，不得等运行时才发现。

运行：

```bash
npm run build
node scripts/prepare-desktop-bundle.mjs
npx vitest run tests/unit/desktop-packaging.test.ts
```

期望：staging 只有桌面入口、standalone runtime、静态资源和最小 package metadata。

staging `package.json` 固定为下面的最小形状，不继承根项目依赖：

```json
{
  "name": "fc-opc-ibot-desktop",
  "productName": "FC-OPC iBot",
  "version": "0.1.0",
  "private": true,
  "main": "desktop/main.cjs"
}
```

### Task 3：Electron Main 管真实生命周期

**文件**

- Create: `desktop/main.cjs`
- Create: `desktop/preload.cjs`

必须按此顺序启动：

1. `app.requestSingleInstanceLock()`；失败则立即退出。
2. `await app.whenReady()`。
3. 解析 `userData` 路径和允许的模型环境变量。
4. `chooseLoopbackPort()` 只绑定 `127.0.0.1`。
5. 用 `utilityProcess.fork(serverEntry, [], options)` 启动 standalone：

```js
{
  cwd: runtimeDir,
  env: {
    ...allowedEnvironment,
    NODE_ENV: "production",
    HOSTNAME: "127.0.0.1",
    PORT: String(port),
    KNOWLEDGE_DATA_DIR: knowledgeDir,
  },
  stdio: "pipe",
  serviceName: "FC-OPC Local Capability",
}
```

6. 日志写入 `logFile`，对 LLM Key 全量脱敏。
7. 轮询 `http://127.0.0.1:<port>/api/local-session`；20 秒内未得到 2xx 则杀掉 utility process，显示真实错误并退出。
8. 创建 `BrowserWindow(createWindowOptions(preloadPath))`。
9. 注册 `will-navigate`、`setWindowOpenHandler`，拒绝 `isAllowedAppUrl` 之外的地址。
10. 加载唯一入口，`ready-to-show` 后再显示窗口。
11. utility process 意外退出时，窗口显示不可伪造的故障提示；不得留一个看似正常但 API 已死的 UI。
12. `before-quit` 主动终止 utility process，等待退出，不留下孤儿服务。

`preload.cjs` 比赛版不得暴露任何接口：

```js
"use strict";
// Intentionally empty. Native capabilities remain in the local Next capability layer.
```

### Task 4：开发启动与 arm64 打包

**文件**

- Create: `scripts/desktop-dev.mjs`
- Create: `scripts/package-desktop.mjs`
- Modify: `package.json`
- Modify: lockfile
- Modify: `.gitignore`

依赖只增加：

```json
{
  "devDependencies": {
    "@electron/packager": "20.0.3",
    "electron": "43.1.1"
  }
}
```

版本已在 2026-07-17 通过 npm registry 核对；Electron Main、BrowserWindow 与 utility process 接口已通过 Context7 核对。若执行时 lockfile 解析出不同版本，必须暂停并说明，不得静默漂移。

脚本行为：

```text
desktop:dev
  启动 Next dev（127.0.0.1:3331）
  健康检查通过后启动 Electron
  任一子进程退出时清理另一个

desktop:prepare
  next build + prepare-desktop-bundle

desktop:package
  desktop:prepare
  @electron/packager(.desktop-stage, darwin, arm64, out/)
```

`scripts/package-desktop.mjs` 的 packager 参数冻结为：

```js
await packager({
  dir: stageDir,
  name: "FC-OPC iBot",
  platform: "darwin",
  arch: "arm64",
  electronVersion: "43.1.1",
  out: outDir,
  overwrite: true,
  prune: false,
  asar: false,
});
```

比赛版选择 `asar: false`，目的是让 Next traced files 和 `@parcel/watcher` 原生产物以最小风险加载。正式分发阶段再单独启用 ASAR 与 native unpack，并重新跑安全门。

`package.json` 至少增加：

```json
{
  "scripts": {
    "desktop:dev": "node scripts/desktop-dev.mjs",
    "desktop:prepare": "npm run build && node scripts/prepare-desktop-bundle.mjs",
    "desktop:package": "npm run desktop:prepare && node scripts/package-desktop.mjs"
  }
}
```

`desktop:dev` 可以通过 `FC_OPC_DESKTOP_DEV_URL=http://127.0.0.1:3331` 让 Main 加载开发地址；Main 只有在 `app.isPackaged === false` 时可以读取该变量，生产构建必须完全忽略它。

## 8. 严重问题：发现即暂停合并

1. `.env.local` 或 API Key 出现在 `.app`、staging、日志或 git diff。
2. BrowserWindow 开启 Node integration、关闭 sandbox/context isolation，或把通用 `ipcRenderer` 暴露给页面。
3. 本地服务监听 `0.0.0.0`、局域网地址或允许跨端口 Origin。
4. 未授权就读取文件，或授权范围外的路径可被访问。
5. 客户端化后 Claim 失去精确 Revision 证据。
6. 后端/模型失败但 UI 显示“已完成”“已读懂”。
7. 打包过程中迁移或覆盖现有用户数据。
8. 为桌面版新增第二套产品入口或复制业务代码。
9. 提交夹带其他窗口未完成的 WIP。
