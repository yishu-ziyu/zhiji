## 3. 三渠对比：主人到底看见什么不同

| 维度 | 本项目 | GitHub **API** | 邮件（以后） | **浏览器回退** |
|------|--------|----------------|--------------|----------------|
| 何时出现 | 默认最先 | 解释「仓上怎么变了」时 | 仅通信证据必要时 | API/结构化源不够时 |
| 授权可见 | 项目内自动 | 源+仓+scope+只读；预授权要 **告知** | scope 重，后置 | URL/登录/扩展权限；先问升级 |
| 一条结果长什么样 | 材料名、卡、事件 | Issue#、sha、PR、文件路径+链接 | 邮件主题/时间/发件人（将来） | URL、可见摘录、可能整页噪声 |
| 检索留痕 | 项目范围+原因 | API 方法/对象 id/时间 | 查询式+邮箱范围 | URL+操作步骤+时间；更难复现 |
| 变依据 | 可点材料 | 稳定外链+对象 id | 邮件 id（将来） | URL 快照；易失效 |
| 确认前 | 候选 | 候选 | 候选 | 候选；且多一道风险提示 |
| 噪声 | 低（自己的文件） | 低–中（结构化） | 中–高 | **高**（布局/广告/登录墙） |
| 风险 | 误引用旧材料 | 读错仓/扩 scope | 隐私、全量同步 | 误点、会话劫持感、不可控外发 |
| 写回项目 | 时间线/依据 | 对象链接进依据/事件 | 将来 | 需明确「浏览器观察」事件类型 |

### 3.1 浏览器相对 API：何时更吵、更险（主人语言）

**更吵：**

- 一页里主内容与导航、广告、推荐混排，摘要易漂。  
- 同一 URL 今天/明天 DOM 不同，依据难复检。  
- 为找一个字段可能连点多步，留痕变成「操作剧本」而非「一个对象」。  

**更险：**

- 容易滑向点击、提交、购买、发送（D-15 必须先问；默认只观察）。  
- 登录态/扩展能力边界不清时，主人难分「只读看了」还是「动了账」。  
- 失败重试可能重复副作用（API 只读则无此问题）。  

**因此 Owner 可见规则：**  
能 API 解决的 **不要**先开浏览器；浏览器结果必须多标一层 **回退 · 高噪声**，且不得自确认。

## 4. 完整可见链（对齐 D-16 · 外部工具版）

命题不变：解释项目为何变化。

| 阶段 | 主人做 | 应看见 |
|------|--------|--------|
| 检索留痕 | 发起解释 | 分层痕迹：A 项目内 →（可选）B GitHub API →（可选）D 浏览器；每层原因/范围/时间 |
| 使用→依据 | 点用 1 条项目材料 + 1 条 GitHub 对象 | 两条都在依据链，来源类型不同、均可点 |
| 候选判断 | 读 Agent 总结「为何变了」 | **候选**判断；脚注挂依据；禁自确认（D-15） |
| Owner 确认 | 确认或否决 | 状态升格/否决痕迹；聊天与工作台一致（D-14） |
| 任务 | 若需跟进 | 同项目工作项；**不**因外部只读自动变「完成」 |
| 结果回流 | 任务或再次读取后 | 结果回同项目+同聊天；旧判断可修正/复查，历史保留 |

**外写（如建 Issue）不在 D-18 第一刀。** 若将来做：执行前冻结账号/仓/标题/正文/依据（见历史 contract 002 精神），且 **先问**；成功只证明外部事项已建，不自动关闭工作项。
