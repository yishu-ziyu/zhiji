# 001 — 重新进入项目时恢复工作状态

## 状态

这是讨论中的草稿，只记录已经说清楚的问题和当前解决方法。尚未完成用户调研，也不是开发完成的说明。“恢复上次工作状态”只是产品必须支持的一个场景，不能据此把整个产品设计成恢复说明页面；最终界面需要先通过图片确认。

## 已经确认的问题

一个人同时推进多个项目，并让多个 Agent 生成代码、文档和研究结果。重新进入某个项目时，他既可能忘记自己离开时正在做什么，也可能不知道离开后其他 Agent 做了什么，因此无法判断原本准备做的下一步现在是否仍然有效。

当前已经确认的真实表现：

- 用户首先想知道“我们上一次碰这个项目时在做什么”。
- 用户既想恢复自己离开时的工作状态，也想知道离开后其他 Agent 推进了什么。
- 文档是否读过、是否影响过决定，目前无法确认，容易变成无法检查的黑盒。
- 当前证据只证明这个问题对首位用户真实存在，尚未证明其他用户同样频繁遇到。

## 问题与解决方法

| 具体问题 | Agent 的解决方法 | 怎么验证 |
|---|---|---|
| 不知道离开时正在做什么 | 在工作过程中持续记录目标、完成情况、未解决事项和原本的下一步 | 用户重新进入项目时，能否不翻旧材料就准确复述上次状态 |
| 不知道离开后发生了什么 | 汇总离开之后新增的代码、文档、会议结果和其他 Agent 的工作结果 | 汇总是否覆盖所有已知的重要变化，并能点回原始材料 |
| 不知道原计划是否仍然有效 | 把后来的变化与原本的下一步进行比较，说明继续、调整或暂时无法判断的原因 | 用户是否能检查依据，并纠正 Agent 的错误判断 |
| 不知道一份材料是否真正影响了决定 | 每个判断都指向使用过的材料；Agent 推断与用户确认分开标明 | 用户能否回答“这个判断依据了什么”，而不是相信无来源的总结 |

## 怎样记录离开时的工作状态

Agent 不应等到用户离开时才开始记录，因为系统未必知道用户什么时候离开。正确做法是在工作过程中持续记录，离开时只整理已有记录。

1. 开始工作时，确定当前项目和这次准备完成的目标。
2. 工作过程中，只记录会改变项目状态的事情：作出决定、完成工作、遇到阻塞、改变范围、提出下一步。
3. 每条记录都包含执行者、发生时间、结果和原始依据，例如对话、文档、代码提交或测试结果。
4. Agent 任务结束、用户切换项目或工作中断时，根据这些记录生成一次简短说明：目标、已完成、未解决、停止原因、原本的下一步。
5. 用户核对过的内容标明“用户已确认”；没有核对的内容标明“Agent 根据材料推断”，不能混在一起。
6. 如果工作意外中断，Agent 根据已经保存的过程记录恢复，并明确指出最后一段状态可能不完整。

## 重新进入项目时给出的答案

Agent 应先回答：

1. 你离开时正在完成什么目标。
2. 当时已经完成什么，还有什么没完成。
3. 你离开后发生了哪些重要变化。
4. 这些变化是否影响原本的下一步，依据是什么。
5. 哪些地方仍然缺少信息，需要用户判断。

搜索和关系图用于继续追查依据，不代替这份主动说明。

## 边界

- 第一阶段不监听用户电脑中的所有应用，只使用明确接入的项目对话、文档、代码和 Agent 工作结果。
- 不把 Agent 生成的文档自动当成事实。
- Agent 负责整理事实和分析影响，不替用户批准新的方向。
- 不承诺已经证明该问题具有普遍市场需求。
- 不因为记录过程而要求用户额外填写一套完整的项目管理表格。

## 验证方法

先用本项目开发过程做真实实验。每次重新进入项目时记录：

- 用户恢复到正确工作状态所需的时间。
- 为了确认状态而重新打开的聊天、文档和代码数量。
- Agent 提出的下一步有多少次因新证据被修改或取消。
- 是否在开发开始前发现原计划已经失效。

产品只有在减少重新查找，并且帮助用户避免按过期信息继续工作时，才算解决了问题。

## 尚未确定

- 什么事件代表一次工作开始或结束。
- 第一阶段具体接入哪些 Agent 对话和项目资料。
- 多份材料互相冲突时，如何确定优先顺序。
- 用户是否愿意核对每次工作结束时的简短说明。
- 这一问题在其他独立开发者和项目制服务者中出现的频率与损失。
