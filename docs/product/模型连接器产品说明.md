# 模型连接器 · 产品与实现说明

> 目标：让用户在桌面客户端里用一个轻量、有动效的下拉窗选择模型供应商和模型，并且每一个“可用”状态都来自真实连接验证。
>
> 状态仍只记在 `docs/product/产品清单.md` §3c；本文是产品与工程合同，不是第二份问题台账。

## 一、产品判断

这里不是“模型商城”，而是 **本地 Agent 的模型连接器**。

- 应用预置供应商资料、协议和推荐模型，不预装密钥、不代售调用额度。
- 用户自己填写 Key；Key 只留在本机，继续遵守现有 BYOK 边界。
- Logo 只帮助识别供应商，不代表连接成功，也不代表官方背书。
- 用户选择的是“本次 Agent 用哪条真实模型连接”，Run 收据必须记录实际 provider、protocol、base URL 主机和 requested model。

## 二、用户看到什么

### 1. 顶栏入口

把现有单独钥匙入口升级成紧凑的模型胶囊：

```text
[供应商 Logo] DeepSeek · deepseek-v4-pro  ˅
```

- 一眼看见当前供应商和模型。
- 未配置时显示 `选择模型`，不显示绿色成功态。
- 点击后在入口下方弹出轻量下拉窗，不跳页面、不制造第二套产品面。

### 2. 快速下拉窗

第一层只解决切换，不塞入所有高级字段：

```text
模型

[OpenAI]    GPT             已连接 / 需配置
[MiniMax]   MiniMax         已连接 / 需配置
[DeepSeek]  DeepSeek        已连接 / 需配置
[StepFun]   阶跃星辰        当前
[Custom]    自定义网关      已连接 / 需配置

当前模型  [ step-3.7-flash                  ˅ ]

管理连接…
```

- 供应商行包含本地 Logo、名称、连接状态；只有真实验证过才写“已连接”。
- 选择供应商后，模型列表随之切换；优先使用该连接 `GET /models` 返回的真实列表，失败时才显示经过维护的推荐值。
- 未配置供应商时，点击进入二级“连接模型”面板，而不是假装已经切换成功。
- 切换已验证连接后，胶囊立即更新；下一次 Agent Run 使用新连接，正在运行的 Run 不热切换。

### 3. 二级连接面板

只在新增或管理连接时出现：

1. 供应商（已由上一步带入）；
2. API Key；
3. 模型；
4. “高级设置”折叠区：Base URL、协议、鉴权方式；
5. `测试连接`；
6. 测试成功后才可 `保存并使用`。

测试状态必须明确：`尚未测试` / `正在连接` / `连接成功` / `连接失败`。错误信息应翻译成用户语言，同时保留可复制的简短诊断，不展示 Key。

## 三、首批预置

接口事实依据：2026-07-17 通过 Context7 查询官方 OpenAI、DeepSeek、MiniMax 文档。模型列表会变化，因此运行时应优先读取供应商的 models endpoint，下面只定义协议和推荐起点。

| preset | 展示名 | 协议 | 默认地址 | 推荐起点 |
|---|---|---|---|---|
| `openai` | OpenAI | `openai_responses` | `https://api.openai.com` | `gpt-5.6` |
| `minimax` | MiniMax | `openai_chat` | `https://api.minimax.io` | `MiniMax-M3` |
| `deepseek` | DeepSeek | `openai_chat` | `https://api.deepseek.com` | `deepseek-v4-pro` |
| `stepfun` | 阶跃星辰 | `anthropic_messages`（沿用当前真实链路） | 当前已验证地址 | `step-3.7-flash` |
| `custom_anthropic` | 自定义网关 | `anthropic_messages` | 用户填写 | 用户填写 |

说明：

- OpenAI 使用 Bearer 鉴权和 `/v1/responses`；模型列表使用 `/v1/models`。
- MiniMax 和 DeepSeek 均提供 OpenAI 兼容 Chat Completions；MiniMax、OpenAI 有 `/v1/models`。DeepSeek 以官方返回或预置推荐值为准。
- MiniMax 也提供 Anthropic 兼容接口，但首版只选一条经过测试的协议，避免同一供应商出现两个含义不明的入口。
- `gpt-5.6-sol` 这类本地代理别名属于“自定义网关”，除非上游身份和响应能被证明，否则不得展示成 OpenAI 官方模型。

## 四、动效

动效服务于空间关系和状态反馈，不模拟 Agent 思考：

- 下拉出现：`opacity 0→1`、`translateY(-4px→0)`、`scale(.985→1)`，160–180ms。
- 供应商切换：选中底色与勾选标记 140ms；模型区 120ms 交叉淡入，不整体跳动。
- 连接测试：按钮内真实 spinner；成功后一次性 check 轻弹，不做循环呼吸灯。
- 关闭：120ms 反向淡出；点击外部、Esc 均可关闭，焦点回到模型胶囊。
- 必须支持键盘上下选择、Enter 确认，并遵守 `prefers-reduced-motion`。

## 五、视觉与 Logo

- 延续现有浅色工业 SaaS：白色浮层、`#007AFF` 主动作、细边框、12–16px 圆角、克制阴影。
- Logo 使用仓库内的本地 SVG 资产，禁止运行时远程请求；统一 20×20 的视觉盒。
- 不魔改品牌图形，不把模型名伪装成供应商 Logo；自定义网关使用中性“节点/插头”图标。
- 下拉不超过约 380px 宽；它是工作台控制器，不抢项目情报简报的注意力。

## 六、真实工程边界

现状只有一个 `anthropic_messages` adapter，固定调用 `{baseUrl}/v1/messages` 和 `x-api-key`。因此 UI 上线前必须同时补齐：

```ts
type LlmProtocol =
  | "anthropic_messages"
  | "openai_chat"
  | "openai_responses";

type LlmConnectionProfile = {
  id: string;
  provider: "openai" | "minimax" | "deepseek" | "stepfun" | "custom_anthropic";
  protocol: LlmProtocol;
  baseUrl: string;
  model: string;
  authMode: "bearer" | "x-api-key";
  verifiedAt: string | null;
};
```

必须满足：

1. adapter 按协议构造正确 endpoint、header、body，并统一抽取文本结果；
2. 连接测试发起最小真实请求或读取 models endpoint，不能只校验字段非空；
3. API 不回传密钥；日志、错误、Run Receipt 不含密钥和完整 query；
4. 保存后立即生效，但仅影响新 Run；旧 Run 保留原 provider/model 证据；
5. Run Receipt 的 provider 不再硬编码 `stepfun`；必须与连接 profile 和真实请求一致；
6. 连接失败不得保存“已连接”，模型失败不得静默回退到另一供应商；
7. 比赛版可只保存一个 active profile；以后再升级为 Keychain 内的多连接密钥库。

## 七、用户语言验收

1. 我点顶栏当前模型 → 看见带 Logo 的供应商下拉窗，当前项清楚，项目画布不被遮满。
2. 我选未配置的 OpenAI → 看见填写 Key 和测试连接，不会立刻显示“已连接”。
3. 我测试成功并保存 → 顶栏显示 OpenAI 和实际模型；新 Agent Run 的收据也显示同一 provider/model。
4. 我选 DeepSeek 或 MiniMax → 请求走各自真实协议；连接失败时明确失败，不沿用 StepFun 的成功态。
5. 我选自定义网关并填代理模型别名 → UI 明确写“自定义网关”，不把别名包装成官方 GPT。
6. 我关闭并重开客户端 → 当前选择仍在，Key 不回显、日志找不到明文。
7. 我打开减少动态效果 → 下拉和状态切换无明显位移或弹跳。

## 八、比赛前保护线

- 当前已经 `ready-for-owner-recording` 的 `.app` 是兜底产物，不覆盖、不删除。
- 先在开发端完成 UI、adapter、连接测试和 receipts；全绿后再另产候选包。
- 若任一供应商没有真实验证，就从“可选项”降为“需配置/接入中”，绝不为了录屏显示假绿。
- 该能力不改变 Demo 主故事：模型是可替换能力层，项目情报简报和有依据的 Owner 决策才是产品价值。

## 九、Owner 当前可用的 Demo 连接矩阵（2026-07-17）

比赛前优先实现 Owner 实际能提供凭证并亲手验证的连接，不以 Logo 数量作为完成标准。

| 连接来源 | 展示方式 | 协议 | Base URL | 当前模型入口 | 鉴权 |
|---|---|---|---|---|---|
| `px_proxy` | `PX Proxy`；模型行带 GPT / Grok / Gemini 图标与“代理”标记 | `anthropic_messages` | `http://127.0.0.1:8317` | `gpt-5.6-sol`、`grok-4.5`、`gemini-pro-agent` | `bearer`（如实测不同，仅允许在高级设置调整） |
| `minimax_token_plan` | `MiniMax · Token Plan`，标记官方直连 | `anthropic_messages` | `https://api.minimax.io/anthropic` | `MiniMax-M2.7`（以账户实际可用模型为准） | `x-api-key` |
| `stepfun_step_plan` | `阶跃星辰 · Step Plan`，标记官方直连 | `anthropic_messages` | `https://api.stepfun.com/step_plan` | `step-3.5-flash`；稳定后再测试 `step-router-v1` | `bearer` |

事实边界：

- PX Proxy 的三项是本地代理暴露的 model alias。即使使用对应品牌 Logo，也必须同时显示“PX 代理”；Run Receipt 的 provider 必须是 `px_proxy`，不能写成 OpenAI / xAI / Google 官方直连。
- MiniMax Token Plan Key 与普通按量 API Key 不通用；比赛版按 Token Plan 的 Anthropic 兼容入口实现。
- StepFun Step Plan 使用专用 `/step_plan` 通道，不得与普通开放平台 `/v1` 地址混用。
- Demo 主界面先展示这三条 Owner 已拥有凭证的连接。OpenAI 官方、DeepSeek 官方等长期预置可保留在架构中，但未 live 验证前不得作为比赛版可用项抢占注意力。
- 任何凭证都由 Owner 在本机 UI 中填写；文档、测试夹具、源码和候选包不得包含真实值。

比赛版最短策略：三条连接共用一个正确实现的 Anthropic Messages adapter；修正顶层 `system`、不同鉴权头、文本块提取和服务端真实测试，再考虑扩展其他协议。
