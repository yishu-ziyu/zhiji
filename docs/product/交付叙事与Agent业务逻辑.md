## 1. 一句话

**知几 是一个运行在本地、以证据为基础的项目情报 Agent：当用户重新进入项目时，它恢复当前态势，并帮助用户完成下一步决定。**

更口语的说法：

> 它不是帮你看见更多，而是让你只看见此刻真正需要看见的东西。

比赛版只证明一件事：

> **当用户重新回到一个复杂项目时，Agent 在明确授权范围内恢复项目态势，给出有证据的当前判断，并帮助用户完成下一步决定。**

长期愿景可以是项目情报系统，Demo 必须是这一个高价值、可验证、可重复的任务。两者不冲突。

---

## 2. 我们在对抗什么

项目的问题往往不是没有信息，而是信息持续增长和变化：

- 文档、计划、记录和实际进展不再一致；
- 新结论、旧假设和临时想法混在一起；
- 人需要反复翻找材料、重建上下文；
- 重要变化和大量无关变化共用同一份注意力；
- 人即使读完了所有信息，仍然可能不知道现在该相信什么、决定什么。

这是一种项目认知的混乱。它消耗的不只是时间，还包括判断质量、交付节奏和对项目的掌控感。

知几 不再给用户增加一个信息入口，而是替用户吸收变化、排除噪声，把注意力留给真正需要人判断的时刻。

## 3. 为谁服务

比赛 Demo 先聚焦一类人：

**同时承担多项任务、需要对项目结果负责的小团队项目负责人。**

他们可能是创业者、产品负责人、客户项目负责人、咨询或交付人员，也可能是一人公司的经营者。

共同的业务压力是：

- 他们无法持续阅读项目中的每一处变化；
- 但关键变化被错过时，最终责任仍在他们；
- 他们需要的不是更长的总结，而是“现在怎样、凭什么、我需要决定什么”。

长期可扩展到客户交付、项目交接、创意制作、研究、内容运营和个人知识工作等场景。比赛阶段不展开为泛化的“万能知识助手”。

## 4. 用户购买的不是总结，而是注意力与判断质量

产品带来的业务价值可以用三个结果表达：

1. **少看**：从大量变化中只看真正会改变判断的部分。
2. **少错**：重要结论能回到真实材料和精确版本，降低基于过期或错误信息行动的风险。
3. **少拖**：把“我还要再看看”收敛成一个可以确认、修改、拒绝或暂缓的下一步决定。

产品并不承诺绝对完整或绝对正确的项目全景。它追求的是：在当前决定所需的范围内，形成足够有依据、足够及时、可以继续工作的判断。

### 4.1 比赛版垂直切片：重新进入项目

**目标用户：** 同时承担多个项目、需要对结果负责的小团队负责人。

**触发场景：** 用户离开项目一段时间后重新回来，或即将交付、开会、接手项目，需要快速恢复判断。

**当前做法：** 重读文档、代码、聊天和任务记录，再向团队反复追问；即使读完，也难以区分工程完成、口头完成和产品真正完成。

**Agent 任务：** 在授权范围内观察、搜索、读取、比较和核验，生成一份最小项目情报简报：

```text
当前判断
为什么现在重要
关键依据
相反证据或限制条件
关键未知
现在需要你决定什么
```

**完成定义：** 用户不需要重新阅读整个项目，就能检查依据、修正 Agent 的判断，并完成一个下一步决定。Agent 是否“说了很多”不算成功。

**为什么必须是 Agent：** 这个任务需要连续调用多个工具，根据读取结果动态决定下一步，维护跨会话状态，暴露失败和未知，并在人的确认之后继续观察结果；一次总结或固定自动化无法完成整个闭环。

## 5. Agent 的业务循环

```text
持续观察
  → 过滤无关变化
  → 更新对项目现状的候选判断
  → 发现冲突、缺口或需要人的决定
  → 用精确依据提出下一步建议
  → 用户确认、修改、拒绝或暂缓
  → 保留决定与理由
  → 继续观察决定后的变化
```

这个循环中，Agent 承担的是程序性自主，人保留真相和高影响决定的权力。

### 5.1 Agent 可以自主完成

- 在用户明确授权的范围内建立项目地图；
- 搜索、读取、比较版本并识别变化；
- 区分噪声、重要变化和会影响决定的变化；
- 形成候选判断，并标记支持、冲突、不足或过期；
- 在依据不足时说明未知，而不是补齐一个答案；
- 提出最小、明确、尽可能可逆的下一步建议；
- 在材料或版本变化后，主动将可能过期的旧判断重新提交审查。

### 5.2 Agent 不能替用户做

- 不能自行扩大读取或数据外发范围；
- 不能把一段普通聊天自动写成项目事实；
- 不能把推测、候选理解或工作建议伪装成已确认真相或正式任务；
- 不能用无关材料或不存在于精确版本的引用装饰结论；
- 不能在模型、工具或依据失败时继续显示“已读懂”；
- 不能自行接受自己提出的 Candidate；
- 不能在当前版本中修改源项目文件、执行任意命令或对外发送。

### 5.3 比赛版 Agent Contract

| 项目 | 合同 |
|---|---|
| 核心目标 | 恢复项目态势，让 Owner 完成一个有依据的下一步决定 |
| 自动能力 | 在已授权范围内建图、搜索、读取、比较、形成 Candidate 和标记未知 |
| 必须展示 | 真实 Run 状态、Tool Receipt、逐条 Claim、精确 Revision 依据 |
| 必须确认 | 正式项目事实、方向变化、正式任务和任何高影响动作 |
| 停止条件 | 已形成可审查简报并提出一个待决定问题；或依据/模型/工具不足而诚实失败 |
| 成功条件 | Owner 能打开依据，接受、修改、拒绝或暂缓判断；刷新后结果仍在 |
| 禁止 | 越权读取、自动扩大范围、聊天自动成事实、建议伪装成任务、假进度和假成功 |

界面上可以只有一个 Agent，内部职责必须分清：观察者负责发现变化，分析者负责形成和挑战判断，简报者负责节制用户注意力，执行者只在授权后行动。

## 6. 为什么是本地端

本地端不只是一个隐私选择，也是产品能力的基础。

### 6.1 持续性

项目变化发生在真实的本地材料和工作环境中。本地进程可以在用户不反复上传、复述和重建上下文的情况下，持续观察已授权信号。

### 6.2 精确性

文件和版本是可核对的。Agent 可以知道一个判断来自哪个文件、哪个版本和哪一段原文，也可以在依据变化后标记判断过期。

### 6.3 用户控制

授权范围、未公开材料、项目事实和决定记录默认由用户的设备保管。文件授权不等于可以发送给任意远程模型，数据外发必须与文件读取分别说明。

### 6.4 表达边界

当前产品只能声称持续理解**用户授权范围内的可观察状态**，不能声称已经看见项目的全部运行状态。

未来接入任务系统、日历、客户沟通和业务指标后，可观察边界才会逐步扩展。

## 7. 为什么它不是普通 Copilot

| 普通 Copilot | 知几 |
|---|---|
| 等用户提问 | 持续观察已授权的项目信号 |
| 回答当前问题 | 维护可继续的项目判断 |
| 倾向给出一个完整答案 | 允许不确定、冲突和依据不足 |
| 对话结束后上下文可能消失 | 候选判断、人的确认和历史版本持续保留 |
| 展示更多内容 | 只在需要人判断时占用注意力 |
| “有答案”是成功 | “用户用最少注意力完成一个足够正确的决定”是成功 |

我们不与通用聊天助手比谁能回答更多，而是解决：**什么时候值得打断人，人在那个时候最少需要看什么。**

## 8. 真相与建议的状态

为了让“有依据”和“人最终负责”成为真实产品行为，系统需要区分：

| 对象 | 含义 | 能否当作已确认事实 |
|---|---|---|
| Observation | 授权范围内的原始变化或文件事实 | 只能作为原始依据 |
| Claim | Agent 从依据中形成的单条判断 | 不能，须检查依据和状态 |
| Candidate | 提交给用户审查的候选理解 | 不能 |
| Work Suggestion | Agent 建议的下一步 | 不能冒充正式任务 |
| Owner Resolution | 用户接受、编辑接受、拒绝或暂缓 | 是否生效由具体决定决定 |
| Confirmed Fact | 已被用户明确确认的项目事实 | 可以，但来源变化后应标记可能过期 |

这些状态不是后台工程名词，它们直接决定用户是否会被 Agent 制造的虚假确定性误导。

## 9. 比赛 Demo 的黄金路径

### 9.1 场景（视频主线）

负责人离开项目几个小时后回来。团队已经完成大量工程工作，并报告“已打包、已测试”，但产品清单仍保留 Owner 验收缺口。负责人没有时间重新阅读整个仓库，却必须决定：继续增加功能，还是冻结范围完成真实演示闭环。

优先使用 zhiji 自身作为真实项目演示。它天然包含材料、代码、工程结果、产品清单和“工程完成不等于产品验收完成”的判断差异。若最终运行不稳定，再换成结构相同、内容真实且不含秘密的演示项目；不得用测试标题或预制成功数据冒充。

### 9.2 演示剧情

1. 用户打开工作台，进入最近正在进行的项目。
2. 用户选择并授权真实本地项目文件夹。
3. Agent 在授权范围内建立地图、搜索并阅读具体材料，过程和真实收据可见。
4. 系统不展示所有变化，而是指出当前最值得看的一件事。
5. Agent 告诉用户：工程产物已经形成，但 Owner 黄金路径尚未完成，因此“可以打包”和“已经验收”不能视为同一状态。
6. 用户点开该判断，可以回到支持它的具体文件、精确版本和原文片段。
7. Agent 提出一个明确的下一步：冻结新增功能，先完成真实演示闭环；它继续标记为建议，不冒充已确认任务。
8. 用户确认、修改、拒绝或暂缓；结果在刷新和重新进入后仍然存在。
9. 支持该判断的材料再次改变时，系统将旧判断标记为可能过期，而不继续当作当前真相。

### 9.3 Demo 必须让评委当场看见

- 读的是真实授权材料，不是预制结果；
- Agent 先观察和读取，后形成判断；
- 结论可以回到原文依据；
- 信息很多，但用户只需要处理一个问题；
- Agent 的建议与用户的最终决定明确分开；
- 失败、未知、未确认和已确认不会共用同一种成功外观；
- 决定被持久保留，不是点击后的局部动效。

## 10. 五分钟现场叙事

| 时间 | 讲什么 | 评委应该明白什么 |
|---|---|---|
| 第 1 分钟 | 项目信息已经超过人的注意力，负责人无法持续重建上下文 | 这是真实、高频、有成本的问题 |
| 第 2 分钟 | 目标用户与一句话产品定义 | 产品帮谁，不是泛用助手 |
| 第 3 分钟 | Agent 如何观察、过滤、形成候选判断并请求决定 | Agent 在业务流程中承担的真实工作 |
| 第 4 分钟 | 演示一个变化、一条证据链和一个人的决定 | Demo 真的跑通，且不是普通聊天 |
| 第 5 分钟 | 本地、可核对、可持续的差异化，以及可产品化方向 | 团队自己的设计在哪里，为什么值得继续做 |

### 10.1 90 秒视频口播底稿

> 我们每天都在同时推进很多复杂项目。最痛苦的不是资料找不到，而是每次重新回来，都要重新理解发生了什么、哪些是真的、现在该做什么。
>
> 传统知识工具帮我们保存和搜索信息，普通 AI 帮我们总结文件，但它们不会持续维护项目的真实态势。
>
> 所以我们做了一个运行在本地的项目情报 Agent。它在用户明确授权的范围内观察项目变化，把零散材料转化为可核验的当前判断。每条重要判断都能回到精确证据；推断不会伪装成事实；不知道的事情会被明确暴露。
>
> 它不会把所有变化都推给用户，而是只告诉你：现在发生了什么，为什么重要，以及现在需要你决定什么。
>
> 用户确认以后，Agent 才会行动，并继续观察行动结果。
>
> 我们希望做到的不是让 AI 替人掌控项目，而是用最少的注意力，帮助人形成足够正确的判断，并完成下一步决定。

## 11. 商业化逻辑

### 11.1 可付费的原因

用户不会单纯为一份项目总结付费，但可能为以下结果持续付费：

- 不需要每天亲自重建多个项目的上下文；
- 重要变化不被噪声淹没；
- 交接、休假、人员变动后能快速继续工作；
- 对客户、团队和自己的重要判断都能说明当时为什么；
- 在过期信息造成误决策前得到提醒。

### 11.2 可能的产品化路径

- 个人版：按用户或活跃项目订阅；
- 小团队版：按席位和项目数订阅，提供共享确认和交接；
- 客户交付版：面向咨询、代运营、制作和软件交付团队，沉淀可审计的项目现状、决定和依据；
- 后续连接任务系统、日历、沟通工具和业务指标，但不改变“少占用人的注意力”这个产品原则。

### 11.3 可验证的价值指标

- 用户重新进入项目后，到完成一个有依据决定所需的时间；
- 每次会话中被 Agent 隐藏的无关变化数量；
- 用户查看候选判断后的接受、修改、拒绝和暂缓比例；
- 来源变化后，过期判断被成功发现的比例；
- 由过期或无依据信息造成的重复沟通或误决定是否减少。

## 12. 与赛题和评审的关系

主办方要求 Agent 进入真实业务场景，帮助真实角色完成具体任务，而不是比较代码量或模型调用次数。

本产品与“公司 / 一人公司效率 Agent”赛道的对应关系：

| 评审关注 | 本产品的回答 |
|---|---|
| 真实场景 | 项目变化超过负责人注意力，导致反复对齐和误决定 |
| 具体用户 | 对项目结果负责的小团队项目负责人 |
| 具体任务 | 在大量项目变化中完成一个足够正确的下一步决定 |
| Agent 作用 | 持续观察、筛选、形成候选判断、发现冲突、提请决定、继续验证 |
| 团队创新 | 本地授权、持续项目状态、Claim—Revision 证据链、候选与已确认真相分离 |
| 体验完成度 | 一条可现场跑通的授权—观察—判断—依据—确认—继续观察路径 |
| 商业化 | 从个人多项目管理扩展到小团队、客户交付和项目交接 |

AI 技术评分只占总分的 10%，重点是链接可访问、基础稳定、响应和核心流程可运行。不应为了增加技术名词而牺牲黄金路径的可靠性。

## 13. 五页 PPT 的内容种子

> PPT 在最终产品版本完成后制作，截图和功能表述以最终真实产品为准。

### 第 1 页：人的注意力跟不上项目的变化

- 项目不是缺信息，而是信息变化超过人的注意力。
- 用户面对的不是“如何读完”，而是“现在该相信什么、决定什么”。
- 用一个具体负责人场景表达，不画泛化信息过载图。

### 第 2 页：本地常驻的项目判断 Agent

- 一句话产品定义。
- 展示“观察—过滤—判断—决定—继续观察”闭环。
- 点明人的决定权与 Agent 的程序性自主。

### 第 3 页：Demo 黄金路径

- 真实授权文件夹。
- 大量变化被过滤后，只出现一个需要人决定的问题。
- 一条可点回原文的证据。
- 一次真实确认，刷新后仍然存在。

### 第 4 页：为什么值得相信，也不只是 Copilot

- 授权边界；
- 精确版本依据；
- Candidate 与 Confirmed Fact 分离；
- 变化后的过期检查；
- 失败不伪装成成功。

### 第 5 页：商业价值与继续方向

- 少看、少错、少拖；
