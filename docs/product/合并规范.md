# 合并规范 · 单一工作台

> 状态：Owner 已定（2026-07-16）  
> 目标：取消独立 MVP 产品面，**旧台为唯一壳**；MVP 业务逻辑在旧台可跑通。

---

## 1. 问题

两套入口（`/track/knowledge` 与 `/track/knowledge/mvp`）让用户学两套产品；MVP 为证业务临时分叉，现业务已可迁回。

## 2. 目标结果

| 项 | 规定 |
|----|------|
| 唯一产品 URL | `http://127.0.0.1:3331/track/knowledge` |
| `/track/knowledge/mvp` | **永久重定向**到 `/track/knowledge`（保留 query，如 `fixture`） |
| 壳与交互 | 旧台：导航、画布、拖入/上传材料、多项目 |
| 业务主线 | 授权本地文件夹 → 边界内理解 → 过程可见 → 你确认（原 MVP） |
| 拖入 | 保留；与「授权读本地夹」并存，不互斥 |
| 命名 | 对用户称「项目理解 / 知识工作台」即可，不宣传 MVP |

## 3. 身份合并（硬）

历史上两套 ID：

- **知识项目** `shared/knowledge`：`randomUUID()`，材料/画布用  
- **授权夹项目** `project-memory`：`projectIdFromCanonicalFolder(path)`  

**合并规则：**

1. 文件夹 `connect` 成功后，**必须**在知识库中 `ensure` 同 id 项目（名称=文件夹名）。  
2. 工作台列表、画布、材料、授权记忆共用该 `projectId`。  
3. 禁止再生成「同夹两个 id」。

## 4. 信息架构（合并后）

### 空工作台

- 价值先见（这是什么 / 选完长什么样）— 见 `首页规范.md`  
- **主 CTA：** 选择项目文件夹（native picker → connect）  
- **次：** 继续上次授权；拖入/上传/新建项目（旧能力）  

### 已有项目

- 旧台画布 + 导航 + 材料拖入  
- 若该项目存在 active source grant：展示理解过程 / 候选确认（复用 MVP 组件与 API，不新造协议）  

### 选夹后确认

- 仍为第二步确认卡（权限边界说明），再 connect  

## 5. 实现分层（严谨顺序）

| 阶段 | 内容 | 完成标准 |
|------|------|----------|
| M0 | 规范落地（本文） | Owner 可读 |
| M1 | 服务端 ensure 知识项目 on connect | 同 id 出现在 `GET /projects` |
| M2 | 旧台接入 picker+connect+continue | 空台可选夹进项目 |
| M3 | 空台 UI 价值先见 + 保留拖入 | 不丢拖入 |
| M4 | 有 grant 时挂理解面板（MVP 组件） | 能看过程/确认 |
| M5 | `/mvp` redirect + 清单/CONTEXT URL | 只推一个 URL |
| M6 | 验收 | 见 §7 |

禁止并行乱改：不建 branch；不改工程师2锁定的 `agent-model-loop.ts` / `reconstruct*.ts`（除非其公开 API 调用方式不变）。

## 7. 验收（BDD）

1. 打开 `/track/knowledge` → 唯一工作台，非工程双首页。  
2. 打开 `/track/knowledge/mvp` → 落到 `/track/knowledge`。  
3. 空台可「选择项目文件夹」→ 确认 → 连接后侧栏出现该项目。  
4. 连接后仍可拖入/上传材料到当前项目。  
5. 连接路径的 projectId 与侧栏项目 id 一致（无双胞胎）。  
6. 有授权时能看到理解相关 UI 或明确入口（过程/候选）。  
7. HTTP 200；关键单测不因 redirect 整体红（允许改期望 URL）。
