## 用法

1. **Demo 前 30 分钟**：通读 `DEMO_SCRIPT.md`，对每个时间节点熟悉
2. **进入答辩场地后**：先看 `FC_ARCHITECTURE.md` 的生产架构图（如果问"能不能上生产"）
3. **答辩中**：准备 1 分钟**录像版**作为 fallback（如果现场 demo 卡了，立刻切视频）

## 三个核心叙事

### 一句话定位
> "一个 AI 团队，替代一家小公司的运营岗位。"

### 商业模式锚点
> "我们订阅 99 元/月，替代一个运营助理 8 千月薪 + 短视频编导 1 万 + 会议助理 6 千 = **24 万 / 年人力成本**。99 vs 24,000 = **0.4%**。"

### 技术叙事锚点
> "本地 demo 跑通，已为阿里云 FC 部署准备就绪——3 周可上线 MVP。"

## 答辩常犯错误 / 红线

| 错 | 怎么改 |
|----|--------|
| 说"我们做了个聊天机器人" | "我们做了一个**虚拟创始团队**" |
| 说"AI 自动化" | "AI **承接一个完整的业务职能**" |
| 说"多 Agent 系统" | "**一个完整的商业闭环**" |
| 调试时公开 debug | 立刻切录像，不让评委看到控制台 |

## 现场兜底方案

| 风险 | 兜底 |
|------|------|
| 网络断 | 用本地 vscode + 视频回放（不是 demo 视频，是 Gif 截图录屏） |
| LLM API 慢（>30s）| 切到 mock 数据，提前录好 4 个功能 demo |
| 电脑崩溃 | 备用设备：第二台装好的同版本代码，docker compose 一键起 |
| 评委质疑技术 | 直接展示 `docs/demo/FC_ARCHITECTURE.md` 架构图 + 代码仓库 |
